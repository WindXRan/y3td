## 二、隐式依赖清单（9 项）

| # | 依赖关系 | 风险 | 说明 |
|---|---------|------|------|
| D1 | `bonds_chain` 必须在 `boot_utils` 之前 | **高** | `boot_utils:5` 直接 `require 'runtime.bonds_chain'` 并写入 `_G.BondSystem`。boot.lua 中从未显式 require bonds_chain，完全依赖 boot_combat→boot_helpers→boot_utils 这条 3 层隐式链 |
| D2 | `boot_helpers` 依赖 `boot_utils` 先加载 | **高** | `boot_helpers:1` `require 'runtime.boot_utils'`，且 boot_helpers 只做转发（两行代码），实际是 boot_utils 的薄包装 |
| D3 | `debug_actions`/`debug_tools` 依赖 `sample_skills_system` 桥 | **中** | `debug_actions:20` 在模块顶层读取 `_G.sample_skills_system`，如果 debug_system 在阶段8.5桥之前加载则拿到 nil |
| D4 | `progression` 依赖 `hero_attr_system` | **中** | `progression:10` 在模块顶层读取 `_G.hero_attr_system` |
| D5 | `progression` 运行时依赖 `rewards` / `attr_choice_system` | **低** | `progression:13-16` 在 `on_hero_level_up()` 函数内部惰性访问 `_G.reward_system` 和 `_G.attr_choice_system` |
| D6 | `outgame_system.create()` 依赖 `audio_system`、`hud_system` | **中** | `boot.lua:508-513` 创建时传入闭包访问 `_G.audio_system` 和 `_G.hud_system` |
| D7 | `session_state.create()` 依赖 6 个系统 | **高** | `session_state:19-24` 访问 `_G.battlefield_system`, `_G.hero_attr_system`, `_G.progression_system`, `_G.runtime_ui_helpers`, `_G.reward_system`, `_G.sample_skills_system`。任何一个未就绪，boot.lua 顶层直接崩溃 |
| D8 | `boot_runtime_setup.create()` 依赖 `debug_tools_system`, `gm_bond_effects_system` | **低** | 在工厂函数内部惰性访问，首次调用时才检查 |
| D9 | `hero_model` 在模块顶层访问 `_G.STATE` | **中** | `hero_model:10` `local STATE = _G.STATE` 在模块顶层立即读取，必须在 STATE 创建之后加载 |

## 三、建议的编号体系

```
1xx — 基础数据与核心运行时
  110 config.entry_config          (数据表根配置)
  120 data.tables.*                (技能/音频/进化等数据表)

2xx — 工具模块与引擎适配
  210 runtime.boot_core            (核心状态工厂)
  220 runtime.boot_helpers         (工具函数入口 → 自动拉入 boot_utils)
  221 runtime.boot_utils           (工具函数 + 隐式加载 bonds_chain)
  222 runtime.bonds_chain          (羁绊系统 — 被 boot_utils 隐式引入)
  230 runtime.boot_combat          (战斗工具函数)
  240 runtime.gear_upgrades        (装备升级)
  250 runtime.boot_ui_enhancements (UI 增强)
  260 runtime.event_bus            (事件总线 — 零依赖)

3xx — 核心状态与 Buff
  310 runtime.buff_system          (Buff 运行时)
  320 runtime.projectile_name_guard (投射物校验)

4xx — 英雄系统
  410 runtime.hero_attr_system     (英雄属性 — 自注册 _G.hero_attr_system)
  420 runtime.hero_model           (英雄模型 — 自注册 _G.hero_model)

5xx — 核心玩法
  510 runtime.progression          (升级系统 — 自注册 _G.progression_system)
  520 runtime.rewards              (奖励/进化 — 自注册 _G.reward_system)
  530 runtime.audio                (音频 — 自注册 _G.audio_system)
  540 runtime.round_choice         (回合选择)

6xx — 技能与战斗
  610 runtime.skill_damage_templates (伤害模板)
  620 runtime.skill_system         (技能系统总入口)
  630 runtime.battle_system        (战场系统总入口)

7xx — UI 系统
  710 runtime.ui_system            (UI 总入口)
  711 runtime.attr_tips_panel      (属性提示面板)
  720 _G.sample_skills_system 桥   (向后兼容桥)

8xx — 调试系统
  810 runtime.debug_system         (调试总入口)
  820 runtime.battle_auto_acceptance (战斗自动接受)

9xx — 局外系统
  910 runtime.outgame_system       (局外总入口)
  911 runtime.boot_runtime_setup   (运行时启动层)
```

### 编号原则

- 百位=层（1数据→2工具→3核心状态→4英雄→5玩法→6战斗→7UI→8调试→9局外）
- 十位内的顺序即加载顺序
- 编号留空便于插入新模块
- 编号可映射到 `_G.SYSTEM` 命名空间

## 四、boot.lua 标注方式

在阶段注释中加入编号范围和依赖链标注：

```lua
-- ============================================================
-- [1xx] 阶段1：基础数据 — 数据表和配置（无运行时依赖，可最先加载）
-- 加载项: 110 config.entry_config
--         120 data.tables.skill.attack_skills
-- ============================================================
local CONFIG = require 'config.entry_config'                              -- [110]
local AttackSkillObjects = require 'data.tables.skill.attack_skills'      -- [120]

-- ============================================================
-- [2xx] 阶段2：工具模块 — 纯函数/工厂，部分有隐式依赖
-- 依赖链: [230]boot_combat → [220]boot_helpers → [221]boot_utils → [222]bonds_chain
-- ⚠ D1/D2: bonds_chain 通过 boot_utils 隐式加载，不可在此阶段之前移动
-- ============================================================
local GearUpgrades     = require 'runtime.gear_upgrades'                  -- [240]
local BootCore         = require 'runtime.boot_core'                      -- [210]
local BootCombat       = require 'runtime.boot_combat'                    -- [230]
local BootHelpers      = require 'runtime.boot_helpers'                   -- [220]→[221]→[222]
local BootUIEnhancements = require 'runtime.boot_ui_enhancements'        -- [250]
local EventBus         = require 'runtime.event_bus'                      -- [260] 零依赖
```

每行 require 末尾的 `-- [NNN]` 标记作为快速索引，与文档编号对应。不需要为每个 require 写冗长注释，只需在阶段头标注依赖链。

## 五、关于 `[显示层]` / `[动作层]` 前缀

**不建议引入**，理由：

- 7xx 阶段自然聚集了所有 UI 模块（ui_system, attr_tips_panel），6xx 聚集了所有战斗逻辑模块。层级通过编号百位已经区分
- `_G.SYSTEM.ui` vs `_G.SYSTEM.battle` 命名空间已提供足够的边界
- 引入中文前缀会增加文件命名复杂度，且用户明确不需要重命名文件

## 六、改进建议

### 优先级 1：消除 D1（高隐式依赖）

将 bonds_chain 改为 boot.lua 显式 require，而非通过 3 层链隐式加载：

```lua
-- 当前（隐式）:
-- boot_combat → boot_helpers → boot_utils → bonds_chain

-- 建议（显式）:
local BondChain = require 'runtime.bonds_chain'  -- [222] 显式声明
```

### 优先级 2：消除 D2（boot_helpers 薄包装）

评估是否可以直接合并 `boot_helpers.lua`（2行）到 `boot_utils.lua`，或直接在 boot.lua 中 require boot_utils。

### 优先级 3：D7 加断言

在 `session_state.create()` 调用前加显式断言，确保 6 个依赖系统已就绪，避免 nil 静默传播到运行期。
