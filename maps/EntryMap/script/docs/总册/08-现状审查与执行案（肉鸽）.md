# 肉鸽守线：现状审查与执行案

## 1. 现状审查结论

### 1.1 已实现（可直接复用）

- 启动链路：`main.lua -> entry_runtime.lua -> runtime/boot.lua`
- 局外/局内切换：`outgame_system.enter_outgame()` + `session_state_system.start_selected_stage(...)`
- 战斗主循环：波次推进、战场更新、循环驱动（`runtime/boot_loops.lua` 挂载）
- 成长构筑：`G` 强化、`F` 羁绊抽取、宝物/烙印奖励队列
- HUD 与面板：运行时 HUD、属性面板、背包/图鉴相关入口
- 调试能力：GM 面板、事件流提示、开发命令注入

### 1.2 未完成（当前主阻塞）

- 正式中央决策面板（目前以多入口散落触发为主）
- 成长武器与词缀的完整闭环（已有模块，体验尚未收敛）
- 正式结算页、奖励记录页
- 挑战奖励写回与局外长期成长回流的完整闭环
- 多英雄与章节专属内容扩展

### 1.3 风险与技术债

- 文档路径已迁移到 `docs/总册`，旧规则仍引用 `docs/开发进度与计划`，存在认知偏差风险
- `entry_runtime.lua` 体量较大、耦合面广，继续叠加功能会提高回归成本
- UI 根节点显隐由运行时强控（`enforce_runtime_ui_phase`），新增面板时容易遗漏同步

## 2. 执行案（Phase 拆分）

## Phase 1：物编与数据基线（`y3-obj-gen` / `y3-obj-edit`）

| 模块 | 功能 | 验收点 | 参考 API/来源 | 参考事件 |
|---|---|---|---|---|
| 英雄基线 | 固化当前单英雄起始属性、成长区间 | 新局出生属性稳定，重开一致 | 运行时读取入口：`runtime/hero_attr_system.lua` | `游戏-初始化` |
| 敌人与 Boss | 统一波次敌人/Boss 的基础模板与奖励 | 5 波 + Boss 切换稳定，奖励不丢 | 战场驱动：`runtime/battlefield.lua` | 波次开始/单位死亡相关事件 |
| 奖励对象 | 宝物/烙印对象字段补齐（品质、描述、唯一标识） | 奖励面板展示信息完整 | 奖励运行时：`runtime/rewards.lua` | 奖励入队/出队触发点 |

## Phase 2：UI 正式化（`y3-ui-pipeline`）

| 模块 | 功能 | 验收点 | 参考 API/来源 | 参考事件 |
|---|---|---|---|---|
| 决策面板 | 统一承接 G/F/宝物/烙印入口 | 任一奖励触发后均落到同一面板流程 | UI 访问：`y3.ui.get_ui(...)`（见 `entry_runtime.lua`） | 奖励可选时机触发 |
| 结算页 | 增加胜利/失败正式结算页 | 对局结束后稳定弹出结算，不与 HUD 冲突 | UI 显隐：`ui:set_visible(...)` | 对局结束事件 |
| 记录页 | 战斗内奖励记录/简报页 | 可查看当局关键获取记录 | 运行时 UI helpers：`runtime/runtime_ui_helpers.lua` | 奖励结算时刷新 |

## Phase 3：核心玩法收敛（`y3-lua-pipeline`）

| 模块 | 功能 | 验收点 | 参考 API/来源 | 参考事件 |
|---|---|---|---|---|
| 奖励队列统一 | 合并多入口奖励触发顺序与互斥 | 不出现多面板抢焦点、卡死 | `runtime/rewards.lua`、`ensure_round_choice_available` 调用链 | 奖励触发类事件 |
| 成长武器闭环 | 补全成长武器升级、提示、收益结算 | 升级可感知、属性变化与提示一致 | `runtime/gear_upgrades.lua` | 升级/击杀/波次推进 |
| 局内状态机 | 明确局外/局内/结算阶段迁移 | 任意中断后可恢复到正确阶段 | `runtime/boot_session.lua` | 游戏初始化、开始战斗、结束战斗 |

## Phase 4：扩展机制（`y3-lua-pipeline`）

| 模块 | 功能 | 验收点 | 参考 API/来源 | 参考事件 |
|---|---|---|---|---|
| 挑战奖励写回 | 完成挑战后写回局外成长数据 | 下局可看到成长结果 | `runtime/outgame_hero_growth.lua`、`ui/outgame` | 挑战完成事件 |
| Boss 时间轴深化 | Boss 阶段行为与提示完善 | Boss 战有阶段反馈和策略窗口 | `runtime/battlefield.lua` + 事件提示系统 | Boss 进场/血线阈值 |
| 多英雄扩展接口 | 保留单英雄玩法，补多英雄接入点 | 新英雄能复用现有成长/奖励链路 | `runtime/boot_session.lua`、`runtime/hero_form_skills.lua` | 选英雄/开局事件 |

## Phase 5：Lua 审查（`y3-lua-review`）

- 审查范围：`runtime/*.lua`、`entry_runtime.lua`、`ui/*.lua`
- 必查项：API 合规、事件名一致性、空指针保护、UI 路径有效性
- 通过标准：无臆造 API；关键主链（启动、波次、奖励、结算）无高危问题

## Phase 6：测试验收（`y3-auto-test`）

- 冒烟：开局 -> 进入战斗 -> 过至少 1 波 -> 触发 1 次奖励选择 -> 正常返回 HUD
- 回归：G/F/宝物/烙印四种入口均能走通，不互相打断
- 结算：胜利与失败各跑 1 次，检查结算页、记录页、局外写回
- 报告：输出测试案与测试报告（MD + HTML）

## 3. 执行优先级（建议）

1. P0：Phase 2 决策面板 + Phase 3 奖励队列统一
2. P1：Phase 2 结算页 + Phase 4 挑战写回
3. P2：Phase 4 Boss 深化 + 多英雄扩展

## 4. 下一步

若按此执行案推进，第一批落地建议从 P0 开始：

1. 统一奖励入口（先不改数值）
2. 补决策面板 UI 节点树
3. 连通队列与 UI 交互
