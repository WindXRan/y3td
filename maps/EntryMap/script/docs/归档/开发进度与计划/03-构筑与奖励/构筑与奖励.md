说明当前局内构筑系统、奖励系统和高价值轮次的完成度，明确哪些已经跑通，哪些还只是设计目标。

## 已实现

- 英雄升级、经验、技能点和 `G` 攻击技能强化已经接入真实 runtime。
- 攻击技能拥有真实运行时状态、槽位、解锁逻辑和升级选择逻辑；当前已跑通单英雄自动战斗与 `1 个固定基础位 + 4 个自由攻击技能位` 主链路。
- `F` 羁绊抽卡、羁绊卡收集、羁绊效果计算和动态 bonus 已由 `runtime/bonds.lua` 接入。
- 当前羁绊位上限为 `7`；羁绊补齐后会按 `result_type` 进入“占槽保留”或“整套吞噬”两条路径；满足配方时会自动把已成型羁绊进化为上位羁绊。
- 羁绊整套吞噬已经接入真实 runtime：完整羁绊会从当前羁绊格子中移出，进入独立的“已吞噬列表”，并持续保留原有单卡属性、成套属性和动态 runtime 加成。
- 宝物 `3 选 1`、`3` 宝物位替换、烙印 `10/20/30/40` 级节点、烙印 `3 选 1` 和 `reward_queue` 排队处理已经进入真实 runtime。
- 宝物、烙印、奖励队列、奖励倍率和被动收入修正已经从总入口协调层拆到 `runtime/rewards.lua`，总入口只保留协调与装配。
- 高价值轮次之间已经有互斥和恢复逻辑：`G`、`F`、宝物、烙印不会彼此直接覆盖待选状态。
- 已落地 `连射猎杀 / 五行共鸣 / 连锁清场 / 不屈反打` 四条主推路线的新增羁绊、配方羁绊和局内实际效果。
- 五个基础攻击技能已经拥有路线化升级项、敌人状态效果和中后期质变节点，不再只是数值强化。
- 战术型临时宝物已经接入真实 runtime，支持 `wave / timed / charges / next_boss / next_challenge` 五种寿命。

## 部分实现

- 当前候选交互主要依赖文本消息、HUD 按钮和数字键，不是正式卡面或中央决策面板体验。
- `reward_queue` 当前主要服务宝物和烙印；`G`、`F` 则依赖各自的 pending round 和恢复逻辑，系统层已经可用但交互层仍偏原型。
- 当前攻击技能、羁绊、宝物、烙印的内容量只够首版闭环，距离设计稿中的更大技能池、更多路线和更丰富 build 内容还有明显差距。
- 现有构筑系统已经可以并存，但仍缺与背包、奖励记录、正式结算摘要的完整联动。

## 未实现

- 设计稿中更大规模的攻击技能池、更多 build 分支和多英雄差异化技能树。
- 局外解锁等级驱动的更完整内容解锁链路。

## 后续开发建议

1. 先接成长武器和词缀，让 `金币 -> 稳定成长` 真正成为闭环，再扩更多构筑内容。
2. 继续保持“静态对象在 `entry_objects`、运行时状态在 `runtime/*.lua`”的分层，不要把新系统塞回总文件。
3. 高价值轮次新增系统时，先明确与 `G / F / 宝物 / 烙印` 的互斥和恢复关系，再落 UI。
