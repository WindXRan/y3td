# 关卡波次配表与实现映射

> 基于《关卡与波次设计》《战斗系统》《UI 与信息表现》整理，用于把 `5 波主线`、`主怪刷新`、`Boss 时间轴`、`区域布局` 与 `挑战模式扩展接口` 收口成可直接落表、落逻辑、落 UI 状态机的实现侧规范。

## 1. 目标与范围
- 本文只关注主线关卡结构相关的实现映射：
  `模式定义`
  `波次定义`
  `主怪刷新曲线`
  `Boss 本体与时间轴`
  `刷新区域`
  `奖励与切波结算`
- 本文不再重复解释玩法愿景，而是重点回答：
  `关卡与波次要拆哪些表`
  `运行时对象怎么组织`
  `Boss 死亡切波与残留怪共存怎么落地`
  `挑战模式后续扩展预留哪些接口`

## 2. 通用约定

### 2.1 统一术语
- `stage_mode`
  指一套完整关卡模式，如 `standard`、`challenge`。
- `wave`
  指主线固定波次，首版标准模式固定 `5` 波。
- `spawn_stage`
  指单波内部的刷怪阶段，如 `preheat`、`pressure`、`sprint`、`post_boss`。
- `boss_profile`
  指某一波对应 Boss 的静态定义。
- `timeline_event`
  指 Boss 时间轴上的单条技能调度记录。
- `reward_bundle`
  指某一波 Boss 或模式结算所发放的一组奖励定义。

### 2.2 字段类型约定
- 所有 `*_sec` 字段统一使用秒。
- 所有 `*_count` 字段统一使用整数。
- 所有 `*_factor` 字段都表示相对倍率，设计稿中按 `1.00` 语义描述；真正落表时再按项目数值规范转为对应格式。
- 所有 `*_tags` 字段建议用 `|` 分隔的标签串，便于筛选与权重修正。

### 2.3 设计边界
- 配表负责静态定义：波次顺序、刷新曲线、Boss 技能节奏、奖励包、区域引用。
- 运行时对象负责当前局状态：当前波次、Boss 是否已刷出、当前普通怪同屏数量、已结算奖励等。
- UI 只读取运行时对象和配表结果，不自行保存“当前到了第几波”或“Boss 是否已进入低血”。
- `普通怪残留`、`挑战怪残留` 与 `Boss 已死切下一波` 必须允许并存；不能依赖“切波前一定清场”这种前提。

## 3. 模式与波次主表

### 3.1 建议拆表
首版建议至少拆 `6` 张主表：
- `stage_mode_base`
- `wave_base`
- `wave_reward_bundle`
- `battlefield_cap_profile`
- `spawn_area_profile`
- `challenge_mode_rule`

补充约定：
- `challenge_mode_rule` 用于给挑战模式预留接口，首版即使不完整实现，也建议先定表结构。
- `spawn_area_profile` 同时供主线、Boss、挑战事件复用，避免区域 ID 在多个文档中各写一套。

### 3.2 `stage_mode_base`
用途：定义“这套模式总共几波、每波多久、胜负怎么判定”。

| 字段 | 含义 |
| --- | --- |
| `mode_id` | 模式唯一 ID，如 `standard`、`challenge` |
| `mode_name` | 模式名称 |
| `total_wave_count` | 总波次数；首版固定 `5` |
| `wave_duration_sec` | 每波怪物阶段时长；首版固定 `240` |
| `boss_spawn_sec` | Boss 登场时间；首版固定 `240` |
| `victory_rule` | 胜利条件，如 `kill_wave_5_boss` |
| `fail_rule` | 失败条件，如 `hero_dead` |
| `cap_profile_id` | 战场容量配置 ID |
| `challenge_rule_id` | 若是挑战模式，对应挑战规则集 ID |
| `ui_mode_title` | UI 显示标题 |

### 3.4 `wave_reward_bundle`
用途：定义“这一波 Boss 死亡后立刻结算什么奖励”。

| 字段 | 含义 |
| --- | --- |
| `reward_bundle_id` | 奖励包 ID |
| `gold_reward` | 金币奖励 |
| `wood_reward` | 木材奖励 |
| `exp_reward` | 经验奖励 |
| `special_reward_kind` | 特殊奖励种类，如 `none`、`treasure_choice`、`chapter_reward` |
| `special_reward_payload` | 特殊奖励内容 |
| `queue_priority` | 若进入高价值队列，对应优先级 |
| `reward_desc` | 奖励说明 |

首版建议：
- `第 1 到 4 波` 主要承担“继续打下一波”的补给作用。
- `第 5 波` 主要承担胜利结算和局外奖励写入，不再提供承接下一波的中期补给心智。

## 4. 主怪刷新曲线

### 4.1 建议拆表
首版建议拆 `2` 张表：
- `wave_spawn_curve`
- `wave_spawn_modifier`

其中：
- `wave_spawn_curve` 定义波内分段节奏。
- `wave_spawn_modifier` 定义波次内被曲线切换时所应用的属性修正。

### 4.2 `wave_spawn_curve`
用途：定义“同一波在不同时间段刷多快、每批刷多少、Boss 后是否继续刷”。

| 字段 | 含义 |
| --- | --- |
| `spawn_curve_id` | 刷新曲线 ID |
| `stage_key` | 阶段键：`preheat`、`pressure`、`sprint`、`post_boss` |
| `stage_order` | 阶段顺序 |
| `start_sec` | 本阶段开始时间 |
| `end_sec` | 本阶段结束时间 |
| `spawn_interval_sec` | 刷新间隔 |
| `spawn_batch_count` | 每批数量 |
| `modifier_id` | 属性修正配置 ID |
| `continue_if_boss_alive` | Boss 存活时是否继续生效 |
| `note` | 备注 |

首版 `wave_1` 示例：

### 4.3 `wave_spawn_modifier`
用途：定义“刷怪阶段切换时，对该波主怪附加哪些数值修正”。

| 字段 | 含义 |
| --- | --- |
| `modifier_id` | 修正 ID |
| `hp_factor` | 生命系数 |
| `damage_factor` | 伤害系数 |
| `move_speed_factor` | 移速系数 |
| `extra_tags` | 补充标签 |
| `note` | 备注 |

首版建议：
- 第 `1 波` 主要用 `刷新密度` 体现加压，不明显抬伤害。
- 第 `3 波` 的 `远程压制感` 主要通过攻击方式体现，不建议再同步暴涨批次数量。
- 第 `4 到 5 波` 再明显提高 `hp_factor`，形成堆场时间压力。

### 4.4 刷怪调度实现约定
- 主怪刷新不是“每波提前列出 240 秒内每一批次”，而是按 `wave_spawn_curve` 动态取当前阶段并滚动调度。
- 当 `main_alive_count >= main_alive_cap` 时，本轮批次延后，不吞掉已经排队的未来阶段。
- 当总单位逼近 `battlefield_cap_profile.total_unit_soft_cap` 时，也只延后当前主线新批次。
- Boss 刷出后，`post_boss` 阶段接管后续普通怪刷新节奏。
- 切到下一波后，旧波的主怪残留不再继续按旧波曲线追加新批次。

- `boss_base`
- `boss_skill_slot`
- `boss_timeline_event`
- `boss_low_hp_profile`

### 5.3 `boss_skill_slot`
用途：把“技能语义角色”和“具体技能 ID”分开，方便时间轴重排。

| 字段 | 含义 |
| --- | --- |
| `boss_profile_id` | 对应 Boss 配置 ID |
| `slot_id` | 技能槽位 ID |
| `slot_type` | 槽位类型：`basic_threat`、`high_threat`、`utility`、`summon` |
| `ability_id` | 实际技能 ID |
| `ui_warn_text` | UI 提示文本 |
| `cast_ai_type` | 施法 AI 类型，如 `toward_hero`、`self_center`、`target_area` |
| `enabled` | 是否启用 |

首版约定：
- 第 `1 到 2 波` Boss 可只用 `2` 到 `3` 个槽位。
- 第 `3 波` 起允许启用 `utility / summon` 槽位。

### 5.4 `boss_timeline_event`
用途：定义“技能首次什么时候放、后续隔多久放、低血时是否被改写”。

| 字段 | 含义 |
| --- | --- |
| `timeline_profile_id` | 时间轴模板 ID |
| `event_id` | 事件 ID |
| `slot_id` | 对应技能槽位 ID |
| `first_cast_sec` | 首次施放时间 |
| `repeat_interval_sec` | 循环间隔 |
| `min_gap_tag` | 与其他高威胁技能的最小错峰标签 |
| `cast_condition` | 施法条件，如 `always`、`boss_alive` |
| `priority` | 调度优先级 |
| `note` | 备注 |

| `timeline_wave_1_boss` | `wave_1_smash` | `skill_smash` | `11` | `18` | `high_threat_gap` | `boss_alive` | `1` | `头目重砸` |

| `timeline_wave_2_boss` | `wave_2_guard` | `skill_summon_guard` | `13` | `22` | `summon_gap` | `boss_alive` | `2` | `护卫征召` |

### 5.5 `boss_low_hp_profile`
用途：定义“低血时哪些技能改频率、改数量、改持续时间”。

| 字段 | 含义 |
| --- | --- |
| `low_hp_profile_id` | 低血配置 ID |
| `trigger_hp_ratio` | 触发血线；首版统一 `0.35` |
| `target_event_id` | 作用的时间轴事件 ID |
| `modify_type` | 修改类型，如 `interval_sec`、`summon_count`、`duration_sec` |
| `modify_op` | `set`、`add`、`mul` |
| `modify_value` | 修改值 |
| `extra_effect_desc` | 额外说明 |

### 5.6 时间轴实现约定
- 时间轴基于 `Boss 实际出生时间` 计算，而不是基于开波时间。
- 若 Boss 因控制、位移、索敌失败错过某次施法窗口，则下一次循环应从“实际施放成功时间”重新起算。
- `high_threat` 与 `summon / utility` 技能首发至少错开 `3 到 5 秒`。
- 低血强化只改现有技能节奏，不临时塞入完全陌生的大机制。
- 召唤出来的护卫要计入战场总单位软上限，但不能因此取消 Boss 核心技能。

## 6. 运行时对象与状态机

### 6.1 `stage_runtime`
用途：保存本局当前主线模式运行状态。

建议结构：

```lua
stage_runtime = {
  mode_id = 'standard',
  state = 'running_wave',
  current_wave_index = 1,
  total_wave_count = 5,
  stage_start_time = 0,
  cap_profile_id = 'cap_profile_standard',
  selected_challenge_modifier_ids = {},
}
```

### 6.2 `wave_runtime`
用途：保存当前波次从开波到 Boss 死亡切波的完整状态。

建议结构：

```lua
wave_runtime = {
  wave_id = 'wave_1',
  wave_index = 1,
  state = 'main_spawn',
  wave_start_time = 0,
  boss_spawn_sec = 240,
  boss_spawned = false,
  boss_dead = false,
  active_spawn_stage_key = 'preheat',
  next_main_spawn_time = 0,
  main_alive_count = 0,
  total_main_spawned = 0,
  reward_bundle_id = 'reward_wave_1_boss',
  boss_profile_id = 'boss_wave_1',
}
```

### 6.3 `boss_runtime`
用途：保存当前波 Boss 的时间轴、低血状态和死亡结算状态。

建议结构：

```lua
boss_runtime = {
  boss_profile_id = 'boss_wave_1',
  state = 'inactive',
  spawn_time = nil,
  unit_id = nil,
  low_hp_triggered = false,
  next_cast_time_by_event = {},
  last_cast_time_by_event = {},
  summon_alive_count = 0,
  reward_claimed = false,
}
```

### 6.4 波次状态机
- `preparing`
  波次刚创建，还未进入主怪刷新。
- `main_spawn`
  主怪按曲线正常刷新，Boss 尚未出生。
- `boss_fight`
  Boss 已出生，主怪按 `post_boss` 阶段继续刷新。
- `reward_settling`
  Boss 已死，正在发放奖励并准备切下一波。
- `closed`
  当前波逻辑已交棒给下一波或胜利结算。

流转规则：
- `preparing -> main_spawn`
  当前波正式开始。
- `main_spawn -> boss_fight`
  达到 `boss_spawn_sec` 且 Boss 创建成功。
- `boss_fight -> reward_settling`
  Boss 死亡。
- `reward_settling -> closed`
  奖励包已经完成入账或写入队列。

### 6.5 模式状态机
- `preparing`
  模式初始化中。
- `running_wave`
  主线某一波进行中。
- `victory`
  第 `5 波 Boss` 已击败。
- `failed`
  英雄死亡或挑战模式触发失败词条。

关键约定：
- `boss_dead` 是切下一波的唯一主触发点，不要求普通怪或挑战怪全部清空。
- 切下一波前必须先完成本波 Boss 奖励的入账与高价值奖励写队列。
- 切下一波后，上一波残留普通怪只作为残留单位存在，不再继续参与上一波刷新逻辑。

## 7. 挑战模式扩展接口

### 7.1 `challenge_mode_rule`
用途：定义“挑战模式是否仍沿用标准 5 波，以及本局如何挂接挑战词条包”。

| 字段 | 含义 |
| --- | --- |
| `challenge_rule_id` | 规则集 ID |
| `base_mode_id` | 继承的基础模式 ID；首版固定 `standard` |
| `modifier_pick_count` | 开局可选挑战词条数量 |
| `modifier_pool_id` | 挑战词条池 ID |
| `reward_bonus_profile_id` | 挑战额外奖励配置 ID |
| `fail_rule_override` | 是否覆盖基础失败条件 |
| `enabled` | 是否启用 |

实现建议：
- 挑战模式应尽量复用 `wave_base`、`boss_base`、`wave_spawn_curve`。
- 挑战词条只改动：
  `怪物系数`
  `Boss 系数`
  `恢复规则`
  `资源掉落`
  `战场规则`
- 挑战模式不要再复制一套新的 5 波主线表，优先做“标准模式表 + 挑战词条修正”。

## 8. UI 与逻辑映射

### 8.1 顶部波次区
- 只读 `stage_runtime.current_wave_index` 与当前 `wave_runtime.boss_spawn_sec`。
- `Boss 来临倒计时` 在 `main_spawn` 状态下持续显示。
- 进入 `boss_fight` 后，顶部倒计时切为 `Boss 血条 + 波次信息` 的并行展示。

### 8.2 Boss 血条与危险提示
- 只读 `boss_runtime.state`、`low_hp_triggered` 与当前施法事件。
- 当 `boss_timeline_event` 命中高威胁技能时，可在 Boss 血条附近显示简短技能名。
- 这类提示不应覆盖顶部波次主信息区。

### 8.3 右侧任务追踪与奖励结算
- 当前波目标说明读取 `wave_base.objective_desc`。
- Boss 死亡后的高价值奖励不直接靠 UI 记忆，必须先写入 `reward_queue` 或立即入账。
- 切波时若存在未处理高价值奖励，只允许“切波编号先更新、奖励仍保留待领”，不允许奖励丢失。

## 9. MVP 最小数据量
- `stage_mode_base`：`2` 条
- `wave_base`：`5` 条
- `wave_reward_bundle`：`5` 条
- `battlefield_cap_profile`：`2` 条
- `spawn_area_profile`：`10` 条左右
- `wave_spawn_curve`：`20` 条
- `wave_spawn_modifier`：`15` 到 `20` 条
- `boss_base`：`5` 条
- `boss_skill_slot`：`12` 到 `16` 条
- `boss_timeline_event`：`12` 到 `16` 条
- `boss_low_hp_profile`：`6` 到 `10` 条
- `challenge_mode_rule`：`1` 到 `2` 条

## 10. 推荐实现顺序
1. 先把 `stage_mode_base + wave_base + stage_runtime + wave_runtime` 跑通，确认 `5 波 -> Boss -> 切下一波` 的主流程成立。
2. 再接 `wave_spawn_curve`，让普通怪从“固定间隔”升级为“波内分段节奏”。
3. 然后接 `boss_base + boss_skill_slot + boss_timeline_event`，先实现固定时间轴循环。
4. 再补 `boss_low_hp_profile` 和 `battlefield_cap_profile`，解决低血强化与同屏软上限。
5. 最后接 `challenge_mode_rule`，把挑战模式做成对标准 5 波的覆写层，而不是复制一套独立关卡。
