# 局内系统配表与实现映射

> 基于《局内成长系统》《资源与经济设计》《UI 与信息表现》整理，用于把设计稿收口为可直接落表、落代码的实现侧规范。

## 1. 目标与范围
- 本文档不重复解释玩法愿景，而是负责回答三个问题：
  `数据表要怎么拆`
  `运行时对象要存什么`
  `UI / 逻辑 / 配表之间如何对齐`
- 首版只覆盖当前最关键的局内系统：
  `成长武器`
  `挑战事件`
  `高价值奖励队列`
- 其他系统如 `攻击技能强化卡池`、`羁绊卡池`、`宝物池` 已在原文档具备较完整定义，后续如要继续工程化，可按本文档同样方式再补。

## 2. 通用约定

### 2.1 命名规则
- 所有 `id`、枚举值、标签值统一使用 `ASCII + snake_case`。
- 所有展示给玩家的文案统一保留中文字段，不把中文直接当主键使用。
- 推荐约定：
  `*_id` 表示唯一主键
  `*_name` 表示展示名称
  `*_desc` 表示展示说明
  `*_tags` 表示标签集合

### 2.2 字段类型约定
- `time`、`duration`、`interval` 一律使用“秒”作为单位。
- 数值字段优先拆成：
  `base_value`
  `bonus_value`
  `ratio_value`
  避免同一列里混填“固定值”和“百分比值”。
- 需要表达多个 ID 时，首版建议用 `|` 分隔的字符串列表，例如：
  `multi_shot|basic_attack|aoe`
- 运行时会频繁判断的字段，不建议只写自由文本；优先落成枚举或标签。

### 2.3 表与运行时的职责边界
- 配表只负责“静态定义”，例如费用、权重、刷新批次。
- 运行时对象只负责“当前局状态”，例如本轮已花金币、事件倒计时、当前候选。
- UI 不保存真状态，只读取运行时对象与配表结果做展示。

## 3. 成长武器

### 3.1 建议拆表
首版建议至少拆 5 张表：
- `growth_weapon_base`
- `growth_weapon_level`
- `growth_weapon_node`
- `growth_weapon_affix`
- `growth_weapon_affix_pool`

### 3.2 `growth_weapon_base`
用途：定义“这把武器是什么”。

| 字段 | 说明 |
| --- | --- |
| `weapon_id` | 武器唯一 ID |
| `weapon_name` | 武器名称 |
| `init_quality` | 初始品质 |
| `init_level` | 初始等级 |
| `max_level` | 最大等级 |
| `base_desc` | 基础描述 |
| `ui_icon` | UI 图标资源 |
| `theme_tags` | 主题标签 |

首版示例：

| weapon_id | weapon_name | init_quality | init_level | max_level | base_desc | ui_icon | theme_tags |
| --- | --- | --- | --- | --- | --- | --- | --- |
| `hunter_bow` | `狩猎长弓` | `common` | `1` | `50` | `稳定成长的基础武器。` | `icon_hunter_bow` | `bow|starter|growth` |

### 3.3 `growth_weapon_level`
用途：定义“每一级要花多少金币、涨什么基础成长”。

| 字段 | 说明 |
| --- | --- |
| `weapon_id` | 对应武器 ID |
| `level_from` | 当前等级 |
| `level_to` | 升级后等级 |
| `gold_cost` | 本次升级金币费用 |
| `bonus_attack` | 本级攻击成长 |
| `bonus_hp` | 本级生命成长 |
| `bonus_armor` | 本级护甲成长 |
| `bonus_all_attr` | 本级全属性成长 |
| `node_type` | 节点类型：`none`、`affix` |
| `node_pool_id` | 若为节点，对应词缀池 ID |

首版约定：
- `1 -> 10` 使用 `25`
- `11 -> 20` 使用 `35`
- `21 -> 30` 使用 `45`
- `31 -> 40` 使用 `60`
- `41 -> 50` 使用 `80`

### 3.4 `growth_weapon_node`
用途：定义“哪些等级会弹出词缀节点，以及这一轮怎么生成”。

| 字段 | 说明 |
| --- | --- |
| `weapon_id` | 对应武器 ID |
| `node_level` | 节点等级 |
| `node_type` | 首版固定 `affix_choice` |
| `choice_count` | 候选数量；首版固定 `4` |
| `free_refresh_count` | 免费刷新次数；首版固定 `3` |
| `affix_pool_id` | 对应词缀池 |
| `ui_title` | 节点标题文案 |

首版示例：

| weapon_id | node_level | node_type | choice_count | free_refresh_count | affix_pool_id | ui_title |
| --- | --- | --- | --- | --- | --- | --- |
| `hunter_bow` | `10` | `affix_choice` | `4` | `3` | `bow_affix_pool_lv10` | `10级词缀选择` |
| `hunter_bow` | `20` | `affix_choice` | `4` | `3` | `bow_affix_pool_lv20` | `20级词缀选择` |

### 3.5 `growth_weapon_affix`
用途：定义“单个词缀是什么效果”。

| 字段 | 说明 |
| --- | --- |
| `affix_id` | 词缀唯一 ID |
| `affix_name` | 词缀名称 |
| `quality` | 品质 |
| `effect_type_1` | 主效果类型 |
| `effect_value_1` | 主效果数值 |
| `effect_type_2` | 副效果类型 |
| `effect_value_2` | 副效果数值 |
| `effect_desc` | 展示描述 |
| `is_unique_in_run` | 是否单局唯一 |
| `type_tags` | 类型标签 |

首版示例：

| affix_id | affix_name | quality | effect_type_1 | effect_value_1 | effect_type_2 | effect_value_2 | effect_desc | is_unique_in_run | type_tags |
| --- | --- | --- | --- | --- | --- | --- | --- | --- | --- |
| `atk_flat_s_30` | `攻击力 +30` | `common` | `attack_flat` | `30` |  |  | `攻击力 +30` | `false` | `flat|attack` |
| `multi_count_1` | `多重数量 +1` | `rare` | `multi_shot_count` | `1` |  |  | `多重数量 +1` | `true` | `unique|basic_attack` |

### 3.6 `growth_weapon_affix_pool`
用途：定义“某个节点从哪些词缀里抽、权重是多少”。

| 字段 | 说明 |
| --- | --- |
| `affix_pool_id` | 词缀池 ID |
| `affix_id` | 词缀 ID |
| `pool_weight` | 基础权重 |
| `allow_repeat` | 是否允许重复出现 |
| `require_tags` | 需要具备的标签 |
| `exclude_tags` | 冲突标签 |

实现建议：
- 若当前词缀 `is_unique_in_run = true` 且已获得，则本轮直接跳过。
- `allow_repeat` 与 `is_unique_in_run` 不重复表达同一件事：
  `is_unique_in_run` 控制整局唯一
  `allow_repeat` 控制同轮候选生成时是否允许重复投中

### 3.7 成长武器运行时对象
首版建议统一维护一个 `weapon_runtime`：

```lua
weapon_runtime = {
  weapon_id = 'hunter_bow',
  level = 1,
  max_level = 50,
  state = 'normal',
  total_gold_spent = 0,
  affix_list = {},
  pending_affix_round_id = nil,
  pending_affix_node_level = nil,
  free_refresh_left = 0,
}
```

运行时只做三件事：
- 记录当前等级与状态
- 记录当前已获得词缀
- 记录当前是否有未结算词缀轮次

### 3.8 成长武器最小实现顺序
1. 先做 `growth_weapon_base + growth_weapon_level`
2. 再做 `normal / pending_affix / max_level` 三态切换
3. 再接 `growth_weapon_node + growth_weapon_affix_pool`
4. 最后接入背包页签和武器格子 UI

## 4. 挑战事件

### 4.1 建议拆表
首版建议至少拆 4 张表：
- `challenge_base`
- `challenge_monster_group`
- `challenge_spawn_plan`
- `challenge_reward`

### 4.2 `challenge_base`
用途：定义“这个挑战怎么开启、持续多久、走哪套奖励”。

| 字段 | 说明 |
| --- | --- |
| `challenge_id` | 挑战唯一 ID |
| `challenge_name` | 挑战名称 |
| `duration_sec` | 持续时间；首版固定 `60` |
| `cost_charge` | 消耗挑战次数；首版固定 `1` |
| `monster_group_id` | 对应怪物组 |
| `reward_id` | 对应奖励定义 |
| `spawn_area_id` | 默认刷新区域 |
| `ui_icon` | 按钮图标 |
| `danger_tags` | 风险标签 |

首版示例：

| challenge_id | challenge_name | duration_sec | cost_charge | monster_group_id | reward_id | spawn_area_id | ui_icon | danger_tags |
| --- | --- | --- | --- | --- | --- | --- | --- | --- |
| `gold_trial` | `金币挑战` | `60` | `1` | `gold_trial_pack` | `reward_gold_trial` | `challenge_spawn_top` | `icon_trial_gold` | `fast|swarm` |
| `treasure_trial` | `宝物挑战` | `60` | `1` | `treasure_trial_pack` | `reward_treasure_trial` | `challenge_treasure_elite_spawn` | `icon_trial_treasure` | `elite|high_risk` |

### 4.3 `challenge_monster_group`
用途：定义“这一类挑战到底刷什么怪”。

| 字段 | 说明 |
| --- | --- |
| `monster_group_id` | 怪物组唯一 ID |
| `display_name` | 怪物组名称 |
| `elite_unit_id` | 首领单位 ID；无则为空 |
| `normal_unit_id` | 普通单位 ID |
| `theme_tags` | 主题标签 |
| `soft_alive_limit` | 软上限 |
| `despawn_on_fail` | 失败时是否清除 |

### 4.4 `challenge_spawn_plan`
用途：定义“每个挑战怪物组分几批、几点刷、从哪里刷”。

| 字段 | 说明 |
| --- | --- |
| `monster_group_id` | 对应怪物组 ID |
| `batch_index` | 批次编号 |
| `spawn_sec` | 事件开始后多少秒刷出 |
| `unit_role` | `elite` 或 `normal` |
| `spawn_count` | 本批次数量 |
| `spawn_area_id` | 刷新区域 |
| `scatter_min_distance` | 最小出生间距 |

首版示例：

| monster_group_id | batch_index | spawn_sec | unit_role | spawn_count | spawn_area_id | scatter_min_distance |
| --- | --- | --- | --- | --- | --- | --- |
| `gold_trial_pack` | `1` | `0` | `normal` | `7` | `challenge_spawn_top` | `120` |
| `gold_trial_pack` | `2` | `12` | `normal` | `7` | `challenge_spawn_top` | `120` |
| `gold_trial_pack` | `3` | `24` | `normal` | `6` | `challenge_spawn_top` | `120` |
| `treasure_trial_pack` | `1` | `0` | `elite` | `1` | `challenge_treasure_elite_spawn` | `180` |
| `treasure_trial_pack` | `2` | `0` | `normal` | `4` | `challenge_treasure_elite_spawn` | `140` |
| `treasure_trial_pack` | `3` | `20` | `normal` | `4` | `challenge_treasure_elite_spawn` | `140` |

### 4.5 `challenge_reward`
用途：定义“成功后给什么、是否进队列”。

| 字段 | 说明 |
| --- | --- |
| `reward_id` | 奖励唯一 ID |
| `reward_kind` | `gold`、`wood`、`exp`、`treasure_choice` |
| `reward_value` | 固定数值或固定包 ID |
| `queue_on_claim` | 是否进入高价值奖励队列 |
| `bonus_gold` | 附带金币 |
| `bonus_wood` | 附带木材 |
| `reward_desc` | 奖励描述 |

首版约定：
- `gold / wood / exp` 直接入账，不进队列。
- `treasure_choice` 进入奖励队列，等待中央面板空闲后展示。

### 4.6 挑战事件运行时对象
首版建议统一维护 `challenge_runtime_context`：

```lua
challenge_runtime_context = {
  instance_id = 'gold_trial_001',
  challenge_id = 'gold_trial',
  state = 'running',
  start_time = 120,
  end_time = 180,
  duration_sec = 60,
  spawn_area_id = 'challenge_spawn_top',
  monster_group_id = 'gold_trial_pack',
  spawned_unit_ids = {},
  alive_count = 20,
  total_count = 20,
  success_rule = 'kill_all',
  fail_reason = nil,
  reward_type = 'gold',
  reward_payload = { gold = 220 },
  ui_track_key = 'track_gold_trial_001',
}
```

### 4.7 挑战事件最小实现顺序
1. 先做 `challenge_base`
2. 再做 `ready / preparing / running / closed` 主流程
3. 再接 `challenge_spawn_plan`
4. 再补 `success_pending_reward` 与奖励队列
5. 最后接右侧追踪区与底部按钮状态

## 5. 高价值奖励队列

### 5.1 适用范围
首版建议把以下奖励统一走 `reward_queue`：
- `宝物候选`
- 后续若接入局内 `烙印候选`
- 其他需要占用中央决策面板且必须 `N 选 1` 的奖励

以下奖励不进队列：
- `经验`
- `金币`
- `木材`
- 普通飘字与统计更新

### 5.2 `reward_queue_entry`
用途：定义“哪个奖励要排队弹出、排在谁前面”。

| 字段 | 说明 |
| --- | --- |
| `reward_entry_id` | 奖励条目唯一 ID |
| `source_type` | 来源类型 |
| `source_instance_id` | 来源实例 ID |
| `reward_kind` | 奖励类型 |
| `priority` | 优先级 |
| `payload` | 奖励内容 |
| `state` | `pending`、`shown`、`claimed` |
| `created_time` | 创建时间 |

### 5.3 优先级建议
- `已有未完成中央面板` 优先级最高，不能被新奖励插队覆盖。
- 同样是队列中的新奖励时，首版建议：
  `宝物候选 > 其他高价值候选`
- 同优先级按 `created_time` 先进先出。

### 5.4 弹出规则
- 若当前没有 `G / F / 宝物候选 / 烙印` 等中央决策面板在占用，则队列头可立即弹出。
- 若当前中央面板忙碌，则只入队，不额外弹 toast 打断操作。
- 玩家完成当前面板选择后，系统立即检查队列头并决定是否弹出下一项。

## 6. UI 与逻辑映射

### 6.1 成长武器
- 底部 `成长武器格子`
  只负责显示 `等级 / 待选词缀 / 下一次费用`。
- 背包 `成长武器页签`
  负责执行 `强化 +1` 和处理 `pending_affix_round`。
- 真状态统一读取 `weapon_runtime`。

### 6.2 挑战事件
- 底部 `金币挑战 / 木材挑战 / 经验挑战 / 宝物挑战` 按钮
  只负责展示挑战入口状态。
- 右侧任务追踪区
  只负责展示 `ui_track_key` 对应的运行中实例。
- 真状态统一读取 `challenge_runtime_context`。

### 6.3 高价值奖励
- 中央弹窗只负责展示当前队列头。
- 队列顺序与是否可弹由 `reward_queue_entry` 决定，不应在 UI 层临时重排。

## 7. MVP 最小落地清单
若要先把这套系统最小跑起来，建议至少准备以下数据：
- `growth_weapon_base`：`1` 条
- `growth_weapon_level`：`49` 条
- `growth_weapon_node`：`5` 条
- `growth_weapon_affix`：`20` 到 `30` 条
- `growth_weapon_affix_pool`：覆盖 `5` 个节点池
- `challenge_base`：`4` 条
- `challenge_monster_group`：`4` 条
- `challenge_spawn_plan`：`10` 到 `14` 条
- `challenge_reward`：`4` 条

## 8. 推荐实现顺序
1. 先把 `成长武器` 跑通，因为它是金币主出口，最容易形成稳定闭环。
2. 再做 `挑战事件主流程`，先不接高价值奖励队列，只做金币/木材/经验直发。
3. 最后把 `宝物挑战 -> reward_queue -> 宝物候选面板` 串起来。

## 10. 羁绊进阶映射补充（2026-04）

### 10.1 分层与品质约定

- `tier` 表示目标设计层级：`t1 / t2 / t3`
- `quality` 表示当前条目显示品质
- 若某羁绊进入进化链，则其升级版本必须同步提升品质
- 当前 runtime 允许保留 `common / rare / epic` 混合品质；后续实现不要把 `tier` 和 `quality` 直接混成一个字段

### 10.2 建议新增 `bond_recipe`

## 11. 宝物补充映射（2026-04）

| `ui_state_hint` | 若为临时宝物，展示剩余波次/时间/次数的提示文案 |

### 11.2 战术型临时宝物运行时建议

首版可以统一维护一个 `treasure_runtime_buff`：

```lua
treasure_runtime_buff = {
  treasure_id = 'ITEM_004',
  treasure_type = 'tactical_temp',
  duration_type = 'timed',
  remaining_time = 30,
  remaining_charges = nil,
  expires_on_wave = nil,
  expires_on_boss = false,
  expires_on_challenge = false,
}
```

补充约定：

- `permanent` 类型不进入该运行时对象。
- `wave` 与 `next_boss`、`next_challenge` 可以共用同一套到期清理流程。
- 当前若举例临时宝物，应优先使用现行图鉴中的神符类宝物，而不是已废弃的旧版单文件宝物 ID。
- 文档阶段先把字段写清楚，不要求本轮立刻实现。
