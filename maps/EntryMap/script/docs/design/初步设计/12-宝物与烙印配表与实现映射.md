# 宝物与烙印配表与实现映射

> 基于《战斗系统》《局内成长系统》《UI 与信息表现》整理，用于把 `宝物候选` 与 `烙印选择` 收口成可直接落表、落逻辑、落 UI 状态机的实现侧规范。

> 2026-04 现状补充：当前战斗内宝物唯一事实源已切换为 `maps/EntryMap/script/data_csv/treasures.csv`。本文中出现的旧版单文件宝物 ID 与效果组示例，仅作为历史方案归档，不能再直接视为现行运行时事实。

## 1. 目标与范围
- 本文只关注两套高价值局内奖励系统：
  `宝物候选系统`
  `烙印节点系统`
- 本文不再重复解释它们为什么存在，而是重点回答：
  `要拆哪些表`
  `运行时状态怎么存`
  `和 reward_queue / 中央决策面板怎么协作`
  `替换、保底、去重和恢复怎么落地`

## 2. 通用约定

### 2.1 统一术语
- `treasure`
  指玩家当前局内持有或曾经获得过的单件宝物。
- `treasure_round`
  指一次完整的 `3 选 1` 宝物候选轮次。
- `mark`
  指一次等级节点中可被选中的单个烙印。
- `mark_round`
  指一次完整的 `3 选 1` 烙印选择轮次。
- `build_profile_tags`
  指根据当前 `攻击技能`、`羁绊`、`成长武器词缀`、`宝物` 汇总出的当前构筑标签快照，用于宝物权重修正。
- `central_panel`
  指和 `G`、`F`、成长武器词缀、宝物候选、烙印选择共用的中央决策面板资源。

### 2.2 状态持久化原则
- `宝物候选` 与 `烙印选择` 都属于高价值奖励，不允许只靠 UI 当前是否打开来判断是否存在。
- 配表负责静态定义；运行时对象负责保存本局真实状态；UI 只读运行时对象。
- 任意时刻最多只允许：
  `1 个未结算 treasure_round`
  `1 个正在显示的 mark_round`
- 但 `reward_queue` 中可以同时排队多个 `treasure_choice` 与 `mark_choice` 奖励条目。

### 2.3 与其他系统的边界
- `宝物` 是独立于 `成长武器`、`G`、`F` 之外的第三套构筑组件。
- `烙印` 不占用攻击技能格子，不参与羁绊计数，也不进入宝物位。
- `宝物候选` 支持 `稍后选择`，关闭后可从 `宝物` 按钮恢复。
- `烙印选择` 一旦正式弹出，不提供刷新，也不提供 `稍后选择`；但若触发节点时中央面板正忙，可先进入 `reward_queue` 排队。

### 3.1 建议拆表
首版建议至少拆 `5` 张表：
- `treasure_base`
- `treasure_effect_group`
- `treasure_pool_rule`
- `treasure_pick_weight_rule`
- `treasure_replace_rule`

补充约定：
- 若后续 `宝物挑战`、`Boss 奖励`、`章节固定奖励` 的来源规则差异明显，再补一张 `treasure_source_rule`。
- 首版来源还比较集中时，可先把来源差异合并进 `treasure_pool_rule`。

### 3.2 `treasure_base`
用途：定义“这件宝物是什么、主要做什么、选中后挂哪组效果”。

| 字段 | 含义 |
| --- | --- |
| `treasure_id` | 宝物唯一 ID |
| `treasure_name` | 宝物名称 |
| `quality` | 品质：`common`、`rare`、`epic` |
| `core_desc` | 核心效果描述 |
| `extra_desc` | 补充说明 |
| `effect_group_id` | 效果组 ID |
| `type_tags` | 职责标签，如 `attack`、`skill`、`survival`、`economy` |
| `build_tags` | 推荐路线标签，如 `basic_attack`、`multi_shot`、`spell_cycle` |
| `pool_weight_base` | 基础权重 |
| `is_unique_in_run` | 本局是否唯一；首版固定 `true` |
| `ui_order` | UI 展示顺序 |

### 3.3 `treasure_effect_group`
用途：定义“玩家拿到这件宝物后，到底改哪些运行时值”。

| 字段 | 含义 |
| --- | --- |
| `effect_group_id` | 效果组 ID |
| `seq` | 效果序号 |
| `effect_type` | 效果类型 |
| `effect_target` | 作用目标，如 `hero_attr`、`attack_skill_runtime`、`battle_event` |
| `effect_key` | 具体字段 |
| `effect_op` | `add`、`mul`、`set`、`unlock_listener` |
| `effect_value` | 数值 |
| `condition_tags` | 生效条件标签 |

历史示例已迁移到 [12-宝物旧版实现映射归档.md](./12-宝物旧版实现映射归档.md)。

实现建议：
- `宝物` 中已经出现明显的“监听事件再触发额外效果”需求，例如 `时隙沙漏`，因此首版就建议用 `effect_group`，不要继续把复杂宝物挤成单字段。
- `condition_tags` 用于处理像 `多重`、`弹射`、`技能释放后` 这类需要读当前构筑状态的效果。
- 旧版示例仅保留在归档文档中，不对应当前图鉴宝物实现。

### 3.4 `treasure_pool_rule`
用途：定义“不同来源、不同波次下，宝物候选的品质和保底怎么出”。

| 字段 | 含义 |
| --- | --- |
| `rule_id` | 规则 ID |
| `source_type` | 来源类型，如 `treasure_trial`、`boss_reward` |
| `wave_range` | 生效波次范围 |
| `choice_count` | 候选数量；首版固定 `3` |
| `common_weight` | 普通品质权重 |
| `rare_weight` | 稀有品质权重 |
| `epic_weight` | 史诗品质权重 |
| `guarantee_rare_after_miss` | 连续未出现 `稀有 / 史诗` 的保底轮次；首版固定 `2` |
| `allow_same_quality_repeat` | 同轮是否允许三张同品质；首版允许 |
| `enabled` | 是否启用 |

首版建议：
- `宝物挑战` 用统一品质权重，不按等级节点做额外阶梯。
- 单局内若连续 `2 次 treasure_round` 都没有出现 `rare / epic`，则下一轮至少强制 `1 件 rare`。
- 同轮不允许出现相同 `treasure_id`。

### 3.5 `treasure_pick_weight_rule`
用途：定义“当前局已经形成的构筑，如何把候选轻微推向相关宝物，但又不完全锁死路线”。

| 字段 | 含义 |
| --- | --- |
| `rule_id` | 规则 ID |
| `trigger_type` | 触发条件类型 |
| `trigger_value` | 触发值 |
| `target_tag` | 作用的 `build_tags` 或 `type_tags` |
| `weight_mul` | 权重乘数 |
| `note` | 备注 |

首版建议覆盖：
- 当前 `build_profile_tags` 命中宝物 `build_tags`：`x1.35`
- 当前宝物与已有宝物同职责但不是同一机制：`x1.10`
- 当前宝物与现有构筑明显无关：`x0.80`
- 当前已经拥有过或弃置过同一宝物：直接视为不可入池，而不是只降权

补充约定：
- `宝物` 的路线纠偏应该弱于 `G / F`，避免宝物把构筑完全拽成“只补当前最强流派”的自动解。
- 首版不建议让同一轮 `3 选 1` 全都命中同一职责标签；理想情况是 `1 张贴构筑 + 1 张通用 + 1 张偏转向`。

### 3.6 `treasure_replace_rule`
用途：定义“当 3 个宝物位已满时，系统如何给出推荐替换目标”。

| 字段 | 含义 |
| --- | --- |
| `rule_id` | 规则 ID |
| `trigger_type` | 条件类型 |
| `trigger_value` | 条件值 |
| `score_delta` | 推荐替换分数修正，越高越推荐被替换 |
| `ui_hint` | UI 提示 |

首版推荐分数规则：

| rule_id | trigger_type | trigger_value | score_delta | ui_hint |
| --- | --- | --- | --- | --- |
| `replace_common_quality` | `quality_is` | `common` | `40` | `优先替换低品质宝物` |
| `replace_rare_quality` | `quality_is` | `rare` | `10` | `可替换中品质宝物` |
| `replace_epic_quality` | `quality_is` | `epic` | `-25` | `谨慎替换高品质宝物` |
| `replace_no_route_match` | `build_match` | `none` | `20` | `当前路线适配度较低` |
| `replace_same_duty_lower_value` | `same_type_lower_quality` | `true` | `30` | `新宝物可覆盖同职责旧宝物` |
| `replace_engine_core` | `has_tag` | `engine_core` | `-35` | `该宝物属于核心机制件` |
| `replace_early_economy` | `wave_leq_and_has_tag` | `2,economy` | `-20` | `前两波经济宝物不建议轻易替换` |

实现建议：
- 系统只负责“推荐”，不锁定玩家手动选择。
- UI 应展示每个候选宝物对应的推荐替换目标，而不是只给一个全局推荐。
- 若替换会导致某些联动宝物价值明显下降，例如 `王冠残片` 的稀有/史诗件数下降，必须给出预览说明。

### 3.8 `treasure_runtime`
用途：保存当前局已经持有、已经弃置、以及品质保底所需的宝物状态。

建议结构：

```lua
treasure_runtime = {
  active_slots = {
    [1] = nil,
    [2] = nil,
    [3] = nil,
  },
  active_by_id = {},
  acquired_treasure_ids = {},
  discarded_treasure_ids = {},
  no_high_quality_rounds = 0,
}
```

补充字段建议：
- `active_by_id[treasure_id]`
  保存 `slot_index`、`source_type`、`source_instance_id`、`acquired_time`。
- `acquired_treasure_ids`
  用于禁止本局重复出现。
- `discarded_treasure_ids`
  用于禁止被替换宝物再次出现，并支持背包页签展示弃置记录。

### 3.9 `treasure_round_runtime`
用途：保存当前未结算的宝物候选轮次。

| 字段 | 含义 |
| --- | --- |
| `round_id` | 当前轮次唯一 ID |
| `reward_entry_id` | 对应的奖励队列条目 ID |
| `source_type` | 来源类型，如 `treasure_trial` |
| `source_instance_id` | 来源实例 ID |
| `candidate_treasure_ids` | 3 件候选宝物 ID |
| `state` | `pending`、`await_replace`、`resolved` |
| `selected_treasure_id` | 已选中的目标宝物 |
| `need_replace` | 是否需要替换已有宝物 |
| `recommended_replace_by_candidate` | 每个候选对应的推荐替换槽位 |
| `created_time` | 创建时间 |

关键约定：
- `宝物候选面板` 被关闭时，不改变 `state`，只改变 UI 是否显示。
- 同一时刻最多只允许 `1 个 state != resolved` 的 `treasure_round_runtime`。
- 若在当前轮未结算时又有新的 `treasure_choice` 奖励，只进入 `reward_queue`，不直接再创建第二个轮次。

### 3.10 宝物轮次状态机
- `idle`
  当前没有待选宝物轮次。
- `pending`
  候选已生成，等待玩家先选中 `1 件` 宝物。
- `await_replace`
  玩家已选中宝物，但当前 `3 个宝物位` 已满，必须再指定 `1 件` 被替换宝物。
- `resolved`
  本轮已经完成写入，等待收尾与 UI 关闭。

流转规则：
- `idle -> pending`
  `reward_queue` 弹出 `treasure_choice`，并成功生成本轮 `3 件候选`。
- `pending -> resolved`
  玩家选中宝物且当前存在空位。
- `pending -> await_replace`
  玩家选中宝物且当前 `3 个宝物位` 已满。
- `await_replace -> resolved`
  玩家完成替换目标选择。
- `resolved -> idle`
  `treasure_runtime` 已完成写入，队列条目标记为 `claimed`。

### 3.11 与 `reward_queue` 的协作
- `宝物挑战`、后续 `Boss 宝物奖励`、章节固定宝物奖励，都优先写成 `reward_kind = treasure_choice` 的奖励条目。
- 若当前没有其他中央决策面板占用，则 `reward_queue` 头部可立即实例化为 `treasure_round_runtime` 并弹出面板。
- 若当前已有 `G / F / 武器词缀 / 烙印 / 宝物候选` 占用中央面板，则只入队，不直接弹窗。
- 底部 `宝物` 按钮优先恢复当前 `pending / await_replace` 的宝物轮次；只有没有待选轮次时，才打开背包 `宝物` 页签。

推荐优先级：
- `mark_choice`：`95`
- `treasure_choice`：`90`

## 4. 烙印系统

### 4.1 建议拆表
首版建议至少拆 `4` 张表：
- `mark_node_base`
- `mark_base`
- `mark_pool_rule`
- `mark_effect_group`

补充约定：
- 首版不建议单独拆 `mark_pick_weight_rule`。
- 烙印的核心体验更偏“少次数、高辨识度、全局泛用”，不需要像 `G / F` 那样做重度路线偏置。
- 若后续发现烙印候选经常和当前构筑严重脱节，再补 `mark_pick_weight_rule`。

### 4.2 `mark_node_base`
用途：定义“哪些等级节点会触发烙印轮次，以及该轮的固定规则”。

| 字段 | 含义 |
| --- | --- |
| `node_id` | 节点唯一 ID |
| `trigger_level` | 触发等级，如 `10`、`20` |
| `choice_count` | 候选数量；首版固定 `3` |
| `pool_rule_id` | 对应烙印池规则 |
| `queue_priority` | 进入 `reward_queue` 时的优先级；建议 `95` |
| `allow_refresh` | 是否允许刷新；首版固定 `false` |
| `allow_defer` | 是否允许稍后选择；首版固定 `false` |
| `ui_title` | 面板标题 |

首版示例：

| node_id | trigger_level | choice_count | pool_rule_id | queue_priority | allow_refresh | allow_defer | ui_title |
| --- | --- | --- | --- | --- | --- | --- | --- |
| `mark_node_lv10` | `10` | `3` | `mark_pool_global` | `95` | `false` | `false` | `10级烙印选择` |
| `mark_node_lv20` | `20` | `3` | `mark_pool_global` | `95` | `false` | `false` | `20级烙印选择` |
| `mark_node_lv30` | `30` | `3` | `mark_pool_global` | `95` | `false` | `false` | `30级烙印选择` |
| `mark_node_lv40` | `40` | `3` | `mark_pool_global` | `95` | `false` | `false` | `40级烙印选择` |

### 4.3 `mark_base`
用途：定义“单个烙印本体是什么”。

| 字段 | 含义 |
| --- | --- |
| `mark_id` | 烙印唯一 ID |
| `mark_name` | 烙印名称 |
| `quality` | 品质：`common`、`rare`、`epic` |
| `core_desc` | 核心效果描述 |
| `extra_desc` | 补充说明 |
| `effect_group_id` | 效果组 ID |
| `positioning_tags` | 定位标签，如 `attack_spell`、`boss`、`combo` |
| `build_tags` | 推荐构筑标签 |
| `pool_weight_base` | 基础权重 |
| `is_unique_in_run` | 本局是否唯一；首版固定 `true` |
| `ui_order` | UI 展示顺序 |

### 4.4 `mark_pool_rule`
用途：定义“烙印节点的统一品质概率和保底规则”。

| 字段 | 含义 |
| --- | --- |
| `pool_rule_id` | 规则 ID |
| `node_level_range` | 生效等级范围；首版可写 `10-40` |
| `choice_count` | 候选数量；首版固定 `3` |
| `common_weight` | 普通品质权重；首版 `65` |
| `rare_weight` | 稀有品质权重；首版 `25` |
| `epic_weight` | 史诗品质权重；首版 `10` |
| `guarantee_high_quality` | 每轮至少 `1 个 rare / epic`；首版固定 `true` |
| `same_round_no_repeat` | 同轮是否禁同名重复；首版固定 `true` |
| `exclude_owned` | 是否排除本局已获得烙印；首版固定 `true` |
| `enabled` | 是否启用 |

关键约定：
- 四个烙印节点共用同一张 `mark_pool_rule`。
- 不采用“等级越高烙印越强”的固定阶梯。
- 每轮至少保底 `1 个 rare / epic`；若前两张都是 `common`，第 `3 张` 强制从 `rare / epic` 池中抽。

### 4.5 `mark_effect_group`
用途：定义“烙印被选中后，到底挂上哪些常驻效果或监听器”。

| 字段 | 含义 |
| --- | --- |
| `effect_group_id` | 效果组 ID |
| `seq` | 效果序号 |
| `effect_type` | 效果类型 |
| `effect_target` | 作用目标，如 `hero_attr`、`attack_skill_runtime`、`battle_event`、`enemy_state` |
| `effect_key` | 具体字段 |
| `effect_op` | `add`、`mul`、`set`、`unlock_listener` |
| `effect_value` | 数值 |
| `condition_tags` | 生效条件标签 |

示例：

### 4.7 `mark_runtime`
用途：保存当前局已经触发过哪些节点、拿过哪些烙印、以及它们的显示顺序。

建议结构：

```lua
mark_runtime = {
  selected_mark_ids = {},
  by_id = {},
  triggered_node_levels = {},
  resolved_node_levels = {},
  queued_round_ids = {},
}
```

补充说明：
- `selected_mark_ids`
  按获得顺序保存，用于 `吞噬界面 -> 烙印页签` 展示。
- `triggered_node_levels`
  表示该等级节点是否已经越过并生成轮次，避免重复触发。
- `resolved_node_levels`
  表示该节点是否已完成选择。
- `queued_round_ids`
  用于处理“中央正忙时，多个等级节点先后被触发”的排队情况。

### 4.8 `mark_round_runtime`
用途：保存单次烙印节点生成的候选轮次。

| 字段 | 含义 |
| --- | --- |
| `round_id` | 轮次唯一 ID |
| `node_id` | 来源节点 ID |
| `node_level` | 来源等级，如 `10` |
| `candidate_mark_ids` | 当前 3 张候选烙印 |
| `state` | `queued`、`pending`、`resolved` |
| `reward_entry_id` | 对应奖励队列条目 ID |
| `created_time` | 创建时间 |

关键约定：
- 烙印节点一旦触发，就立即生成候选并固化到 `mark_round_runtime` 中；不要等真正弹窗时再临时抽取。
- 这样可以正确处理“先触发 10 级节点、后触发 20 级节点，但中间中央面板一直被别的系统占用”的情况。
- 同一节点只能生成 `1 个 round_id`。

### 4.9 烙印轮次状态机
- `queued`
  轮次已经生成，但当前中央面板正忙，还未正式展示。
- `pending`
  烙印面板正在等待玩家做 `3 选 1`。
- `resolved`
  玩家已选定，效果已写入运行时。

流转规则：
- `trigger -> queued`
  当英雄跨过节点等级，且中央面板被其他系统占用。
- `trigger -> pending`
  当英雄跨过节点等级，且中央面板当前空闲。
- `queued -> pending`
  `reward_queue` 轮到该 `mark_choice` 条目。
- `pending -> resolved`
  玩家完成选择。

补充约定：
- `烙印选择` 不提供 `稍后选择`，因此 `pending` 阶段不应该再被用户主动隐藏。
- 若一段时间内连续升级，允许多个 `mark_round_runtime` 先后处于 `queued`，但 `pending` 同时最多只有 `1 个`。

### 4.10 与 `reward_queue` 的协作
- 当节点等级被触发时：
  如果中央面板空闲，则直接进入 `pending` 并弹出烙印面板。
  如果中央面板忙碌，则创建 `reward_kind = mark_choice` 的奖励条目，并把对应轮次标记为 `queued`。
- `mark_choice` 建议优先级高于 `treasure_choice`，因为烙印不提供稍后选择，且节点次数固定只有 `4` 次。
- 若当前已有 `pending` 的烙印面板，则后续节点只允许继续入队，不允许覆盖当前烙印面板。

## 5. UI 与逻辑映射

### 5.1 宝物
- 底部 `宝物` 按钮：
  默认打开背包 `宝物` 页签。
  若当前存在 `pending / await_replace` 的宝物轮次，则优先恢复该轮次。
- `宝物候选面板`：
  只读 `treasure_round_runtime`。
  `是否需要替换`
  `推荐替换目标`
  `将触发替换`
  这些都应由运行时字段驱动，而不是 UI 临时推导。
- 背包 `宝物` 页签：
  只读 `treasure_runtime.active_slots` 与 `discarded_treasure_ids`。

### 5.2 烙印
- 烙印没有常驻底部按钮入口。
- `烙印选择面板`：
  只读当前 `pending` 的 `mark_round_runtime`。
  面板一旦打开，玩家必须完成选择。
- `已吞噬` 按钮对应的 `吞噬界面 -> 烙印页签`：
  只读 `mark_runtime.selected_mark_ids` 与 `mark_runtime.by_id`。
  按获得顺序展示 `品质`、`烙印名称`、`来源等级节点`、`效果说明`。

### 5.3 中央决策面板占用顺序
- 下列系统共享同一中央面板层：
  `成长武器词缀`
  `G`
  `F`
  `宝物候选`
  `烙印选择`
- `reward_queue` 只负责决定“哪个高价值奖励能被实例化并弹出”，不直接保存 UI 展示状态。
- 实际展示时建议遵循：
  `已有未结算中央面板 > mark_choice > treasure_choice`

## 6. MVP 最小数据量
- `treasure_base`：`12` 条
- `treasure_effect_group`：约 `15` 到 `20` 条
- `treasure_pool_rule`：`1` 到 `3` 条
- `treasure_pick_weight_rule`：`4` 到 `6` 条
- `treasure_replace_rule`：`5` 到 `8` 条
- `mark_node_base`：`4` 条
- `mark_base`：`13` 条
- `mark_pool_rule`：`1` 条
- `mark_effect_group`：约 `16` 到 `22` 条

## 7. 推荐实现顺序
1. 先把 `treasure_runtime + treasure_round_runtime` 跑通，重点是 `待选恢复` 与 `满 3 件后的替换流程`。
2. 再把 `reward_queue -> treasure_choice -> 宝物候选面板` 串起来，对齐现有 `宝物挑战`。
3. 再补 `treasure_replace_rule` 和推荐替换提示，让满位时决策成本可控。
4. 然后实现 `mark_node_base` 的等级触发与 `mark_round_runtime` 的生成。
5. 最后把 `mark_choice -> reward_queue -> 烙印选择面板 -> 吞噬界面` 串起来。
