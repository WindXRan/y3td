# 攻击技能与羁绊配表与实现映射

> 基于《战斗系统》《局内成长系统》《UI 与信息表现》整理，用于把 `G 攻击技能强化` 与 `F 羁绊抽卡` 收口成可直接落表、落逻辑、落 UI 状态机的实现侧规范。

## 1. 目标与范围
- 本文档只关注两套局内构筑系统：
  `攻击技能强化系统（G）`
  `羁绊抽卡系统（F）`
- 本文档不再重复解释它们为什么存在，而是重点解决：
  `要拆哪些表`
  `运行时状态怎么组织`
  `候选轮次怎么保存`
  `替换 / 保底 / 权重修正怎么落地`

## 2. 通用实现原则

### 2.1 统一术语
- `skill` 指攻击技能本体，例如 `basic_attack`、`arcane_arrow`。
- `skill_card` 指通过 `G` 看到的候选卡，包含：
  `new_skill`
  `enhance`
- `bond` 指一整套羁绊规则，例如 `blessing`、`greed`。
- `bond_card` 指通过 `F` 抽到的单张羁绊卡。
- `round` 指一次完整候选轮次，例如：
  `G 打开后的一轮 3 选 1`
  `F 扣费后的一轮 3 选 1`

### 2.2 设计边界
- 配表定义静态规则。
- 运行时对象保存当前局状态。
- UI 只读运行时，不自己保存候选结果。
- 候选轮次必须可恢复，不能只靠“当前打开的面板”存在。

### 2.3 多系统并存原则
- `G` 与 `F` 共用中央决策面板资源，但各自维护独立轮次数据。
- 同一时刻允许：
  `同时存在 1 个未结算 G 轮次`
  `同时存在 1 个未结算 F 轮次`
- 但 UI 层一次只展示其中一个；未展示的轮次必须仍可恢复。

## 3. 攻击技能强化系统（G）

### 3.1 建议拆表
首版建议至少拆 5 张表：
- `attack_skill_base`
- `attack_skill_card`
- `attack_skill_pool_rule`
- `attack_skill_pick_weight_rule`
- `attack_skill_effect_group`

当前工程补充约定：
- 第二批通用攻击技能的设计蓝图已独立沉淀在 `maps/EntryMap/script/entry_objects/attack_skill_blueprints/second_batch_skills.lua`。
- 该蓝图当前属于 `设计与配表约束源`，不是 runtime 已直接读取的正式运行时表。
- 后续若要把第二批技能真正接入 `G` 系统，建议优先以该蓝图为上游数据源，生成或映射到 `attack_skill_base` 与 `attack_skill_card`，避免文档、蓝图、运行时三套口径继续分叉。

### 3.2 `attack_skill_base`
用途：定义“攻击技能本体是什么”。

| 字段 | 说明 |
| --- | --- |
| `skill_id` | 技能唯一 ID |
| `skill_name` | 技能名称 |
| `skill_type` | 技能类型，如 `normal`、`attack_spell` |
| `default_slot` | 默认技能位；普攻固定 `1` |
| `unlock_level_outgame` | 局外解锁等级 |
| `damage_type` | 伤害类型 |
| `element_type` | 元素类型 |
| `base_desc` | 基础说明 |
| `positioning_tags` | 定位标签 |
| `is_starter` | 是否初始自带 |

首版示例：

| skill_id | skill_name | skill_type | default_slot | unlock_level_outgame | damage_type | element_type | base_desc | positioning_tags | is_starter |
| --- | --- | --- | --- | --- | --- | --- | --- | --- | --- |
| `basic_attack` | `普攻` | `normal` | `1` | `1` | `physical` | `none` | `射出箭矢攻击目标。` | `starter|single_target` | `true` |
| `arcane_arrow` | `奥术箭` | `attack_spell` | `0` | `1` | `magic` | `energy` | `射出穿透型奥术箭。` | `pierce|burst` | `false` |

第二批技能接线补充：
- `basic_attack` 之外，局内总共应支持 `4` 个自由攻击技能位，而不是旧版的 `3` 个自由位。
- 第二批技能进入配表层后，应与首批技能共享同一张 `attack_skill_base`，不要额外再开“第二批专用技能本体表”。
- `attack_skill_base` 至少需要能承接以下第二批字段：
  `archetype`
  `legend_name`
  `damage_type`
  `base_damage_ratio`
  `base_cooldown`
  `base_range / radius / duration / pierce / bounce`

### 3.3 `attack_skill_card`
用途：定义“G 候选里的一张卡是什么”。

| 字段 | 说明 |
| --- | --- |
| `card_id` | 卡牌唯一 ID |
| `card_name` | 卡牌名称 |
| `card_kind` | `new_skill` 或 `enhance` |
| `quality` | 品质 |
| `target_skill_id` | 归属技能 ID |
| `category_tag` | `basic`、`form`、`route`、`advanced`、`ultimate` |
| `type_tags` | 类型标签 |
| `synergy_tags` | 联动标签 |
| `core_desc` | 核心说明 |
| `extra_desc` | 补充说明 |
| `effect_group_id` | 效果组 ID |
| `prereq_cards` | 前置卡列表 |
| `exclude_cards` | 排斥卡列表 |
| `max_stack` | 最大叠层 |
| `is_unique` | 是否唯一 |
| `outgame_unlock_level` | 局外解锁等级 |
| `pool_weight_base` | 基础权重 |
| `ui_new_label` | 是否显示 `NEW` |

建议：
- `effect_group_id` 比 `effect_1_type / effect_2_type` 更容易扩展复杂效果。
- 若当前实现阶段还偏轻量，可以先保留 `effect_1_type / effect_1_value`，但长期建议切到效果组。

第二批技能补充约定：
- 第二批技能卡牌固定采用 `普通 3 / 优秀 3 / 稀有 3 / 传奇 1`。
- `category_tag` 不再建议沿用旧的 `basic / form / route / advanced / ultimate` 五层语义。
- 建议直接切换为可与设计蓝图一一对应的字段：
  `damage`
  `frequency`
  `function`
  `range`
  `count`
  `form`
  `state`
  `elite`
  `trigger`
  `legendary`
- `传奇` 卡应固定 `is_unique = true`，并在运行时只允许获取一次。

### 3.4 `attack_skill_effect_group`
用途：定义“一张卡被选中后到底改哪些值”。

| 字段 | 说明 |
| --- | --- |
| `effect_group_id` | 效果组 ID |
| `seq` | 效果序号 |
| `effect_type` | 效果类型 |
| `effect_target` | 作用目标：`skill_runtime`、`hero_attr`、`global_runtime` |
| `effect_key` | 具体字段 |
| `effect_op` | `add`、`mul`、`set`、`unlock` |
| `effect_value` | 数值 |
| `condition_tags` | 生效条件标签 |

首版示例：

| effect_group_id | seq | effect_type | effect_target | effect_key | effect_op | effect_value | condition_tags |
| --- | --- | --- | --- | --- | --- | --- | --- |
| `eff_unlock_arcane_arrow` | `1` | `unlock_skill` | `skill_runtime` | `arcane_arrow` | `unlock` | `1` |  |
| `eff_multishot_plus_1` | `1` | `skill_bonus` | `skill_runtime` | `multi_shot_count` | `add` | `1` | `basic_attack` |

### 3.5 `attack_skill_pool_rule`
用途：定义“什么情况下允许哪类卡进入候选池”。

| 字段 | 说明 |
| --- | --- |
| `rule_id` | 规则 ID |
| `card_kind` | 作用卡种 |
| `min_equipped_skill_count` | 最少已装配技能数 |
| `max_equipped_skill_count` | 最多已装配技能数 |
| `wave_range` | 波次范围 |
| `guarantee_after_miss` | 连续未命中后保底次数 |
| `max_new_skill_per_round` | 单轮最多几张新技能卡 |
| `enabled` | 是否启用 |

首版重点约定：
- `new_skill` 单轮最多 `1` 张。
- 当前 `1 个固定基础位 + 4 个自由攻击技能位` 已满时，`new_skill` 不再入池。
- 连续 `3 次` 未出现新技能卡且技能位未满时，下一轮强制至少 `1 张 new_skill`。

### 3.6 `attack_skill_pick_weight_rule`
用途：定义“已有技能强化卡在不同技能间怎么分权重”。

| 字段 | 说明 |
| --- | --- |
| `rule_id` | 规则 ID |
| `trigger_type` | 触发条件类型 |
| `trigger_value` | 触发值 |
| `weight_mul` | 权重乘数 |
| `note` | 备注 |

建议覆盖的规则：
- 已经拿过该技能强化卡：`x1.35`
- 已经形成该路线前置：`x1.55`
- 最近一次通过 `G` 选择过该技能：`x1.25`
- 新解锁技能接下来 `2` 次：`x1.60`
- 前 `2 波` 的普攻额外加权：`x1.20`

## 4. 攻击技能运行时对象

### 4.1 `attack_skill_runtime`
用途：保存当前局已装配技能与技能内成长状态。

建议结构：

```lua
attack_skill_runtime = {
  slots = {
    [1] = 'basic_attack',
    [2] = nil,
    [3] = nil,
    [4] = nil,
    [5] = nil,
  },
  by_id = {
    basic_attack = {
      skill_id = 'basic_attack',
      level = 1,
      picked_card_ids = {},
      route_tags = {},
      runtime_values = {},
    },
  },
  upgrade_pick_counts = {},
  recent_pick_skill_id = nil,
  miss_new_skill_rounds = 0,
}
```

接线说明：
- 如果 runtime 仍沿用旧的 `4 槽位数组`，建议把 `slot[1]` 明确改为固定基础位 `basic_attack`，其余 `slot[2] ~ slot[5]` 作为自由位。
- 若暂时不想改槽位数组长度，也至少要在运行时语义上明确“普攻不占自由装配位”，否则抽卡与 UI 仍会继续按旧口径理解。

### 4.2 `g_round_runtime`
用途：保存当前未结算的 `G` 轮次。

| 字段 | 说明 |
| --- | --- |
| `round_id` | 当前轮次唯一 ID |
| `candidate_card_ids` | 3 张候选卡 ID |
| `state` | `pending`、`picked` |
| `cost_skill_point` | 本轮消耗技能点；首版固定 `1` |
| `created_time` | 创建时间 |
| `snapshot_equipped_skills` | 生成候选时的技能快照 |
| `source_wave_index` | 生成轮次时的波次 |

关键约定：
- `G` 按钮第一次打开时扣除 `1 技能点` 并生成轮次。
- 若当前已有 `pending` 轮次，再按 `G` 只恢复该轮次，不重复扣点。

### 4.3 G 轮次状态机
- `idle`
  当前没有待选 G 轮次。
- `pending`
  已生成候选，等待玩家选择。
- `resolved`
  玩家已经选中，等待收尾写入。

流转规则：
- `idle -> pending`
  玩家按 `G` 且技能点足够。
- `pending -> resolved`
  玩家选中 1 张卡。
- `resolved -> idle`
  效果组已写入技能运行时后完成。

## 5. 羁绊抽卡系统（F）

### 5.1 建议拆表
首版建议至少拆 5 张表：
- `bond_base`
- `bond_card`
- `bond_pool_rule`
- `bond_pick_weight_rule`
- `bond_replace_rule`

### 5.2 `bond_base`
用途：定义“一整套羁绊的触发规则与成套效果”。

| 字段 | 说明 |
| --- | --- |
| `bond_id` | 羁绊唯一 ID |
| `bond_name` | 羁绊名称 |
| `quality` | 羁绊品质 |
| `required_count` | 触发张数 |
| `bond_effect_desc` | 羁绊效果说明 |
| `bond_effect_group_id` | 羁绊效果组 |
| `theme_tags` | 主题标签 |
| `pool_weight_base` | 基础权重 |

### 5.3 `bond_card`
用途：定义“F 抽到的具体单卡”。

| 字段 | 说明 |
| --- | --- |
| `bond_card_id` | 羁绊卡唯一 ID |
| `bond_id` | 所属羁绊 ID |
| `card_name` | 单卡名称 |
| `quality` | 品质 |
| `base_effect_group_id` | 基础效果组 |
| `base_effect_desc` | 基础效果说明 |
| `type_tags` | 类型标签 |
| `pool_weight_base` | 单卡基础权重 |
| `is_unique_in_run` | 单局是否唯一 |
| `ui_order` | 展示顺序 |

### 5.4 `bond_pool_rule`
用途：定义“不同波次、不同局面下 F 池子怎么调权重”。

| 字段 | 说明 |
| --- | --- |
| `rule_id` | 规则 ID |
| `wave_range` | 生效波次 |
| `common_weight` | 普通权重 |
| `rare_weight` | 稀有权重 |
| `epic_weight` | 史诗权重 |
| `refresh_keep_wave_weight` | 刷新是否沿用当前波次权重 |
| `enabled` | 是否启用 |

首版建议：
- 品质权重按波次提升。
- 同一轮抽卡的手动刷新，不改变当前波次品质权重。

### 5.5 `bond_pick_weight_rule`
用途：定义“羁绊相关候选如何向当前路线靠拢”。

| 字段 | 说明 |
| --- | --- |
| `rule_id` | 规则 ID |
| `trigger_type` | 触发类型 |
| `trigger_value` | 触发值 |
| `weight_mul` | 权重乘数 |
| `note` | 备注 |

建议覆盖：
- 当前已拥有同羁绊卡：提高该羁绊权重
- 当前只差 `1 张` 即可激活：提高该羁绊权重
- 当前已经明显超占位但尚未激活的羁绊：降低其新卡权重

### 5.6 `bond_replace_rule`
用途：定义“满 7 张后，系统推荐替换谁”。

| 字段 | 说明 |
| --- | --- |
| `rule_id` | 规则 ID |
| `trigger_type` | 触发条件 |
| `priority_score` | 推荐替换分值 |
| `ui_hint` | UI 提示 |

推荐分值可参考：
- 不影响任何激活羁绊：高优先替换
- 只影响未成型羁绊进度：中优先替换
- 会让已激活羁绊失效：低优先替换

## 6. 羁绊运行时对象

### 6.1 `bond_runtime`
用途：保存当前局已持有、生效中、已吞噬的羁绊卡状态。

建议结构：

```lua
bond_runtime = {
  active_card_ids = {},
  swallowed_card_ids = {},
  bond_progress = {},
  active_bond_ids = {},
  draw_count = 0,
}
```

### 6.2 `f_round_runtime`
用途：保存当前未结算的 `F` 轮次。

| 字段 | 说明 |
| --- | --- |
| `round_id` | 当前轮次唯一 ID |
| `candidate_bond_card_ids` | 当前 3 张候选 |
| `state` | `pending`、`picked`、`await_replace` |
| `cost_wood` | 本轮消耗木材；首版固定 `100` |
| `free_refresh_left` | 免费刷新次数 |
| `paid_refresh_count` | 已付费刷新次数 |
| `current_refresh_cost` | 当前下一次刷新费用 |
| `need_replace` | 是否需要替换 |
| `created_time` | 创建时间 |

关键约定：
- 打开 `F` 且木材足够时立即扣 `100 木材`。
- 若当前已有 `pending / await_replace` 轮次，再按 `F` 只恢复该轮次，不重复扣费。
- `稍后选择` 不清空当前轮次。

### 6.3 F 轮次状态机
- `idle`
  当前没有待结算 F 轮次。
- `pending`
  已生成候选，等待先选 1 张。
- `await_replace`
  因当前已持有 `7 张生效卡`，需从现有卡中替换 1 张。
- `resolved`
  已完成选择和必要替换，等待收尾。

流转规则：
- `idle -> pending`
  玩家按 `F` 且木材足够。
- `pending -> await_replace`
  玩家选中新卡，且当前生效卡已满 `7 张`。
- `pending -> resolved`
  玩家选中新卡，且当前仍有空位。
- `await_replace -> resolved`
  玩家完成替换目标选择。
- `resolved -> idle`
  本轮奖励写入、进度重算完成。

## 7. G / F 与中央决策面板的协作规则

### 7.1 优先级原则
- 若当前已有 `pending G`，再次按 `G` 直接恢复该轮次。
- 若当前已有 `pending F`，再次按 `F` 直接恢复该轮次。
- `G` 与 `F` 不互相覆盖未结算轮次，只是在 UI 上切换当前展示焦点。

### 7.2 推荐展示优先级
- `当前玩家主动点击的系统`
- `若无主动点击，则优先恢复更早创建的未结算轮次`
- `高价值奖励队列` 只能在 `G / F` 都没有未处理中央决策时弹出

### 7.3 最小防丢失要求
- `G` 已扣技能点但未选卡，轮次必须持久存在。
- `F` 已扣 `100 木材` 但未选卡，轮次必须持久存在。
- 任何时候都不能因为面板被别的入口顶掉而丢失当前候选。

## 8. 最小配表示例集

### 8.1 G 系统最小数据
- `attack_skill_base`：`5` 条
- `attack_skill_card`：`30` 到 `40` 条
- `attack_skill_pool_rule`：`4` 到 `8` 条
- `attack_skill_pick_weight_rule`：`5` 到 `10` 条
- `attack_skill_effect_group`：按卡牌效果拆分

### 8.2 F 系统最小数据
- `bond_base`：`4` 到 `8` 条
- `bond_card`：`11` 到 `20` 条
- `bond_pool_rule`：按波次至少 `5` 条
- `bond_pick_weight_rule`：`4` 到 `8` 条
- `bond_replace_rule`：`3` 到 `5` 条

## 9. 推荐实现顺序
1. 先把 `G 轮次保存与恢复` 跑通。
2. 再把第二批技能蓝图映射到 `attack_skill_base / attack_skill_card` 跑通。
3. 然后把 `F 轮次保存与恢复` 跑通。
4. 再补 `满 7 张羁绊后的替换流程`。
5. 最后再做 `权重修正、保底、新技能概率、路线偏向` 这些优化层。
