# 羁绊系统

## 模块职责

说明 `F` 抽卡、羁绊卡收集、羁绊激活、动态效果刷新和奖励修正由谁实现。

## Source Of Truth

- 主实现：`maps/EntryMap/script/runtime_bonds.lua`
- 接入层：`maps/EntryMap/script/entry_runtime.lua`
- 设计参考：`maps/EntryMap/script/docs/design/初步设计/11-攻击技能与羁绊配表与实现映射.md`

## 关键状态与数据流

- 羁绊定义：`BOND_DEFS`
- 羁绊卡定义：`BOND_CARD_DEFS`
- 当前局羁绊状态：`STATE.bond_runtime`
- 接入入口：
  - `BondSystem.create_runtime()`
  - `BondSystem.try_draw()`
  - `BondSystem.apply_choice()`
  - `BondSystem.refresh_effects()`
  - `BondSystem.update_effects()`
  - `BondSystem.build_reward_with_bonus()`

## 当前已实现行为

- `F` 系统已经真实接入 `runtime_bonds.lua`
- 抽卡会生成候选，选择后写入卡槽、进度、激活效果
- 羁绊既有静态属性加成，也有动态 runtime bonus
- 击杀奖励、Boss/精英增伤、低血增伤、成长型击杀收益等都已接入

## 当前实现特点

- `entry_runtime.lua` 里仍保留一层 legacy 包装函数，但主逻辑不再在那里
- `runtime_bonds.lua` 已经覆盖抽卡、替换、效果刷新、奖励修正、动态触发
- 文档必须把羁绊写成“已实现系统”，不能再写成占位

## 部分实现与未实现

- 当前仍是文本驱动选择，没有正式中央面板 UI
- 没有拆成外部配表
- 没有与宝物、烙印、成长武器形成统一高价值决策队列

## Agent 修改建议与风险点

- 改羁绊时优先改 `runtime_bonds.lua`
- 若只改 `entry_runtime.lua` 中 legacy 包装而不改 `runtime_bonds.lua`，大概率会出现认知偏差
- 新增羁绊效果时必须同时考虑静态属性、动态效果、击杀/奖励回调三个落点
