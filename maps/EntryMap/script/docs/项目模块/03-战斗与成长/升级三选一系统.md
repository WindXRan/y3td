# 升级三选一系统

## 模块职责

说明 `G` 强化系统如何生成候选、消耗技能点并写回攻击技能运行时。

## Source Of Truth

- 强化候选与应用：`maps/EntryMap/script/entry_runtime_attack_upgrades.lua`
- 轮次协调接入：`maps/EntryMap/script/entry_runtime.lua`
- 设计参考：`maps/EntryMap/script/docs/design/初步设计/11-攻击技能与羁绊配表与实现映射.md`

## 关键状态与数据流

- 技能点：`STATE.skill_points`
- 当前升级轮次：`STATE.awaiting_upgrade`、`STATE.current_upgrade_choices`
- 升级统计：`STATE.attack_skill_state.upgrade_counts`
- 新技能投放统计：`STATE.attack_skill_state.unlock_offer_fail_streak`
- 强化池：`build_upgrade_pool()`、`pick_upgrade_choices()`
- 应用入口：`apply_upgrade(index)`

## 当前已实现行为

- 英雄升级后获得技能点
- 按 `G` 可打开一轮三选一；若已有待选轮次，会继续当前轮次
- 候选会在“新技能解锁”和“已有技能强化”之间做权重控制
- 当存在空技能位时，会根据当前已解锁技能数控制新技能投放概率，并在连续失败后保底给解锁项
- 选项生效后会写回攻击技能状态，并更新技能描述/数值

## 当前实现特点

- 当前 `G` 系统已经是可玩逻辑，不是纯文本占位
- 当前是单轮 pending 模型，但已经与 `F`、烙印、宝物做了轮次互斥
- 当前主要通过文本消息与 HUD 按钮交互，没有正式中央面板

## 未实现或占位项

- 没有拆成设计文档中的多张配表
- 没有中央决策面板 UI
- 没有刷新、锁定、保底展示等更完整的 roguelike 决策体验

## Agent 修改建议与风险点

- 先改候选池和应用逻辑，再谈 UI，避免表现先行
- 如果新增高价值轮次系统，要先定义与 `F`、宝物、烙印的排他关系
- 不要让 `G` 直接依赖 UI 状态，应该只依赖 `STATE`
