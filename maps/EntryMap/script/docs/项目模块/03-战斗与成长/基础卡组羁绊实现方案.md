
# 基础卡组羁绊实现方案

## 1. 目标

本方案用于落地“基础卡组”整套羁绊系统，覆盖以下大类：

- 成长
- 暴击
- 箭术
- 法术
- 经济
- 体术

本次要实现的不是“末级羁绊平铺集合”，而是**按 XMind 从左到右逐层推进的链式羁绊树**。

例如：

- 成长 -> 敏捷 -> 猎魔人 -> 弓神
- 成长 -> 力量 -> 野蛮人 -> 战神
- 成长 -> 智力 -> 秘法师 -> 法神
- 体术 -> 生命 -> 血誓 / 血魔
- 体术 -> 战术
- 体术 -> 固守 -> 陷阵

玩家获得羁绊时，必须沿着这类链路逐层解锁，不能直接跳到最右侧节点。

## 2. 核心决策

### 2.1 采用方案

采用“**每个羁绊节点一个独立技能对象，但 Lua 按机制类型复用模板**”的方案。

这里的“羁绊节点”不是只指最右侧节点，而是指链路上的每一级节点。

例如下列节点都算独立节点：

- `敏捷`
- `猎魔人`
- `弓神`

### 2.2 不采用的方案

- 不采用“只给最末级羁绊做技能对象”
  - 不符合 XMind 从左到右解锁规则
  - 中间节点无法被明确标识与追踪
- 不采用“每个羁绊节点一份独立 Lua”
  - 文件数量会迅速膨胀
  - 同类触发逻辑重复很多
  - 平衡调整和机制修正成本高
- 不采用“大类共用一个技能对象”
  - 无法满足每个羁绊名称都有独立 icon 的要求
  - 不利于物编侧识别、调试和 UI 展示

### 2.3 最终原则

- 每个羁绊节点都必须有独立的：
  - 节点 id
  - 技能对象
  - icon
  - 名称
  - 描述
- Lua 层按“触发机制”复用模板，不按“节点名称”一对一拆几十个实现文件
- 候选发放必须受“前置节点是否已解锁”约束

## 3. XMind 解锁模型

### 3.1 节点与链路

基础卡组本质上是一棵**有向解锁树**。

每个节点都具备：

- 自己的身份
- 自己的技能对象
- 自己的解锁前置
- 自己的后继节点

### 3.2 从左到右规则

解锁规则按 XMind 链路推进：

1. 先解锁左侧节点
2. 左侧节点解锁后，右侧相邻节点才进入候选池
3. 更右侧节点不能越级出现

例如：

- 若玩家还没有 `敏捷`
  - 就不能出现 `猎魔人`
- 若玩家只有 `敏捷`
  - 可以出现 `猎魔人`
  - 不能直接出现 `弓神`
- 若玩家已有 `敏捷` 与 `猎魔人`
  - 才允许出现 `弓神`

### 3.3 同层分叉

如果某节点右侧有多个分支，则这些分支应视为“当前可选后继节点”。

例如：

- `体术 -> 生命 -> 血誓 / 血魔`

则：

- 解锁 `生命` 之后
  - `血誓`
  - `血魔`

都可以进入候选池。

## 4. 物编设计

### 4.1 技能对象粒度

每个羁绊节点对应一个物编技能对象。

例如：

- `bond_growth_agility`
- `bond_growth_demon_hunter`
- `bond_growth_bow_god`
- `bond_body_life`
- `bond_body_blood_oath`
- `bond_body_blood_demon`

### 4.2 技能对象职责

技能对象主要承担这些职责：

- 在物编里提供稳定的资源入口
- 挂独立 icon
- 挂独立名称与描述
- 作为 UI、调试、日志的识别锚点

### 4.3 技能对象边界

物编技能对象尽量做“展示壳子”，不要堆复杂逻辑。

建议保留在物编层的内容：

- 图标
- 名称
- 描述
- 品质
- 被动技能类别
- 必要特效资源

建议放到 Lua 层的内容：

- 每秒成长
- 击杀叠层
- 暴击触发
- 技能释放触发
- 生命阈值切换
- 区域结算
- 冷却刷新

## 5. Lua 运行时设计

### 5.1 模块划分

建议新增：

- `maps/EntryMap/script/runtime/bond_skills.lua`
- `maps/EntryMap/script/runtime/bond_skill_templates/*.lua`

其中：

- `bond_skills.lua`
  - 统一注册节点
  - 统一激活与卸载
  - 统一派发事件
  - 统一按链路刷新候选池
- `bond_skill_templates/*.lua`
  - 负责不同触发机制模板的实际执行

### 5.2 模板类型

建议首批模板拆为：

1. `static_attr`
   - 常驻属性型

2. `per_second_growth`
   - 每秒成长型

3. `kill_stack`
   - 击杀叠层型

4. `basic_attack_modifier`
   - 普攻修正式

5. `critical_trigger`
   - 暴击触发型

6. `hp_threshold`
   - 血量阈值型

7. `skill_cast_trigger`
   - 技能释放触发型

8. `aura_zone`
   - 区域/光环型

9. `hybrid_special`
   - 少数复杂特例

### 5.3 模板与节点关系

每个羁绊节点在配置里声明自己的模板类型。

同一模板可以服务多个节点，但每个节点仍保留独立配置。

例如：

- `力量成长`、`敏捷成长`、`智力提升`
  - 可共用 `per_second_growth`
- `力量之源`、`敏捷之源`、`智力之源`
  - 可共用 `kill_stack`
- `致命`、`大炮`
  - 可共用 `critical_trigger`
- `广射`、`月刃`、`多重箭`
  - 可共用 `basic_attack_modifier`

## 6. 数据结构建议

### 6.1 羁绊节点定义

每个节点建议包含以下字段：

```lua
return {
  id = 'growth_agility',
  display_name = '敏捷',
  group_id = 'growth',
  line_id = 'growth_agility_line',
  tier = 1,
  parent_id = nil,
  next_ids = { 'growth_demon_hunter' },
  quality = 'rare',
  icon = '...',
  editor_skill_id = 1342xxxx,
  template = 'kill_stack',
  attr = {
    ['敏捷'] = 100,
  },
  runtime = {
    kill_agility = 0.1,
    per_second_agility = 0.2,
    physical_damage_bonus = 0.08,
  },
  desc = {
    single = '敏捷 +100。',
    advanced = '杀敌敏捷 +0.1，每秒敏捷 +0.2，物理伤害 +8%。',
  },
}
```

### 6.2 字段说明

- `group_id`
  - 所属大类，如 `growth` / `magic` / `body`
- `line_id`
  - 所属链路，如 `growth_agility_line`
- `tier`
  - 在链路中的层级，从左到右递增
- `parent_id`
  - 前置节点
- `next_ids`
  - 解锁后的后继节点列表

### 6.3 运行时状态

运行时建议记录：

- 已解锁节点集合
- 当前候选节点集合
- 每条链已推进到哪一级
- 每个节点是否激活
- 每个节点挂载了哪些运行时实例

## 7. 候选池与抽取规则

### 7.1 候选池来源

候选池不能再是“所有羁绊全局随机”，必须按链路推进。

候选池来源建议为：

1. 当前可作为起点的根节点
2. 已解锁节点的 `next_ids`

### 7.2 具体规则

例如：

- 初始候选可能包含：
  - `敏捷`
  - `力量`
  - `智力`
  - `法术`
  - `经济`
  - `体术`

若玩家选中 `敏捷`：

- 下一轮允许出现：
  - `猎魔人`

若玩家随后选中 `猎魔人`：

- 下一轮允许出现：
  - `弓神`

### 7.3 禁止跳级

必须显式禁止：

- 未拥有前置节点时出现后继节点
- 同链路更右节点越过中间节点直接出现

## 8. 激活流程

建议统一流程为：

1. 玩家获得某个羁绊节点
2. 系统校验其 `parent_id` 是否已满足
3. 将对应技能对象挂到英雄
4. 将节点注册到 `bond_skills`
5. `bond_skills` 按模板类型挂载到对应模板管理器
6. 刷新该节点的后继候选池

## 9. 卸载与刷新流程

必须支持：

1. 获得节点后立即生效
2. 节点被替换或吞噬后正确移除
3. 链路推进后正确刷新后继候选
4. 套装或链路判定变化后正确刷新效果

刷新时应当：

1. 移除旧实例监听与状态
2. 依据当前已解锁节点重建

## 10. 套装与链路判定

这里的“套装效果”不应再只理解为“末级羁绊集合效果”，而应改成“链路推进效果”。

建议按以下口径处理：

- 单节点效果
  - 节点本身带来的静态或运行时收益
- 链路推进效果
  - 某条线推进到第 2 层、第 3 层后追加的新效果
- 大类效果
  - 某大类内解锁一定数量节点后触发的附加效果

例如：

- `成长 -> 敏捷 -> 猎魔人 -> 弓神`
  - 每往右推进一步，都在原有效果基础上叠加

## 11. 基础卡组分组建议

### 11.1 成长

- 三条主链：
  - 敏捷 -> 猎魔人 -> 弓神
  - 力量 -> 野蛮人 -> 战神
  - 智力 -> 秘法师 -> 法神

### 11.2 暴击

- 致命 -> 大炮

### 11.3 箭术

- 广射 -> 月刃
- 速攻
- 射术 -> 多重箭

### 11.4 法术

- 魔法师
- 魔能
- 魔术
- 急速
- 元素师

法术大类下如果后续 XMind 还有更细右侧节点，应继续按相同链式规则推进。

### 11.5 经济

- 贪婪
- 挑战

### 11.6 体术

- 生命 -> 血誓 / 血魔
- 战术
- 固守 -> 陷阵

## 12. 为什么选这套方案

这套方案同时满足：

- 符合 XMind 从左到右逐层解锁规则
- 每个羁绊节点都有独立 icon
- 每个节点都能在物编里对应到独立技能对象
- Lua 不会碎成大量重复脚本
- 候选池、UI、日志、调试都能按节点维度识别
- 后续扩展新链路时，只需补节点与前后继关系

## 13. 后续实施顺序

建议按下面顺序落地：

1. 先整理基础卡组节点总表
   - 明确 `id / display_name / icon / editor_skill_id / parent_id / next_ids / template`
2. 建立链路索引与候选池规则
3. 先做最常见的 4 个模板
   - `static_attr`
   - `per_second_growth`
   - `kill_stack`
   - `basic_attack_modifier`
4. 先落地成长、暴击、箭术
5. 再补法术、经济、体术的特殊模板
6. 最后处理复杂 SSR 节点与混合特例

这样可以降低一次性改动范围，也便于边做边验证。
