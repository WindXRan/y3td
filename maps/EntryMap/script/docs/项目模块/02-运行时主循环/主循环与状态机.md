# 主循环与状态机

## 模块职责

说明当前 runtime 如何用 `STATE`、奖励队列和定时循环驱动整局游戏。

## Source Of Truth

- 状态容器与轮次协调：`maps/EntryMap/script/entry_runtime.lua`
- 战场循环：`maps/EntryMap/script/entry_runtime_battlefield.lua`
- 成长循环：`maps/EntryMap/script/entry_runtime_progression.lua`
- 攻击技能循环：`maps/EntryMap/script/entry_runtime_attack_skills.lua`
- HUD 刷新：`maps/EntryMap/script/entry_runtime_hud.lua`
- 配置源：`maps/EntryMap/script/entry_config.lua`

## 关键状态与数据流

- 战场状态：`STATE.hero`、`STATE.all_enemies`、`STATE.total_enemy_alive`、`STATE.enemy_info_map`
- 主线状态：`STATE.current_wave_index`、`STATE.started_wave_count`、`STATE.active_wave`
- 挑战状态：`STATE.active_challenges`、`STATE.challenge_charges`、`STATE.challenge_recover_elapsed`
- 成长状态：`STATE.hero_progress`、`STATE.skill_points`、`STATE.attack_skill_state`
- 羁绊状态：`STATE.bond_runtime`
- 烙印状态：`STATE.mark_runtime`
- 宝物状态：`STATE.treasure_runtime`
- 高价值奖励状态：`STATE.reward_queue`
- UI/调试状态：`STATE.runtime_hud`、`STATE.gm_ui`、`STATE.dev_commands_registered`

## 当前已实现行为

- `reset_state()` 会重建整局必要状态
- `start_runtime_loops()` 至少驱动以下周期逻辑：
  - `update_passive_resources(0.25)`
  - `update_wave(0.25)`
  - `update_challenges(0.25)`
  - `update_challenge_charges(0.25)`
  - `update_bond_effects(0.25)`
  - `update_attack_skills(0.25)`
  - `ensure_runtime_hud()` / `refresh_runtime_hud()`
  - 炮击类被动每秒结算一次
  - 调试模式下的 GM 面板刷新
- `finish_game()` 会终止主局状态并显示胜负结果

## 状态机重点

- 启动后约 1 秒进入第 1 波，不存在独立大厅态
- 每波由“主怪阶段 + Boss 登场后并行压力”构成
- 挑战和主线并行，不需要清场切态
- 当前波 Boss 被击败后立即切换到下一波；最终 Boss 被击败后直接胜利
- `G`、`F`、烙印、宝物不会同时展开，`ensure_round_choice_available()` 会先要求完成当前轮次
- 奖励队列会在当前待选轮次结束后自动弹出下一个宝物或烙印奖励

## 当前实现边界

- 已有统一的待选轮次互斥与奖励队列，但还没有正式中央决策面板 UI
- 宝物、烙印已经有独立 runtime context
- 没有局外成长/存档驱动的跨局状态机
- 当前仍是单英雄、单章节、5 波原型结构

## Agent 修改建议与风险点

- 想加新局内系统，优先先定义 `STATE` 的归属和生命周期
- 想加新的高价值奖励，先决定它是进入 `reward_queue` 还是与现有轮次共用互斥入口
- 不要让 UI 自己保存真状态，UI 只读 `STATE`
- 新增循环时优先归并进现有 loop，避免分散为大量孤立 timer
