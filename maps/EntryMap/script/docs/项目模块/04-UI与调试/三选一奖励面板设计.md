# 三选一奖励面板设计

## 文档职责

定义 `G 技能强化`、`F 羁绊抽卡`、`宝物候选` 三类决策的统一三选一面板方案，明确：

- UI 资产写在哪里
- Lua 表现层写在哪里
- 图片资源怎么绑定
- 刷新 / 暂时隐藏 / 恢复选择怎么走
- 与当前 runtime 状态机、奖励队列和旧 merge UI 的边界怎么划

## Source Of Truth

- 战斗主循环与轮次互斥：`maps/EntryMap/script/entry_runtime.lua`
- G 强化候选与应用：`maps/EntryMap/script/entry_runtime_attack_upgrades.lua`
- F 羁绊候选与应用：`maps/EntryMap/script/runtime_bonds.lua`
- 现有 HUD 主入口：`maps/EntryMap/script/entry_runtime_hud.lua`
- 当前模块化 HUD 实现：`maps/EntryMap/script/ui/runtime_hud.lua`
- UI 资源索引：`maps/EntryMap/script/ui/res.lua`
- 地图 UI 资产目录：`maps/EntryMap/ui`
- UI 适配规则：`maps/EntryMap/script/UI适配规则.md`

## 目标

把当前依赖文本消息、HUD 按钮和数字键的三选一交互，升级成一套正式的中央三卡面板，同时保留现有热键入口和 runtime 主链路。

首期目标：

- 接管 `G / F / 宝物` 的“展示候选 -> 点击/按键选择 -> 关闭/恢复”流程
- 保留原有候选生成、应用结果、奖励队列与互斥逻辑
- 保留底部 `暂时隐藏` / `刷新` 两个操作位
- 预留面板背景、卡底、卡框、图标、徽记、按钮状态图等美术接入位
- 后续汇总到项目 `docs`，作为 UI 开发的事实依据

非目标：

- 本期不做左侧构筑摘要
- 本期不做烙印 UI
- 本期不把羁绊满 7 张时改成手动替换 UI，继续使用当前“推荐自动吞噬”的真实逻辑
- 本期不沿用旧 merge 中杂乱且错误的 HUD decision 分支作为主方案

## 当前实现事实

截至 `2026-04-06`，首版已经按“纯 Lua 动态 UI”落地：

- 已新增 `maps/EntryMap/script/entry_runtime_choice_panel.lua`
- 已新增 `maps/EntryMap/script/ui/choice_panel.lua`
- 已新增 `maps/EntryMap/script/ui/choice_panel_layout.lua`
- 已由 `entry_runtime.lua` 统一给新面板喂 `G / F / 宝物` 的三选一模型
- 旧 `ui/runtime_hud.lua` 中的 decision 分支已退回隐藏状态，不再作为主展示链路

也就是说，本文件中的“静态 UI 资产层”在首版没有采用编辑器 `json` 面板，而是保留为后续如需回填编辑器资产时的可选方向。

## 最终界面结构

面板外观以用户确认的参考图为准，固定为：

- 中央横排 `3` 张纵向卡片
- 每张卡固定层级：
  - 顶部徽记：如 `N / R / E`
  - 标题
  - 进度文本：如 `(0/4)`，无进度时可为空
  - 主图标
  - 副标题
  - 多段正文描述
- 底部固定两个按钮：
  - 左：`暂时隐藏`
  - 右：`刷新`

不采用的结构：

- 左侧摘要栏
- 上标题 + 左摘要 + 中候选 + 下操作 的旧中央面板大结构
- 与不同系统一起变化的散列式按钮区

## 文件分层

### 1. UI 资产层

首版**实际实现**没有新增 `maps/EntryMap/ui/*.json`，而是直接在 Lua 里动态创建：

职责边界：

- 面板壳子、三张卡、底部按钮都由 `ui/choice_panel.lua` 在 `GameHUD` 下即时创建
- 不在这里定义业务规则
- 不复制 runtime 状态

后续如果需要回填编辑器资产，再把以下结构拆回静态资产：

- 全屏遮罩根节点
- 三卡舞台
- 底部两个操作按钮
- 单卡的徽记 / 标题 / 进度 / 图标 / 副标题 / 正文槽位

### 2. Lua 表现层

新增统一三选一 presenter：

- 新增 `maps/EntryMap/script/entry_runtime_choice_panel.lua`
  - 入口文件，风格对齐 `entry_runtime_hud.lua`
- 新增 `maps/EntryMap/script/ui/choice_panel.lua`
  - 真正实现面板创建、节点绑定、显隐控制、刷新显示
- 新增 `maps/EntryMap/script/ui/choice_panel_layout.lua`
  - 维护面板尺寸、三卡位置与底部按钮布局常量

职责边界：

- 负责打开、隐藏、恢复、刷新显示、点击绑定、数字键映射
- 只读取现有 runtime 状态，不拥有第二份业务状态
- 不负责决定“候选是怎么抽出来的”

### 3. 业务数据层

现有业务模块继续负责候选生成与应用：

- G 强化：`maps/EntryMap/script/entry_runtime_attack_upgrades.lua`
- F 羁绊：`maps/EntryMap/script/runtime_bonds.lua`
- 宝物：`maps/EntryMap/script/entry_runtime.lua`

表现层从这些模块或 `STATE` 读取当前待选轮次，把它们转成统一卡片数据，再交给三选一面板显示。

## 不采用旧 merge UI 的原则

当前 merge 中已经存在一条“HUD decision model”支路，位置在：

- `maps/EntryMap/script/ui/runtime_hud.lua:648`

这条支路当前存在以下问题：

- 直接混在 HUD 刷新里，职责过重
- 节点组织和卡面结构不符合本次确认的参考图
- 更新链路杂乱，容易和真实待选轮次状态脱节

本次三选一面板实现不建立在这条支路上，而是新增独立 choice panel 模块。后续实现时可以删除或下线无用的旧 decision UI 更新逻辑，但删除前必须保证：

- `G / F / 宝物` 的真实待选轮次已经全部由新 choice panel 接管
- HUD 只保留顶部/底部常驻信息与入口按钮，不再承担候选卡展示职责

## 面板状态与 runtime 边界

### 现有主状态

当前已存在的真实待选状态：

- G：
  - `STATE.awaiting_upgrade`
  - `STATE.current_upgrade_choices`
- F：
  - `STATE.bond_runtime.awaiting_choice`
  - `STATE.bond_runtime.current_choices`
- 宝物：
  - `STATE.treasure_runtime.awaiting_choice`
  - `STATE.treasure_runtime.current_choices`
  - `STATE.treasure_runtime.awaiting_replace`
  - `STATE.treasure_runtime.pending_replace_choice`

对应轮次互斥入口：

- `maps/EntryMap/script/entry_runtime.lua:1925`
- `maps/EntryMap/script/entry_runtime.lua:1955`

### 新增的最小 UI 状态

新增 UI 状态只允许保存“面板表现信息”，不复制候选本体：

- `STATE.choice_panel_hidden`
  - 当前待选轮次是否被临时隐藏
- `STATE.choice_panel_visible_kind`
  - 当前显示的是 `upgrade / bond / treasure / treasure_replace`

禁止新增的状态：

- 第二份 `current_choices`
- 第二份 `awaiting_xxx`
- 与真实候选断开的缓存轮次数据

原则：

- 面板每次打开时都根据当前真实 runtime 重新组装显示模型
- 真正的选择、刷新、关闭、恢复都以原有 `STATE` 和业务模块为准

## 统一显示模型

choice panel 只接受统一显示模型，不直接理解业务规则。

### 面板模型

```lua
{
  kind = 'upgrade' | 'bond' | 'treasure' | 'treasure_replace',
  title = '攻击技能强化' | '羁绊抽卡' | '宝物候选' | '替换宝物',
  hide_enabled = true,
  refresh = {
    visible = true,
    enabled = true,
    free_left = 0,
    wood_cost = 40,
    text = '刷新',
  },
  cards = {
    [1] = { ... },
    [2] = { ... },
    [3] = { ... },
  },
}
```

### 单卡模型

```lua
{
  index = 1,
  badge_text = 'N',
  quality = 'common' | 'rare' | 'epic',
  title_text = '成长',
  progress_text = '(0/4)',
  icon_res = 123456,
  subtitle_text = '力量提升',
  body_blocks = {
    { text = '每秒力量 +0.5', color = 'green' },
    { text = '激活成长套装效果：每秒木材 +0.3', color = 'mix' },
  },
  clickable = true,
}
```

说明：

- `progress_text` 可为空字符串，不为 `nil`
- `body_blocks` 按多文本块渲染，不强依赖富文本 API
- `quality` 只负责决定卡框、徽记与默认色板，不替代业务稀有度字段

## 正文渲染策略

虽然引擎常量中存在 `富文本` 组件类型，但当前项目 Lua 包装层没有现成、稳定且已验证的“通用富文本赋值”主链路。

因此首版正文采用“多文本块”方案：

- `ChoiceRewardCard` 内预留若干正文文本节点
- presenter 按 `body_blocks` 顺序逐块设置文本和颜色
- 混合强调文案拆成多行，而不是在本期依赖复杂富文本嵌套

这样可以：

- 先稳定落地 UI
- 保持与当前 `set_text` / `set_text_color` 体系一致
- 避免把富文本控件接入问题引入主线

若后续确认富文本控件能稳定赋值，再把正文从多文本块升级为富文本字符串。

## 图片绑定规则

### 1. 固定 UI 壳子图

固定不随候选变化的 UI 图片，一律走：

- `maps/EntryMap/script/ui/res.lua`

新增一个语义化资源分组，例如：

- `M.skin.images.choice_panel.panel_bg`
- `M.skin.images.choice_panel.card_bg`
- `M.skin.images.choice_panel.frame_common`
- `M.skin.images.choice_panel.frame_rare`
- `M.skin.images.choice_panel.frame_epic`
- `M.skin.images.choice_panel.badge_bg_common`
- `M.skin.images.choice_panel.badge_bg_rare`
- `M.skin.images.choice_panel.badge_bg_epic`
- `M.skin.images.choice_panel.action_button_bg`
- `M.skin.images.choice_panel.action_button_shadow`

原则：

- 逻辑文件不直接写散落图片 ID
- 换皮只改 `ui/res.lua`

### 2. 业务图标

跟具体候选绑定的图标资源，写入业务定义：

- G：在 `maps/EntryMap/script/entry_runtime_attack_upgrades.lua` 里的 upgrade 定义上补 `ui_icon`
- F：在 `maps/EntryMap/script/runtime/bond_nodes.lua` 上维护羁绊节点图标与展示文本
- 宝物：在 `maps/EntryMap/script/entry_objects/treasures/*.lua` 上补 `ui_icon`

首版不强制补大插画，只要求每个候选有一张可识别主图标。

### 3. 徽记和稀有度

徽记文本不从图片里写死，而由数据驱动：

- G：可用 `N` 表示新技能，`R` 表示强化，后续可继续细分
- F：按羁绊品质或卡品质显示 `N / R / E`
- 宝物：按宝物品质显示 `N / R / E`

稀有度只决定：

- 卡框图片
- 徽记底板图片
- 默认标题 / 副标题 / 高亮色板

## 交互规则

### 1. 打开

- `G` 继续走原热键和原入口按钮
- `F` 继续走原热键和原入口按钮
- 宝物继续走原 reward queue 和挑战链路
- 如果某个系统已经存在未结算待选轮次，再次打开时直接恢复当前轮次，不重新抽取

### 2. 选择

- 点击第 `1 / 2 / 3` 张卡，与按 `1 / 2 / 3` 完全等价
- 最终仍走现有应用函数：
  - G -> `apply_upgrade(index)`
  - F -> `apply_bond_choice(index)`
  - 宝物 -> `apply_treasure_choice(index)`

### 3. 暂时隐藏

- `暂时隐藏` 只隐藏面板，不改变真实轮次状态
- 不退还任何资源
- 不清空当前候选
- 不重置免费刷新次数
- 不重置付费刷新费用阶梯
- 再按原入口键或点原 HUD 按钮时，恢复同一轮候选

### 4. 刷新

刷新规则统一为“免费次数 + 木材计费阶梯”：

- 付费阶梯：`40 / 80 / 100 / 100...`
- 也就是第 `1` 次付费刷新 `40` 木材
- 第 `2` 次付费刷新 `80` 木材
- 第 `3` 次及以后固定 `100` 木材

#### G 刷新规则

- 每轮 `3` 次免费刷新
- 免费次数用完后进入付费阶梯
- 刷新只重抽当前 3 张候选，不重新消耗技能点
- 选择完成或本轮结束后，刷新次数和阶梯重置

#### F 刷新规则

- 每轮 `0` 次免费刷新
- 打开面板后可直接进入付费阶梯
- 抽卡本体 `100 木材` 成本不退
- 刷新只重抽当前 3 张候选，不再次收取 `100 木材`
- 选择完成后，刷新次数和阶梯重置

#### 宝物刷新规则

- 每轮 `3` 次免费刷新
- 免费次数用完后进入付费阶梯
- 刷新只在“待选宝物”第一阶段可用
- 一旦进入“已选中宝物、等待替换宝物位”的第二阶段，刷新按钮灰置
- 选择完成后，刷新次数和阶梯重置

## 宝物替换阶段

宝物当前已具备两阶段状态机：

- `pending`
- `await_replace`

本期继续沿用这套状态：

- 第一阶段显示 `3 个新宝物候选`
- 第二阶段仍复用同一三卡布局，但 `3 张卡` 改为当前已持有的 `3 个宝物位`
- 标题改为 `替换宝物`
- `刷新` 按钮灰置
- 点击 `1 / 2 / 3` 表示指定被替换的宝物位

## 羁绊满 7 张的处理

当前羁绊系统已经有自动推荐吞噬逻辑：

- `maps/EntryMap/script/runtime_bonds.lua:664`

首期不改成手动替换 UI，原因：

- 当前真实逻辑已经闭环
- 手动替换会显著扩大本期 UI 范围
- 本期优先把 `G / F / 宝物` 三选一面板稳定落地

因此当羁绊位满 `7` 张时：

- 仍展示正常 3 张候选卡
- 卡面正文和提示区只负责告知“本次选择会按推荐自动吞噬 1 张旧卡”
- 真正吞噬规则继续使用现有 runtime 逻辑

## 与 HUD 的关系

HUD 继续负责：

- 顶部状态信息
- 底部入口按钮
- 当前资源、波次、挑战、待领奖励数

choice panel 负责：

- 高价值三选一的卡面展示与交互

关系约束：

- HUD 不再负责展示候选卡正文
- HUD 上的入口按钮在存在待选轮次时，只负责恢复当前 choice panel
- choice panel 不负责更新常驻资源栏，资源栏继续由 HUD 刷新

## UI 适配规则

按 `maps/EntryMap/script/UI适配规则.md` 执行：

- 根节点挂在 `GameHUD`
- 容器优先使用图片控件 + 空图 `999`
- 面板舞台与按钮位置优先使用百分比定位
- 卡片内部子节点可使用固定设计像素布局

首期面板要求：

- 1920x1080 设计分辨率下三张卡稳定居中
- 不因卡面正文长度改变三列结构
- 刷新与隐藏按钮位置固定，不随系统变化漂移
