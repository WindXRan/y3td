# 代码调用总览

## 模块职责

本文只回答一件事：当前一局游戏从哪里启动、经过哪些模块、由谁维护核心状态。

## Source Of Truth

- 启动入口：`maps/EntryMap/script/main.lua`
- 主运行时协调器：`maps/EntryMap/script/entry_runtime.lua`
- 配置输入：`maps/EntryMap/script/entry_config.lua`
- 静态对象输入：`maps/EntryMap/script/entry_objects`
- 羁绊子系统：`maps/EntryMap/script/runtime_bonds.lua`
- 运行时子系统：`maps/EntryMap/script/entry_runtime_*.lua`

## 调用总链路

```text
游戏初始化
  -> maps/EntryMap/script/main.lua
  -> load_runtime()
  -> require 'entry_runtime'
  -> 加载 entry_config / entry_objects / runtime_bonds / entry_runtime_*.lua
  -> entry_runtime.bootstrap()
  -> validate_config()
  -> reset_session_state()
  -> register_runtime_events()
  -> register_dev_commands()
  -> start_runtime_loops()
  -> ensure_gm_panel()
  -> outgame_system.load_profile()
  -> outgame_system.enter_outgame()

局外点击开始
  -> outgame_system.start_selected_stage()
  -> reset_battle_state()
  -> create_hero()
  -> initialize_hero_progression()
  -> setup_basic_attack_ability()
  -> ensure_runtime_hud()
  -> ensure_gm_panel()
  -> start_wave(1)

单局结束
  -> finish_game()
  -> battle_result
  -> cleanup_battle_units()
  -> outgame_system.apply_battle_result()
  -> outgame_system.enter_outgame(result)
```

## 关键状态与数据流

- 全局运行时状态在 `entry_runtime.lua` 的 `STATE`
- 静态玩法配置在 `entry_config.lua` 的 `CONFIG`
- 波次、挑战、攻击技能、宝物、烙印、羁绊静态对象主要来自 `entry_objects/*`
- 战场/成长/攻击技能/HUD/调试分别下放到 `entry_runtime_*.lua`
- 羁绊静态定义与抽卡/效果计算在 `runtime_bonds.lua`
- UI 资产 ID 映射在 `ui_res.lua`

## 当前已实现行为

- 入口层只负责加载与兜底，不承载玩法规则
- `entry_runtime.lua` 负责总协调，不再独占所有实现细节
- `entry_runtime_outgame.lua` 负责局外 UI、章节/模式选择、解锁和局外存档骨架
- 主线推进、挑战、升级、胜负结算分别委托给 `entry_runtime_battlefield.lua`、`entry_runtime_progression.lua`、`entry_runtime_attack_upgrades.lua` 等子模块
- `runtime_bonds.lua` 已被真实调用，负责 `F` 抽卡与羁绊效果刷新
- 宝物、烙印与奖励队列在 `entry_runtime.lua` 内完成轮次协调，并与 `G/F` 互斥
- 运行时 HUD 由 `entry_runtime_hud.lua` 持续刷新，并随局外/战斗阶段显隐

## 需要特别排除的错误认知

- `maps/EntryMap/script/可重载的代码.lua` 不是主运行时
- 根目录 `global_script/global_main.lua` 不是当前游戏入口
- 根目录 `global_trigger` 没有参与当前玩法逻辑
- `maps/EntryMap/script/docs/design` 是策划设计，不是运行时代码

## Agent 修改建议与风险点

- 改玩法时先从 `entry_runtime.lua` 找到对应子模块，再进入实现，不要只盯着总文件
- 改数值、波次、挑战规则时优先查 `entry_config.lua` 与 `entry_objects/*`
- 改羁绊时同时看 `runtime_bonds.lua` 与 `entry_runtime.lua` 的接入环境
- 改宝物、烙印、奖励轮次时优先看 `entry_runtime.lua` 中的队列与互斥逻辑
