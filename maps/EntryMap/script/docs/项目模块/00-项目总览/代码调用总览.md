# 代码调用总览

## 模块职责

本文只回答一件事：当前一局游戏从哪里启动、经过哪些模块、由谁维护核心状态。

## Source Of Truth

- 启动入口：`maps/EntryMap/script/main.lua`
- 主运行时：`maps/EntryMap/script/entry_runtime.lua`
- 配置输入：`maps/EntryMap/script/entry_config.lua`
- 羁绊子系统：`maps/EntryMap/script/runtime_bonds.lua`

## 调用总链路

```text
游戏初始化
  -> maps/EntryMap/script/main.lua
  -> load_runtime()
  -> require 'entry_runtime'
  -> entry_runtime.bootstrap()
  -> validate_config()
  -> reset_state()
  -> create_hero()
  -> register_runtime_events()
  -> register_dev_commands()
  -> start_runtime_loops()
  -> ensure_gm_panel()
  -> start_wave(1)
```

## 关键状态与数据流

- 全局运行时状态在 `entry_runtime.lua` 的 `STATE`
- 静态玩法配置在 `entry_config.lua` 的 `CONFIG`
- 羁绊静态定义与抽卡/效果计算在 `runtime_bonds.lua`
- UI 资产 ID 映射在 `ui_res.lua`

## 当前已实现行为

- 入口层只负责加载与兜底，不承载玩法规则
- 主线推进、挑战、升级、胜负结算都在 `entry_runtime.lua`
- `runtime_bonds.lua` 已被真实调用，负责 `F` 抽卡与羁绊效果刷新

## 需要特别排除的错误认知

- `maps/EntryMap/script/可重载的代码.lua` 不是主运行时
- 根目录 `global_script/global_main.lua` 不是当前游戏入口
- 根目录 `global_trigger` 没有参与当前玩法逻辑
- `maps/EntryMap/script/docs/design` 是策划设计，不是运行时代码

## Agent 修改建议与风险点

- 改玩法时优先查 `entry_runtime.lua`，不要先去触发器目录找逻辑
- 改数值、波次、挑战规则时优先查 `entry_config.lua`
- 改羁绊时同时看 `entry_runtime.lua` 与 `runtime_bonds.lua`，后者是主实现，前者有兼容包装
