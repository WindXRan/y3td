# 代码审查清单

## 一、变量命名规范（必查）

### 1.1 禁止使用混淆变量名
- [ ] 变量名不得为单个字符（如 `a`, `b`, `I`, `K`）
- [ ] 变量名不得为大小写混合的无意义组合（如 `aJ`, `bL`, `cE`, `dQ`）
- [ ] 函数参数必须有明确含义的名称（如 `unit`, `show_weapon`, `camera_config`）

### 1.2 变量命名要求
- [ ] 使用 camelCase 命名风格
- [ ] 变量名应反映其用途（如 `slot_index` 而非 `bL`）
- [ ] 避免缩写，除非是广泛理解的缩写（如 `UI`, `API`, `ID`）

---

## 二、代码可读性（必查）

### 2.1 注释要求
- [ ] 每个模块文件顶部有注释说明其职责
- [ ] 复杂逻辑有注释解释
- [ ] 公共 API 函数有注释说明参数和返回值
- [ ] 魔法数字应提取为常量或有注释说明

### 2.2 代码格式
- [ ] 缩进统一（建议使用空格）
- [ ] 空格使用一致
- [ ] 行长度不超过 120 字符
- [ ] 函数定义清晰，职责单一

---

## 三、错误处理（必查）

### 3.1 空值检查
- [ ] 访问对象属性前检查对象是否为 nil
- [ ] 函数参数有必要的有效性检查
- [ ] 使用 `pcall` 包装可能失败的操作

### 3.2 错误处理
- [ ] 有适当的错误日志输出
- [ ] 错误情况下有合理的返回值
- [ ] 不会因未定义变量导致运行时崩溃

---

## 四、代码质量（必查）

### 4.1 代码重复
- [ ] 无明显的代码重复
- [ ] 重复逻辑已提取为函数
- [ ] 配置与代码分离

### 4.2 性能考虑
- [ ] 避免不必要的计算
- [ ] 合理使用缓存机制
- [ ] 避免在循环中进行昂贵操作

---

## 五、架构一致性（必查）

### 5.1 模块设计
- [ ] 符合单一职责原则
- [ ] 模块间依赖清晰
- [ ] 使用 `Module.create(env)` 模式接收依赖

### 5.2 状态管理
- [ ] 使用 STATE 子表按系统边界组织状态
- [ ] 避免全局变量滥用
- [ ] 状态变更有清晰的来源

---

## 六、测试要求（建议）

### 6.1 测试覆盖
- [ ] 核心逻辑有单元测试
- [ ] 新增功能有相应测试用例
- [ ] 测试能覆盖边界情况

---

## 七、提交前检查清单

- [ ] 运行项目测试，确保无失败
- [ ] 使用 grep 检查是否有遗漏的混淆变量
- [ ] 检查是否有未定义变量引用
- [ ] 代码已通过 lint 检查（如适用）
- [ ] 提交信息清晰描述改动内容

## 混淆变量检测命令

在 PowerShell 中运行以下命令检测混淆变量：

```powershell
# 检测单个字符变量和大小写混合变量
Get-ChildItem -Path "script\" -Recurse -Filter "*.lua" | ForEach-Object {
    Select-String -Path $_.FullName -Pattern '\b([a-z][A-Z]|[A-Z][a-z])\b' | ForEach-Object {
        Write-Host "$($_.Filename):$($_.LineNumber): $($_.Line)"
    }
}
```

---

## 常用检查模式

| 模式 | 说明 | 示例 |
|------|------|------|
| `\b[a-z][A-Z]\b` | 小写开头后跟大写的双字符变量 | `aJ`, `bL`, `cE` |
| `\b[A-Z][a-z]\b` | 大写开头后跟小写的双字符变量 | `Ab`, `Bc`, `Cd` |
| `^local\s+[a-zA-Z]$` | 单个字符的局部变量 | `local a`, `local I` |
