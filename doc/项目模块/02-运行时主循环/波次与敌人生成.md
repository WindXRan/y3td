# 波次与敌人生成

## 1. 配置来源

波次系统的主要数据来自 `entry_config.lua`：

- `waves`
- `points`
- `areas`
- `total_enemy_soft_cap`
- `unit_ids`

其中：

- `points` 定义英雄出生点、防线点
- `areas` 定义主怪区、Boss 区、挑战区
- `waves` 定义每波时间轴、主怪单位、Boss 单位、奖励与主题

## 2. 波次的基本结构

每个 `wave` 配置至少包含：

- `id`、`index`、`name`
- `main_unit_id`
- `boss_unit_id`
- `spawn_area_id`
- `boss_spawn_area_id`
- `boss_spawn_sec`
- `batch_min`、`batch_max`
- `max_alive`
- `spawn_segments`
- `post_boss_interval_sec`
- 主怪/Boss 奖励

这说明当前波次不是固定“刷 N 只就结束”，而是按时间片持续出怪，并在指定时间点插入 Boss。

## 3. 启动一波时发生什么

`M.start_wave(index)` 会：

- 读取对应 `CONFIG.waves[index]`
- 更新 `STATE.current_wave_index`
- 更新 `STATE.started_wave_count`
- 创建 `STATE.active_wave`

`active_wave` 中保存：

- 当前波次配置
- 已流逝时间
- 是否仍在进行
- Boss 是否已生成
- 当前波内存活数
- 下一次刷怪时间点

这就是单波的运行态控制器。

## 4. 主怪生成机制

0.25 秒循环会调用 `update_wave(dt)`。

在这个函数里：

1. 累加 `runner.elapsed`
2. 根据 `runner.next_spawn_sec` 判断是否到达下一个刷怪时点
3. 调用 `can_spawn_main_batch(runner)` 检查能否刷怪
4. 调用 `spawn_main_batch(runner)` 生成一批主怪
5. 按波次不同阶段重新计算下一次刷怪间隔

影响主怪生成的关键因素包括：

- 当前波配置的 `spawn_segments`
- 当前波允许的 `max_alive`
- 全局软上限 `total_enemy_soft_cap`

## 5. Boss 生成机制

当 `runner.elapsed >= runner.wave.boss_spawn_sec` 且还未生成 Boss 时，会执行 `spawn_boss(runner)`。

Boss 与主怪的区别在于：

- 有独立出生区域
- 有单独奖励
- 会被标记为 `is_boss = true`
- 会影响波次推进与最终胜负

当前波推进并不是“主怪刷完立刻结束”，而是要结合 Boss 状态一起判断。

## 6. 敌人生成后如何进入战场

无论是主怪还是挑战敌人，底层都统一走 `spawn_enemy()`：

- 按区域随机生成点
- 创建单位
- 写入敌人运行时信息
- 加入 `STATE.all_enemies`
- 让单位 `attack_move(STATE.defense_point)`

因此地图上的敌人并不是只靠编辑器摆放，而是由代码按区域动态生成并向防线移动。

## 7. 挑战敌人与主线敌人的关系

挑战也复用了同一套敌人生成能力，但会附带不同元信息：

- `kind = 'challenge'`
- `owner = challenge instance`
- 可额外标记 `is_elite` 或 `is_boss`

这意味着主线与挑战并行时：

- 使用统一的敌人集合
- 使用统一的死亡结算入口
- 但可以通过 `enemy_info_map` 区分来源和结算方式

## 8. 当前波次系统的特点

当前设计明显偏向塔防玩法的“持续压力型出怪”：

- 主怪按时间段不断刷新
- Boss 在固定秒数后加入
- 主线和挑战允许并行
- 全局软上限控制场面密度

这比“按波次一次性刷完再等清场”的脚本结构更接近实时压力推进型塔防。
