# 编辑器导出资产说明

## 1. 这类内容是什么

当前仓库里有一大类内容，并不是手写业务逻辑，而是 Y3 编辑器导出的资产与元数据。它们的作用是：

- 保存编辑器里的对象结构
- 记录资源树、分组树、文件夹信息
- 给可视化 UI、物编对象、触发器提供底层组织方式

## 2. 主要目录

### `maps/EntryMap/editor`

保存编辑器元信息，例如：

- 文件夹树
- UI 树结构
- 资源分组信息
- 编辑器内部组织数据

这是“编辑器如何看这张地图”的数据层。

### `maps/EntryMap/editor_table`

保存对象编辑器导出的综合表数据，例如：

- `abilityall`
- `editorunit`
- `editoritem`
- `modifierall`
- `projectileall`
- `soundall`
- `technologyall`

它更像是“对象表资产总汇”。

### `maps/EntryMap/custom_eca`

自定义 ECA 资产扩展位。当前：

- `custom_eca.json` 基本为空
- `custom_python.py` 为空文件

所以这里目前更像保留位。

### `maps/EntryMap/global_trigger`

地图级可视化触发器资产。虽然它也会参与编辑器导出体系，但应单独看待，因为里面已经有成体系的 UI 逻辑资产。

## 3. 为什么它们不是“业务模块”

这些目录通常有三个特点：

- 文件非常多
- 文件名多为 ID 或编辑器生成名
- 更多是在表达“数据与结构”，而不是“业务流程”

因此在阅读时，不应该把它们当成和 `entry_runtime.lua` 同层级的业务代码模块。

## 4. 这些资产和业务代码的关系

可以简单理解成：

- 业务代码决定“何时做什么”
- 编辑器导出资产决定“可用什么对象、资源、界面、触发器结构”

例如：

- Lua 说“创建某个单位 ID”
- 编辑器资产负责提供该单位定义

或者：

- Lua 说“挂到 GameHUD 上”
- UI 资产负责提供 HUD 面板本体

## 5. 当前项目中的阅读建议

如果你的目标是理解玩法与调用关系，优先顺序仍应是：

1. `maps/EntryMap/script`
2. `maps/EntryMap/ui`
3. `maps/EntryMap/global_trigger`
4. `maps/EntryMap/editor` 与 `editor_table`

因为后两者更偏“资产底座”，前两者才是最接近当前实际运行逻辑的部分。

## 6. 本项目中的结论

当前可以把编辑器导出资产总结为：

- 重要，但不是第一阅读入口
- 会支撑运行时，但不直接描述整局流程
- 适合在理解主线脚本后，再回头做资源与结构盘点

这也是本次文档将它们单独放在“触发器与编辑器资产”章节的原因。
