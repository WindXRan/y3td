# 链式羁绊内容重做方案

## 1. 背景

当前仓库里的链式羁绊机制本身已经成型，主要承载文件如下：

- `maps/EntryMap/script/runtime/bonds_chain.lua`
- `maps/EntryMap/script/runtime/bond_nodes.lua`
- `maps/EntryMap/script/runtime/choice_panel_model.lua`
- `maps/EntryMap/script/runtime/overview_model.lua`

这套机制已经具备以下能力：

- `F` 抽取链式羁绊三选一
- 按前置节点逐步解锁后继节点
- 右下角展示链路进度
- 卡面展示当前效果 / 进阶方向 / 后继节点
- 编辑器中每个节点有独立技能对象、图标和描述

问题不在机制，而在内容：

- 当前 `bond_nodes.lua` 里的节点名称、说明和多数效果仍然带着旧版本语义
- `羁绊图片.md` 已经被定为当前唯一有效的羁绊设计主文档，图片外补出的 OCR 与文字恢复内容不再作为依据
- `Pasted image 20260408215653.png` 对应的 xmind 图明确给出了新的主链拓扑

因此本轮改造的结论是：

- 保留现有“链路玩法机制”
- 重做“链路内容本身”
- 内容来源以 `羁绊图片.md` 和 xmind 图为准；图片外的额外文字内容如果比图片更多，默认都算错
- 技术承载继续使用 `bonds_chain.lua + bond_nodes.lua`

## 2. 目标与非目标

### 2.1 目标

本轮重做要达成以下目标：

- 将当前链式羁绊重排为 6 条主链：`成长 / 暴击 / 箭术 / 法术 / 经济 / 体术`
- 保留 29 个节点规模，不推翻现有抽卡、解锁、进度、面板机制
- 以 xmind 的上下级关系替换当前链路结构
- 以图片表里的卡牌内容重写节点名称、描述、效果方向
- 尽量复用现有节点 id、`editor_skill_id`、图标资源与 UI 桥接

### 2.2 非目标

本轮不做以下事项：

- 不回退成“平铺套装计数”的旧羁绊玩法
- 不重做 `F` 抽卡 UI 结构
- 不重做链路进度 UI 结构
- 不全面推翻 `bonds_chain.lua` 的抽卡和解锁算法
- 不在第一阶段追求把图片表中的所有特殊机制 1:1 完整复刻

## 3. 内容来源与解释口径

本方案只保留两类有效来源，主次已经固定：

- `羁绊图片.md`
  - 当前羁绊设计主文档，提供单件卡名称、核心属性、特殊效果、套装效果与 UI 参考
- `Pasted image 20260408215653.png`
  - 提供最终采用的链路拓扑

本轮采用以下解释口径：

- 图片表中的“平铺单件卡”不是直接恢复成原始平铺系统
- 图片表中的内容会被压缩进当前链式节点系统
- xmind 图只决定“谁是谁的前置、谁是谁的后继”
- 图片表决定“每个节点叫什么、代表什么、效果偏向什么”
- 任何不在图片里直接出现、却在其他文字稿里额外补出来的羁绊内容，默认不采纳

## 4. 结构决策

### 4.1 采用方案

采用“结构保留、内容重做版”：

- 保留现有 29 节点骨架
- 保留现有链式抽取和逐步解锁逻辑
- 用新羁绊名称和新效果语义替换旧节点内容

### 4.2 压缩规则

图片表中的平铺卡牌数量多于当前链式节点数，因此必须压缩。压缩规则如下：

1. 根节点负责承接该主链的“基础属性 / 入门效果”
2. 中段节点负责承接“专精机制开始成形”
3. 终段节点负责承接“该线的终局身份或强特化”
4. 图片表中的“卡池加入某类基础牌”统一翻译为：
   - 解锁该根节点后，该链的 `next_ids` 进入候选池
5. 图片表中的“套装效果”统一翻译为：
   - 链路推进效果，分摊到中段和终段节点

### 4.3 稳定性决策

为避免联动系统一起失稳，本轮默认保留以下稳定锚点：

- 运行时节点 id 尽量不改
- `editor_skill_id` 不改
- 现有节点图标优先复用，必要时后续再换
- 现有 `route_tags` 只做增补，不做大面积删除

## 7. 平铺套装到链路节点的翻译规则

### 7.1 根节点

根节点统一承接这类内容：

- 单件核心属性
- 最容易理解的立即收益
- 该主链的入口身份

例如：

- `敏捷之源 -> 敏捷`
- `力量之源 -> 力量`
- `智力之源 -> 智力`
- `回春 / 生命祝福 -> 生命`

### 7.2 中段节点

中段节点统一承接这类内容：

- 套装效果开始生效
- 该线职业化身份明确化
- 玩家能直接感知到“方向不同了”

例如：

- `猎魔人` 开始偏向高攻速 / 暴击 / 物理成长
- `野蛮人` 开始偏向击杀滚雪球
- `秘法师` 开始偏向技能成长

### 7.3 终段节点

终段节点统一承接这类内容：

- 终局专精
- 该线最有辨识度的玩法表达
- 若图片表中存在 SSR 组，则优先映射到终段节点

例如：

- `弓神` 承接弓系终局远程专精
- `战神` 承接力量终局爆发专精
- `法神` 承接智力终局法术专精

## 8. 与现有运行时的兼容策略

### 8.1 运行时 id 保持稳定

第一阶段默认不改以下字段：

- 节点 id
- `editor_skill_id`
- `line_id`
- `tier`
- `parent_id`
- `next_ids`

允许调整的字段：

- `display_name`
- `group_id`
- `attr`
- `runtime`
- `desc.single`
- `desc.advanced`
- `route_tags`
- `icon`
- `quality`

### 8.2 route tag 兼容

当前其他系统仍在使用部分旧 route tag。为了避免攻击技能、自动效果、奖励系统一起失效，第一阶段必须遵守：

- 不删除现有关键兼容 tag，只允许增补
- 需要继续保留功能含义的 tag 包括：
  - `growth`
  - `boss`
  - `burst`
  - `clear`
  - `tempo`
  - `skill`
  - `spell_cycle`
  - `resonance`
  - `burn`
  - `cold_tide`
  - `shock`
  - `economy`
  - `survival`
  - `blood_pact`
  - `armor_break`

解释规则如下：

- `route_tags` 第一层继续服务技术联动
- `display_name` 和 `desc` 才是玩家可见语义
- 只有在对应联动系统也同步换语义时，才允许删除旧 tag

## 9. 模板复用与代码边界

### 9.1 当前模板现状

当前 `bond_templates` 中已有 4 个模板：

- `static_attr`
- `per_second_growth`
- `kill_stack`
- `basic_attack_modifier`

这 4 个模板本质上都只是“激活/卸载静态包”的薄封装。实际效果主要来自：

- `node_def.attr`
- `node_def.runtime`
- `bonds_chain.lua` 中的动态更新和事件钩子

因此第一阶段的结论是：

- 不需要因为内容重做就大规模新增模板文件
- 第一阶段优先改 `bond_nodes.lua`
- 真正不够的是“运行时钩子能力”，不是模板文件数量

### 9.2 当前已支持的效果类型

第一阶段可直接复用的能力包括：

- 静态属性加成
- 每秒属性成长
- 击杀后增加攻击
- 普攻伤害加成
- 技能伤害加成
- 全伤加成
- 暴击伤害
- 吸血
- 低血量增伤
- 精英/Boss 增伤
- 攻速、范围、投射物、弹射、多重箭
- 击杀金币收益
- 击杀奖励倍率
- 技能回响概率
- 五行伤害并兼容旧元素 tag

### 9.3 第一阶段不强求完整复刻的效果

以下效果需要在第二阶段补运行时，不作为第一阶段阻塞项：

- `魔爆术` 这类定时主动释放法术
- `概率立即重置冷却`
- `斗气领域` 这类持续范围伤害
- `敌人护甲降低 / 攻击降低`
- `每点敏捷额外提供攻击力`
- `每 100 点生命上限转属性`
- 解锁时立即发放大量金木资源
- 每秒金木收益
- 弃牌 / 吞牌 / 资源换吞噬类效果

## 10. 第一阶段代码改造范围

第一阶段代码只动“内容层”，不动“框架层”。

## 13. 最终结论

本轮链式羁绊重做的核心原则如下：

- 机制不动，内容重做
- xmind 管结构，图片表管内容
- 29 节点骨架保留
- 节点 id 和 `editor_skill_id` 优先不动
- 第一阶段先把名字、描述、主链结构和基础效果拉回正确语义
- 第二阶段再补法术、体术、经济中的高辨识度特殊机制
