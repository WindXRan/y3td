## 1. 目标

本轮 UI 重做以 `maps/EntryMap/script/docs/UI` 下的截图为视觉与信息架构参考，采用“按系统逐个重做”的方式推进。

本次先做方案对齐，不直接进入代码实现。

同时明确一个前提：

- **代码尽量重新实现**
- 不以复用旧 `script/ui/runtime_hud.lua`、`script/ui/outgame.lua` 为前提
- 旧代码可以作为数据字段与交互行为的参考，但不作为新架构约束

## 3. 重做原则

### 3.1 不是像素级抄图，而是结构级重建

目标是把截图里的信息层级、分区关系、视觉权重重建到当前项目，而不是机械一比一复刻。

优先保留：

- 区域划分
- 信息阅读顺序
- 主次按钮关系
- 主界面气质

首版可放宽：

- 贴图一致性
- 花纹装饰复杂度
- 个别图标和边角细节

### 3.2 新代码和旧代码彻底分层

新一版 UI 建议独立目录，不与当前运行时模块混写。

建议新目录：

- `maps/EntryMap/script/ui_rebuild/`

建议首批模块：

- `hud_root.lua`
- `hud_top_bar.lua`
- `hud_bottom_bar.lua`
- `hud_mainline_tracker.lua`
- `panel_profile.lua`
- `panel_inventory.lua`
- `panel_talent.lua`
- `panel_treasure_chest.lua`
- `panel_treasure_pool.lua`
- `outgame_mode_select.lua`
- `ui_state.lua`
- `ui_bindings.lua`

旧目录 `script/ui/` 暂时不删，作为以下用途保留：

- 查现有数据来源
- 查已有交互
- 对照现有节点命名

### 3.3 编辑器 UI 与 Lua 代码同步重建

新 UI 不建议继续完全依赖旧 `GameHUD` 节点树。

建议采用：

- 编辑器负责主面板骨架与固定节点
- 新 Lua 负责数据刷新、按钮事件、显隐切换、分页和动态列表

也就是说，本轮不是只改 JSON，也不是只改 Lua，而是新骨架 + 新控制层一起建。

## 5. 推荐实施顺序

为了保证每做完一块都能接进当前游戏，建议按下面顺序推进：

1. 系统 A：局内顶部 HUD
2. 系统 C：局内主线追踪区
3. 系统 B：局内底部主操作区
4. 系统 H：局外模式选择页
5. 系统 D：局内角色信息页
6. 系统 E：局内装备页
7. 系统 F：局内天赋 / 通用存档页
8. 系统 G：局内夺宝相关页

这个顺序的理由：

- A + C + B 组成局内主框架，做完后整体观感就已经成立
- H 独立度高，适合在局内 HUD 稳定后单独切
- D / E / F / G 都属于弹出式面板，适合在主框架稳定后逐个补

## 6. 每个系统的交付方式

每一个系统都按同一节奏推进：

1. 先出系统设计小文档
2. 再出节点草图
3. 再建编辑器 UI 骨架
4. 再写新 Lua 控制层
5. 最后接真实数据与交互

每次只处理一个系统，不跨系统同时大改。

## 7. 新 UI 架构建议

## 7.1 层级划分

建议拆成 4 层：

- `UIConfig`：常量、尺寸、颜色、路径、资源映射
- `UIView`：负责节点创建与节点缓存
- `UIPresenter`：负责把运行时数据映射到 UI
- `UIRouter`：负责面板切换、按钮分发、系统间跳转

### 7.2 代码职责

`UIConfig`

- 存节点路径
- 存默认文案
- 存样式常量

`UIView`

- 获取节点引用
- 初始化静态状态
- 提供安全 set_text / set_visible / set_image 方法

`UIPresenter`

- 刷新顶部资源
- 刷新波次
- 刷新 Boss
- 刷新任务
- 刷新角色属性

`UIRouter`

- 打开关闭面板
- 处理页签切换
- 处理按钮点击
- 统一热键入口

### 7.3 为什么这样拆

这样拆以后有几个好处：

- 不会再出现一个文件把创建、逻辑、刷新、事件全塞在一起
- 每个系统可以单独重写
- 后续接编辑器节点时更稳
- 测试也能按系统拆

## 8. 与旧实现的关系

本轮对旧代码的策略是“只借数据，不借结构”。

可以参考的旧内容：

- 资源字段来自哪里
- 波次 / Boss / 挑战 / 属性值怎么取
- 哪些按钮已有逻辑
- 哪些系统已有状态存储

不建议继续沿用的旧内容：

- 巨型单文件式 HUD 架构
- 旧节点路径直接拼接的写法
- 界面创建和业务逻辑强耦合
- 一个系统跨多个临时 UI 节点散落创建

## 9. 第一阶段建议定案

我建议先定下面这件事：

- **第一刀先做“系统 A：局内顶部 HUD”**

原因：

- 它在截图里识别最明确
- 它是整套局内 UI 的视觉锚点
- 它和底部面板、主线面板耦合较低
- 做完后最容易判断新风格是否对路

第一版顶部 HUD 先纳入这些块：

- 左上按钮组
- 中间模式 / 难度 / 波次 / Boss 区
- 顶部资源栏
- 顶部播报条
- 右上玩家信息表

第一版暂不纳入：

- 复杂动画
- 多人完整榜单扩展
- 所有调试按钮
- 老系统零散浮动按钮
