# 局内成长系统

> 从《项目主 GDD（初版）》拆分而来，便于按专题维护。

## 6. 局内成长系统

### 6.1 等级成长
- 击杀获得经验，英雄每升 `1 级` 获得 `1 个技能点`。
- `技能点` 只用于强化攻击技能。
- `羁绊抽卡` 是独立系统，不消耗技能点。
- 每次羁绊抽卡固定消耗 `100 木材`。
- 当前首发英雄采用 `1 个固定基础位 + 4 个自由攻击技能位` 的战斗框架。
- 当前首版局内成长已定案 `普攻`、`青木灵矢`、`赤炎箭`、`寒泉箭`、`乙木天雷` 五套攻击技能树。
- 卡牌池通过前置、排斥、重复次数和稀有度共同控制 build 成型路径。
- 每次按下 `G` 或点击 `技能` 按钮时，固定进入一次 `攻击技能强化 3 选 1`，候选中既可能出现 `新攻击技能卡`，也可能出现 `已有攻击技能强化卡`。

#### 6.1.1 攻击技能强化池结构
当前首版攻击技能强化池采用 `全局混合池` 结构。每次 `3 选 1` 都从全局攻击技能池中生成，但每张卡都必须明确标记所属技能。全局池固定包含两大类内容：
- `新攻击技能卡`
- `已有攻击技能强化卡`

局内攻击技能位采用 `1+4` 结构：
- `1` 个固定基础位，固定为 `普攻`
- `4` 个自由攻击技能位，用于装配局内获得的通用攻击技能

`新攻击技能卡` 规则：
- 只会从当前尚未装配的通用攻击技能中生成。
- 当玩家当前自由攻击技能位未满时，候选中有概率出现 `新攻击技能卡`。
- 当玩家已经装满 `4 个自由攻击技能位` 时，`新攻击技能卡` 从候选池中移除，后续只生成 `已有攻击技能强化卡`。
- 玩家选择 `新攻击技能卡` 后，该技能立刻装配到当前最左侧空闲自由位。
- 单局目标节奏为 `30 分钟`，推荐在 `15-20 分钟` 开始出现第一张终局牌，因此前中期的新技能投放必须更积极。

推荐投放节奏：
- 当前只有 `普攻` 时，前 `5 分钟` 应高概率补出第 `2` 个攻击技能。
- 当前已有 `2-3` 个攻击技能时，`5-15 分钟` 应逐步补齐第 `4`、第 `5` 个攻击技能。
- 当前已满 `5 个攻击技能` 时，后续成长重心完全转向已有技能强化卡。
- 若玩家连续 `3 次` 强化都没有看到 `新攻击技能卡`，且自由位仍未装满，则下一次候选强制至少出现 `1 张新攻击技能卡`。

`已有攻击技能强化卡` 规则：
- 可以来自当前已装配的任意攻击技能，包括 `普攻`。
- 每张强化卡必须显示所属技能标签。
- 强化卡的前置、排斥、唯一性与重复次数，严格按对应技能自己的规则校验。
- 候选分配应优先覆盖 `2 个以上不同技能`，避免长期只喂单一技能。
- 新解锁的攻击技能在解锁后的接下来 `2 次 G 强化` 内应获得明显加权，保证玩家尽快看到技能成型方向。

当前攻击技能强化统一采用 `9 张强化牌 + 1 张终局牌` 模板：
- `普通 3`
  负责基础伤害、频率、功能成长
- `优秀 3`
  负责范围、数量、形态成长
- `稀有 3`
  负责状态、对单或高威胁目标、特殊触发成长
- `传奇 1`
  负责终局形态与技能毕业

该模板适用于 `普攻`、首批四个通用技能与第二批通用技能。除 `传奇` 外，单张卡默认只承载一类核心成长方向。

#### 6.1.2 卡牌出现规则
每张卡牌应至少带有以下字段：
- `名称`
- `品质`
- `类型`
- `所属技能`
- `局外解锁等级`
- `效果`
- `前置`
- `排斥`
- `是否可重复`
- `最大叠加次数`
- `分类标签`
- `推荐联动标签`

出现逻辑：
- 局内攻击技能强化只根据当前局构筑状态判定，不再设置局内等级门槛。
- 如果前置未满足，则不能进入候选池。
- 如果排斥卡已拥有，则不能进入候选池。
- 已达到最大叠层次数，则不再出现。
- 已经拥有的唯一卡不再重复出现。
- 如果卡牌尚未被局外等级开放，则不能进入候选池。

#### 6.1.2A 攻击技能强化卡配表字段
首版建议把 `G` 相关内容拆为 `技能表` 与 `强化卡表` 两张主表，避免“技能定义”和“卡牌定义”混在一张表里难维护。

`技能表`

| 字段 | 含义 |
| --- | --- |
| `skill_id` | 技能唯一 ID，如 `normal_attack`、`arcane_arrow` |
| `skill_name` | 技能名称 |
| `skill_type` | 技能类型，如 `normal`、`active` |
| `default_slot` | 默认技能位；`普攻` 固定为 `1`，其余默认空 |
| `unlock_level_outgame` | 局外解锁等级 |
| `damage_type` | 旧兼容字段，如 `物理`、`法术` |
| `damage_form` | 运行时伤害形态，如 `weapon`、`spell` |
| `element` | 五行字段，如 `metal`、`wood`、`water`、`fire`、`earth`、`none` |
| `damage_label` | 玩家可见伤害标签，如 `金行箭罡`、`木行灵矢` |
| `base_desc` | 技能基础描述 |
| `positioning_tags` | 定位标签，如 `aoe`、`control`、`burst` |
| `is_starter` | 是否为初始自带技能 |

`强化卡表`

| 字段 | 含义 |
| --- | --- |
| `card_id` | 卡牌唯一 ID |
| `card_name` | 卡牌名称 |
| `card_kind` | 卡牌大类：`new_skill` 或 `enhance` |
| `quality` | 品质：`common`、`excellent`、`rare`、`legendary` |
| `target_skill_id` | 所属技能；新技能卡填将被解锁的技能 ID，强化卡填被强化技能 ID |
| `category_tag` | 分类标签，如 `damage`、`frequency`、`function`、`range`、`count`、`form`、`state`、`elite`、`trigger`、`legendary` |
| `type_tags` | 类型标签，如 `new_skill`、`key_upgrade`、`route_shift` |
| `synergy_tags` | 推荐联动标签 |
| `core_desc` | 核心效果描述 |
| `extra_desc` | 补充说明 |
| `effect_1_type` | 第 1 条效果类型 |
| `effect_1_value` | 第 1 条效果数值 |
| `effect_2_type` | 第 2 条效果类型 |
| `effect_2_value` | 第 2 条效果数值 |
| `prereq_cards` | 前置卡列表 |
| `exclude_cards` | 排斥卡列表 |
| `max_stack` | 最大叠加次数 |
| `is_unique` | 是否唯一 |
| `outgame_unlock_level` | 局外解锁等级 |
| `pool_weight_base` | 基础权重 |
| `ui_new_label` | 是否显示 `NEW` |

补充约定：
- `effect_x_type + effect_x_value` 只用于首版快速落地；后续复杂效果可以升级为 `effect_group_id` 或独立效果表。
- `prereq_cards` 与 `exclude_cards` 建议存 ID 列表，不直接存文案。
- `target_skill_id` 是攻击技能强化系统里最关键的字段，所有权重、标签和 UI 高亮都围绕它结算。

#### 6.1.3 卡牌品质
当前版本固定采用四品质：
- 普通
- 优秀
- 稀有
- 传奇

品质与作用挂钩：
- 普通：基础伤害、频率、功能强化
- 优秀：范围、数量、形态变化
- 稀有：状态特化、对精英 / Boss 特化、触发强化
- 传奇：终局形态与毕业卡

#### 6.1.4 卡池设计原则
- 每次消耗 `1 个技能点` 时固定生成 `3 张攻击技能强化候选卡牌`。
- 同一次 `3 选 1` 中，允许同时出现 `新攻击技能卡` 与 `已有攻击技能强化卡` 的混合候选。
- 当玩家当前自由攻击技能位未装满时，候选池需要保留一定概率投放 `新攻击技能卡`，保证 build 有机会逐步成型完整技能组。
- 当玩家当前已拥有 `普攻 + 4 个自由攻击技能` 时，候选池自动转为纯 `已有攻击技能强化卡`。
- 新攻击技能卡的投放概率以 `当前已装配技能数` 为主，以 `当前波次` 为辅，不按玩家等级单独做阶梯。
- 首版推荐让玩家在正常节奏下于前 `10 分钟` 基本成型 `3-4` 个攻击技能，并在 `15 分钟` 左右接近补满第 `5` 个攻击技能。
- 候选构成上优先保证“每轮最多 `1 张新攻击技能卡`”，其余位置默认交给 `已有技能强化卡`。
- `已有技能强化卡` 的技能归属分配，以 `已有投入`、`最近选择`、`是否刚解锁` 三类权重共同决定，不按固定轮转平均分配。
- 前期卡池不生成前置不足的高阶卡。
- 同一局中只要前置满足，高阶卡即可在任意一次升级时进入候选池。
- 当玩家已经形成某条路线后，同路线卡的出现权重提升。
- 排斥路线的卡牌权重直接归零。

#### 6.1.5 当前攻击技能卡池分支
当前攻击技能卡池固定按“技能分支 + 标准模板”维护：
- `普攻` 分支
  作为固定基础位，承担持续输出主核
- 首批四个通用技能分支
  `青木灵矢`、`赤炎箭`、`寒泉箭`、`乙木天雷`
- 第二批通用技能分支
  `剑气`、`奥术激光`、`奥术射线`、`冰霜新星`、`闪电链`、`地震`、`龙卷风`、`电磁网`、`陨石`、`飓风`、`火球`

每个技能分支都按同一套 `普通 3 / 优秀 3 / 稀有 3 / 传奇 1` 模板生产，不再使用旧的 `5 层结构` 语言。

#### 6.1.6 首版卡池定案
首版攻击技能强化池定案目标不是预设少量手工层级卡，而是先把所有攻击技能统一纳入同一套模板：
- `普攻`
- `青木灵矢`
- `赤炎箭`

第二批通用攻击技能按章节扩展追加：

- `1-4` 追加 `奥术射线`

- `1-6` 追加 `闪电链`
- `1-7` 追加 `地震`
- `1-8` 追加 `龙卷风`
- `1-9` 追加 `电磁网`

- `1-11` 追加 `飓风`

章节扩展后，攻击技能池应能够稳定支撑：

#### 6.1.7 与属性面板的关系
这套攻击技能强化池并不是独立系统，而是直接改写属性面板：
- `攻速提升 / 超速射击` 对应 `攻速` 和 `攻击间隔`
- `强化箭矢` 对应 `普攻伤害`
- `多重箭 / 多重增伤` 对应 `多重数量 / 多重伤害`
- `弹射箭 / 弹射增伤` 对应 `弹射次数 / 弹射伤害`
- 五行箭矢分支对应 `火行伤害 / 水行伤害 / 木行伤害 / 金行伤害`

也就是说：
- 面板属性定义的是“你现在有多强”
- 攻击技能强化池定义的是“你这一把的攻击技能是怎么变强的”

#### 6.1.8 五个基础技能定位校准（2026-04）

- `普攻`
  - 目标身份：`持续物理主核`
  - 中期质变：`多重 / 弹射 / 对单猎杀`
  - 终盘质变：`真实分裂 / 神箭手 / 破甲强弩`
- `青木灵矢`
  - 目标身份：`高频法术覆盖`
  - 中期质变：`次级箭 / 齐射 / 爆裂`
  - 终盘质变：`区域高频奥术覆盖`
- `赤炎箭`
  - 目标身份：`爆炸灼烧主核`
  - 中期质变：`点燃 -> 扩散 -> 二次爆炸`
  - 终盘质变：`爆燃火箭 / 火箭爆破 / 炎爆术`
- `寒泉箭`
  - 目标身份：`控场穿透主核`
  - 中期质变：`穿透 / 冰片 / 控制增伤`
  - 终盘质变：`先控住再打碎`
- `乙木天雷`
  - 目标身份：`连锁导电主核`
  - 中期质变：`感电 / 额外目标 / 磁暴`
  - 终盘质变：`怪群导电扩散`

### 6.2 装备系统
- 当前版本装备系统固定采用 `单装备成长武器` 方案。
- 全游戏首版默认只有 `1 件装备`，即当前装备中的 `成长武器`。
- 成长武器长期装备在角色身上，不再拆分 `武器 / 防具 / 饰品` 多部位结构。
- 成长武器可以使用 `金币` 强化。
- 成长武器每提升 `10 级` 可选择 `1 次词缀`。
- 装备品质首版保留显示字段，但首发成长武器默认从 `普通` 品质开始。
- 成长武器的核心作用是稳定提供 `全属性` 与定向成长词缀，而不是靠频繁换装驱动 build。
- `宝物` 不并入成长武器词缀系统，而是作为独立于成长武器之外的第三套局内构筑组件。

#### 6.2.1 首版成长武器定案
首版成长武器固定参考当前截图方案：
- 名称示例：`狩猎长弓`
- 初始品质：`普通`
- 基础效果：提供固定的 `全属性加成`
- 强化消耗：`金币`
- 成长方式：通过金币强化提高装备等级
- 词缀规则：每 `10 级` 触发一次词缀选择

#### 6.2.1A 成长武器强化费用与操作规则
- 首版成长武器等级上限固定为 `50 级`，并在 `10 / 20 / 30 / 40 / 50 级` 触发词缀节点。
- 玩家可通过 `点击底部成长武器格子` 或 `按下 B 打开背包 -> 成长武器页签` 进入强化入口。
- 成长武器强化不切出战斗场景，不额外进入独立商店阶段；玩家在战斗继续进行时自行决定是否停手强化。
- 每次点击 `强化` 按钮只提升 `1 级`，不做长按连升、自动补满或一键升到下一个词缀节点，避免误操作把金币一次性花空。
- 当成长武器到达 `10` 的整数倍等级时，立即进入 `待选词缀` 状态；在该词缀完成选择前，武器不能继续提升等级。
- 若金币不足当前强化费用，则 `强化` 按钮保留显示，但状态改为 `金币不足`，并展示下一次升级所需金币。
- 首版建议把金币强化做成稳定的线性递增费用，便于读懂和调数；推荐费用分段为：

| 等级区间 | 单次强化费用 | 设计目标 |
| --- | --- | --- |
| `1 -> 10` | `25 金币 / 级` | 第 1 波内快速建立第一次金币消费反馈 |
| `11 -> 20` | `35 金币 / 级` | 第 2 波开始要求玩家做基础金币分配 |
| `21 -> 30` | `45 金币 / 级` | 第 3 波成为中期主要金币出口 |
| `31 -> 40` | `60 金币 / 级` | 第 4 波前后开始明显拉开 build 完整度 |
| `41 -> 50` | `80 金币 / 级` | 第 5 波承担终盘金币消耗与 Boss 检定 |

- 按上述建议，成长武器从 `1 级` 升到 `50 级` 的总金币消耗约为 `2450`，应被设计为一整局标准模式中的核心金币出口。
- 首版推荐的局内强化目标为：
  `第 1 波结束前` 达到约 `8 到 10 级`
  `第 2 波结束前` 达到约 `16 到 20 级`
  `第 3 波结束前` 达到约 `24 到 30 级`
  `第 4 波结束前` 达到约 `34 到 40 级`
  `第 5 波期间` 冲击 `45 到 50 级`
- 词缀节点的 `4 选 1` 只提供 `3 次免费刷新`，首版不继续开放额外付费刷新，避免金币同时承担 `升级` 与 `刷词缀` 两种成本而导致主循环变钝。
- 成长武器不支持 `出售`、`重置`、`退还金币` 或 `局内临时换装`；首版要强调的是“持续投入同一件武器”，而不是商店倒货式操作。

#### 6.2.1B 成长武器运行时状态机
首版建议把成长武器系统明确实现为一个小型状态机，而不是靠若干 UI 条件散落判断。

状态定义：
- `normal`
  武器可正常查看；若金币足够且等级未满，则允许执行 `强化 +1`。
- `pending_affix`
  当前刚好达到 `10` 的整数倍等级，存在一组未结算的词缀候选；在完成本轮词缀选择前，禁止继续强化。
- `max_level`
  已达到 `50 级`，不再允许强化，但仍允许查看武器详情与已获得词缀。

状态切换规则：
- `normal -> pending_affix`
  当一次强化结算后，武器等级到达 `10 / 20 / 30 / 40 / 50` 节点时触发。
- `pending_affix -> normal`
  当前词缀候选完成 `4 选 1` 后触发；若当前节点并非最终节点，则回到可继续强化状态。
- `pending_affix -> max_level`
  当当前等级已到 `50` 且完成最后一个词缀节点选择后触发。
- `normal -> max_level`
  仅当外部调试指令或补偿逻辑直接把武器等级改到 `50` 且当前不存在待选词缀时允许进入；正式流程中一般不会直接走这条边。

首版强化流程建议固定为：
1. 点击 `强化`
2. 校验 `当前状态 = normal`
3. 校验 `等级 < 50`
4. 校验 `金币 >= 当前强化费用`
5. 扣除金币并提升等级
6. 应用本级基础成长
7. 若进入词缀节点，则生成一轮 `pending_affix_round`
8. 刷新武器格子与背包页签显示

首版词缀处理流程建议固定为：
1. 打开当前未结算词缀轮次
2. 若玩家点击 `刷新`，消耗本轮免费刷新次数并重生成 `4 个候选`
3. 若玩家选中 `1 个候选词缀`，立即写入武器词缀列表
4. 清空当前 `pending_affix_round`
5. 根据当前等级决定返回 `normal` 还是 `max_level`

实现约束：
- `pending_affix` 必须是运行时唯一状态，不允许同一把武器同时挂多轮未结算词缀。
- 当 `pending_affix` 存在时，成长武器格子、背包页签和任何快捷入口都应统一表现为 `待选词缀`，避免玩家从不同入口看到不一致状态。
- 首版不允许“先把 20、30、40 节点一起压着，最后集中选”，这样会显著提高状态复杂度且削弱每个节点的即时反馈。

#### 6.2.2 词缀选择规则

- 词缀选择偏向直接属性成长，不与攻击技能强化池、羁绊抽卡池混用。

#### 6.2.4A 词缀轮次与刷新实现建议
首版建议把每次词缀节点都实现为独立的 `词缀轮次`，而不是临时生成一组候选后直接散在 UI 层缓存。

建议字段：

| 字段 | 含义 |
| --- | --- |
| `round_id` | 当前词缀轮次唯一 ID |
| `weapon_id` | 对应武器 ID |
| `node_level` | 对应节点等级 |
| `candidate_affix_ids` | 当前 4 个候选词缀 ID |
| `refresh_used_count` | 当前轮次已使用刷新次数 |
| `free_refresh_left` | 当前剩余免费刷新次数 |
| `state` | `pending`、`selected` |
| `created_time` | 轮次创建时间 |

实现规则：
- 每次进入词缀节点时，立即生成一条新的 `词缀轮次`。
- 刷新操作只重置 `candidate_affix_ids`，不改动 `node_level`、`round_id` 与 `free_refresh_left` 的来源定义。
- 一旦玩家完成选择，该 `round_id` 立即标记为 `selected`，并从武器运行时数据中移除引用。
- 首版不支持 `稍后选择` 之外的多轮并行，因此同一时刻最多只能存在 `1 条 state = pending` 的武器词缀轮次。

#### 6.2.5 宝物系统
- `宝物` 是独立于 `成长武器` 与 `羁绊卡` 的第三套局内构筑资源，主要承担机制补完、路线放大与稀有功能效果。
- 首版宝物的核心目标不是提供另一套“可无限堆叠白字词缀”，而是提供少量高辨识度、可被玩家记住的功能组件。
- 宝物的主要来源是 `宝物挑战成功奖励`，次要来源可以是高波次 Boss 奖励或章节固定奖励，但首版先以 `宝物挑战` 作为主要入口。
- 宝物不消耗 `技能点`、`木材` 或 `金币` 抽取，不与 `G / F` 的资源决策抢同一货币。

#### 6.2.6 宝物位与替换规则
- 玩家同一时间最多只能持有 `3 件生效中的宝物`。
- 宝物不占用 `成长武器` 格子，也不占用 `攻击技能位`、`羁绊卡位` 或 `烙印位`。
- 当玩家未持满 `3 件宝物` 时，选择新宝物后直接放入空闲宝物位。
- 当玩家已经持有 `3 件宝物` 时，再获得新的宝物候选，必须立刻从当前 `3 件` 中替换 `1 件`。
- 被替换下来的宝物进入本局 `已弃置宝物记录`，不再提供任何效果，也不会重新回到本局候选池。
- 首版中同一具体宝物默认 `单局唯一`，已经获得过或已经弃置过的宝物都不再重复出现。

#### 6.2.7 首版宝物品质与职责
- 宝物首版固定采用 `普通 / 稀有 / 史诗` 三品质结构，作为宝物系统自己的品质语言。
- `普通宝物` 负责提供稳定的通用增益与低门槛功能补完。
- `稀有宝物` 负责提供明显的路线放大、机制联动或中期 build 转折点。
- `史诗宝物` 负责提供少量高辨识度的终盘功能效果，但不应多到淹没技能树与羁绊系统本身。
- 单局内若连续 `2 次` 宝物候选都没有出现 `稀有 / 史诗`，则下一次宝物候选至少保底 `1 件稀有宝物`。

#### 6.2.8 宝物候选生成规则
- 每次触发宝物奖励时，固定展示 `3 件宝物候选`，玩家必须 `3 选 1`。
- 单次 `3 件候选宝物` 默认不出现完全相同的具体宝物。
- 候选优先从 `未获得且未弃置` 的宝物池中生成，避免局内反复看到已经处理过的同一件宝物。
- 若玩家当前已明显走成 `普攻 / 技能 / 生存 / 经济` 某条路线，则与该路线存在强联动的宝物权重可适度上调，但不应高到失去转向选择空间。
- 当玩家当前 `3 个宝物位` 已满时，宝物候选卡面必须提前显示 `将触发替换`，并在候选详情里附带系统推荐替换目标。

#### 6.2.9 首版宝物结构要求
- 每件宝物固定包含：`品质`、`名称`、`核心效果`、`类型标签`、`推荐路线标签`。
- 宝物文案应优先突出“改变了什么机制”，而不是把全部价值都写成白字数值堆叠。
- 首版宝物优先覆盖 `普攻放大`、`技能循环`、`生存补强`、`经济滚雪球` 四类职责。
- 若某件宝物会与当前已有宝物、成长武器词缀或羁绊产生明显联动，卡面可以附带 `可联动`、`当前适配` 等弱提示，但不强制锁选。

### 6.3 羁绊卡系统
- 羁绊卡系统是独立于攻击技能强化池之外的第二套局内成长系统。
- 羁绊的获得方式只有 `抽取附带羁绊名称的卡牌`。
- 羁绊卡抽取固定消耗 `100 木材`，不消耗技能点。
- 羁绊抽卡与攻击技能强化一样，采用 `主动消耗资源 -> 点击按钮 -> 打开 3 选 1 面板` 的交互形式，不会因为达到某个条件而自动强制弹出。
- 羁绊卡的基础收益来自卡牌本身，成套收益来自羁绊激活效果。
- `祝福` 不是系统名，而是一个具体的羁绊名称。

#### 6.3.1 羁绊卡位与替换规则
- 玩家同一时间最多只能持有 `7 张未被吞噬的羁绊卡`，这 `7 张` 即为当前生效中的羁绊卡位。
- 当玩家已经拥有 `7 张未被吞噬的羁绊卡` 时，再选择新的羁绊卡，必须立即从当前 `7 张` 中指定 `1 张` 进行替换。
- 被替换的旧羁绊卡视为 `已吞噬`，不再提供该卡的基础效果，也不再计入任何羁绊进度。
- 羁绊系统的核心决策应围绕 `7 张有效卡位` 展开，前期偏向凑出可触发的基础羁绊，中后期偏向为了终盘羁绊替换低价值旧卡。

#### 6.3.2 羁绊触发与层数规则
- 每个羁绊只配置 `1 层激活效果`，不采用 `2/4/6`、`3/6/9` 这类多层羁绊结构。
- 单张羁绊卡先提供自身 `基础效果`，当未被吞噬的同名羁绊卡数量达到要求时，再额外激活该羁绊唯一的 `羁绊效果`。
- 首版大多数羁绊的触发需求应控制在 `2 到 4 张` 之间。
- `2 张羁绊` 负责提供通用起手收益与早期成型感。
- `3 张羁绊` 作为首版最常见的主流触发档位，用于支撑大多数中期 build。
- `4 张羁绊` 作为高占位高回报羁绊，主要承担终盘构筑与强绑定流派。
- 同名羁绊即使超过触发数量，也不会把羁绊效果叠到第二层；额外同名卡只保留其卡牌自身基础效果。
- 羁绊进度一律只统计 `未被吞噬的羁绊卡`，文案统一显示为 `当前有效数量 / 触发需求`，例如 `2/3`、`3/4`。

#### 6.3.3 首版羁绊卡结构要求
- 每张羁绊卡固定包含 `品质`、`羁绊名称`、`基础效果`、`触发需求`、`当前进度`、`唯一羁绊效果`。
- 基础效果负责让单卡也有拿取价值，避免前期抽到未成套羁绊时手感过空。
- 羁绊效果负责定义流派方向，优先提供机制变化、乘区补强或资源节奏改变，而不是纯粹堆白字。
- 同一羁绊下的多张卡应共享羁绊名称与触发目标，但允许各自拥有不同的基础效果，以提升替换决策空间。
- 首版优先做 `易于 2 张成型的通用羁绊` 与 `需要 3 到 4 张成型的路线羁绊` 两类，避免一开始就把羁绊池做得过散。
- 首版中每个羁绊的 `推荐卡数` 必须与 `触发张数` 保持一致，避免玩家长期单留 `1 张未成套羁绊卡` 占用有效卡位。
- 因此首版默认不设计“超过触发张数的同羁绊冗余卡”，后续若要扩展同羁绊变体卡，必须先同步重做替换体验与权重规则。
- 羁绊卡固定采用 `普通 / 稀有 / 史诗` 三品质结构，不再做成只有名称不同但品质无区别的平铺池。
- 同一羁绊下的所有羁绊卡品质必须保持一致，不采用“同一羁绊内部混合普通 / 稀有 / 史诗”的设计。
- 羁绊卡品质与职责挂钩：
  `普通` 负责提供稳定白字与基础构件，
  `稀有` 负责提供路线放大与关键中阶效果，
  `史诗` 负责提供高价值终盘组件或稀有构筑支点。
- 品质不仅影响卡面颜色，也应影响候选权重、基础效果强度与玩家的保留意愿。
- 首版默认采用与攻击技能强化一致的视觉语言：
  `普通` 为黑 / 深色，
  `稀有` 为蓝色，
  `史诗` 为紫色。
- 为避免羁绊成型过度看脸，羁绊抽卡每次展示 `3 张候选卡` 时，固定采用品质保底：
  如果前两张均为 `普通`，则第 `3 张` 必须从 `稀有 / 史诗` 池中生成。
- 羁绊品质权重不固定写死，而是随 `当前波次` 动态提升高品质出现率。
- 首版建议采用以下波次品质权重：
  `第 1 波`：`普通 68%`、`稀有 27%`、`史诗 5%`
  `第 2 波`：`普通 60%`、`稀有 31%`、`史诗 9%`
  `第 3 波`：`普通 52%`、`稀有 34%`、`史诗 14%`
  `第 4 波`：`普通 45%`、`稀有 37%`、`史诗 18%`
  `第 5 波`：`普通 38%`、`稀有 40%`、`史诗 22%`
- 这样处理的目标是：
  前期以 `普通羁绊` 帮助快速起手，
  中期提高 `稀有羁绊` 的成型概率，
  终盘让 `史诗羁绊` 有足够出现率承担高风险高回报路线。
- 羁绊的成套触发主要看 `有效卡数量`，不额外要求品质组合；但高品质羁绊卡应承担更强的单卡基础效果，以形成“先成套，再追高品质关键卡”的局内体验。
- 羁绊品质优先作为“整套路线稀有度”使用，而不是作为同羁绊内部卡位的层级区分。

#### 6.3.4 首版正式羁绊池定案
首版正式羁绊池先固定为 `10 个羁绊`，按 `4 个 2 张羁绊 + 4 个 3 张羁绊 + 2 个 4 张羁绊` 组成，总卡量为 `28 张`。这样既能保证前期容易成型，也能在 `7 张有效卡位` 的限制下形成明确取舍。

`祝福`：`普通`，通用续航羁绊，`2 张触发`
- 推荐卡数：`2 张`
- `圣水`：`生命恢复 +8`
- `祈愿`：`生命值 +160`
- 羁绊效果：每 `8 秒` 回复 `6% 最大生命`，并获得 `8% 伤害减免`，持续 `2 秒`。

`狂战`：`普通`，低血爆发羁绊，`2 张触发`
- 推荐卡数：`2 张`
- `怒意`：`攻击力 +18`
- `热血`：`攻速 +12%`
- 羁绊效果：生命低于 `50%` 时，额外获得 `攻速 +35%` 与 `普攻伤害 +25%`。

`猎手`：`普通`，精英 / Boss 对策羁绊，`2 张触发`
- 推荐卡数：`2 张`
- `追猎`：`BOSS伤害 +12%`
- `剿灭`：`精英伤害 +12%`
- 羁绊效果：对 `精英` 与 `Boss` 的伤害额外 `+25%`，且首次命中其时再额外造成 `50%` 当前攻击力的附加伤害。

`贪欲`：`普通`，经济滚雪球羁绊，`2 张触发`
- 推荐卡数：`2 张`
- `铜币`：`杀敌金币 +20%`
- `囤积`：`每秒金币 +2`
- 羁绊效果：每击杀 `30 个敌人`，额外获得 `40 木材` 与 `30 金币`。

`连射`：`稀有`，普攻铺场羁绊，`3 张触发`
- 推荐卡数：`3 张`
- `疾弦`：`攻速 +10%`
- `开弓`：`普攻伤害 +12%`
- `扩散`：`多重伤害 +15%`
- 羁绊效果：`多重数量 +1`，`多重伤害 +35%`。

`连锁`：`稀有`，弹射清怪羁绊，`3 张触发`
- 推荐卡数：`3 张`
- `回响`：`弹射伤害 +15%`
- `折返`：`攻击范围 +40`
- `追击`：`普攻伤害 +10%`
- 羁绊效果：`弹射次数 +1`，`弹射伤害 +35%`。

`奥术`：`稀有`，技能输出羁绊，`3 张触发`
- 推荐卡数：`3 张`
- `咏唱`：`技能伤害 +12%`
- `导能`：`攻击增幅 +8%`
- `聚焦`：`攻击范围 +40`
- 羁绊效果：`技能伤害 +35%`，每次释放技能后接下来 `3 秒` 内再获得 `所有伤害 +12%`。

`处决`：`稀有`，斩杀终结羁绊，`3 张触发`
- 推荐卡数：`3 张`
- `破绽`：`普攻伤害 +10%`
- `压制`：`精英伤害 +10%`
- `突刺`：`攻击力 +16`
- 羁绊效果：对生命低于 `35%` 的敌人造成的伤害额外 `+40%`。

`成长`：`史诗`，长期叠层羁绊，`4 张触发`
- 推荐卡数：`4 张`
- `磨砺`：`攻击增幅 +8%`
- `积累`：`杀敌攻击 +1`
- `历战`：`杀敌加成 +10%`
- `蓄势`：`攻击力 +20`
- 羁绊效果：每击杀 `20 个敌人`，永久获得 `攻击力 +12`；每达到 `100` 击杀，额外获得 `攻击增幅 +4%`。

`坚壁`：`史诗`，高占位防守羁绊，`4 张触发`
- 推荐卡数：`4 张`
- `铁甲`：`生命值 +180`
- `壁垒`：`伤害减免 +4%`
- `自愈`：`生命恢复 +10`
- `稳固`：`护甲 +6`
- 羁绊效果：生命高于 `80%` 时额外获得 `伤害减免 +12%`；生命低于 `50%` 时每秒回复 `3% 最大生命`。

#### 6.3.5 首版羁绊池实现规则
- 首版优先采用 `同羁绊下多张不同卡` 的设计，不建议用同一张具体卡重复抽取来凑进度。
- 同一张具体羁绊卡默认 `单局唯一`，避免出现“同名同效果重复叠满”导致替换决策失真。
- 首版每个羁绊的卡牌数量固定等于其触发张数，玩家只要决定保留该羁绊，就应以“完整凑齐”作为默认目标。
- 首版采用“按羁绊整体定品质”的规则：
  `祝福 / 狂战 / 猎手 / 贪欲` 为 `普通羁绊`，
  `连射 / 连锁 / 奥术 / 处决` 为 `稀有羁绊`，
  `成长 / 坚壁` 为 `史诗羁绊`。
- 同一羁绊下的所有卡在候选池、卡面描边、提示文本与吞噬界面中都使用同一品质表现。
- `2 张羁绊` 主要承担前期稳定成型与补足通用能力。
- `3 张羁绊` 主要承担主流输出路线成型。
- `4 张羁绊` 主要承担终盘大方向，不建议在前 5 分钟大量出现。
- 所有羁绊抽卡与手动 `刷新` 都读取当前波次的品质权重，不单独保留旧波次权重。
- 当玩家已经形成某一羁绊路线时，同路线剩余卡的权重可以适度提高，但不能高到失去替换博弈。
- 若某羁绊的高品质关键卡尚未出现，则该羁绊虽然理论上未成套，但卡池权重可以对其剩余卡做轻度倾斜，保证玩家有机会追完整条路线。
- 羁绊池与攻击技能池必须互相补位：
  `连射 / 连锁 / 处决` 更偏普攻路线。
  `奥术` 更偏技能路线。
  `祝福 / 坚壁` 负责生存补强。
  `贪欲 / 成长` 负责经济与滚雪球。

#### 6.3.6 羁绊抽卡生成与替换规则
- 每次消耗 `100 木材` 进入羁绊抽卡时，固定生成 `3 张候选卡`，流程顺序为：
  先按当前波次决定每个候选位的 `品质`，
  再按构筑状态决定对应品质下的 `羁绊权重`，
  最后从该羁绊的 `未拥有具体卡` 中随机抽取 `1 张`。
- 单次 `3 张候选卡` 中，默认不重复出现同一个羁绊，优先保证玩家能在“补已有路线”和“转向新路线”之间做选择。
- 已经获得过的具体羁绊卡，无论当前是 `生效中` 还是 `已吞噬`，本局都不再重复出现。
- 如果某个羁绊的全部具体卡都已获得，则该羁绊从本局后续候选池中移除。

生成权重规则：
- 没有任何进度的全新羁绊，使用该品质下的 `基础权重`。
- 已经拥有同羁绊 `1 张` 时，该羁绊剩余卡权重提升为 `基础权重 x 1.8`。
- 已经拥有同羁绊 `2 张` 时，该羁绊剩余卡权重提升为 `基础权重 x 2.4`。
- 已经拥有同羁绊 `3 张` 时，该羁绊剩余卡权重提升为 `基础权重 x 3.0`。
- 如果玩家当前已经有 `2 组及以上未成套羁绊`，则全新羁绊的基础权重额外乘以 `0.6`，避免局内构筑被过多半成品羁绊稀释。
- 如果某羁绊已经成套激活，但仍有尚未拿到的同羁绊卡，则其剩余卡只保留 `基础权重 x 0.8`，避免系统过度继续投喂已成型路线。
- `普通羁绊` 在第 `1 到 2 波` 可以作为起手主池；`稀有羁绊` 在第 `3 到 4 波` 应更容易接手主流路线；`史诗羁绊` 在第 `4 到 5 波` 才应明显进入终盘决策区。

刷新规则：
- 羁绊抽卡面板提供手动 `刷新`，用于重抽当前 `3 张候选卡`。
- 每次打开一次羁绊抽卡面板时，固定拥有 `1 次免费刷新`。
- 若继续刷新，则同一轮抽卡内刷新费用按 `20 / 40 / 60 木材` 递增，第 `4 次及以后` 固定为 `60 木材`。
- 当玩家完成本次选择并关闭面板后，刷新次数与刷新费用阶梯重置；下一次重新抽卡时重新获得 `1 次免费刷新`。
- 刷新不会改变当前波次品质权重，只会基于当前波次重新生成一组新候选。

候选锁定与暂存规则：
- 羁绊抽卡在扣除 `100 木材` 后，本轮 `3 张候选卡` 立即进入 `待选择` 状态，直到玩家选中其中 `1 张` 为止。
- 玩家可以通过 `稍后选择` 临时关闭面板，但这不会退还 `100 木材`，也不会清空本轮候选。
- `稍后选择` 只负责隐藏面板，不改变当前 `刷新次数`、`刷新费用阶梯`、`候选卡内容` 与 `是否需要替换` 的判定结果。
- 当存在 `待选择` 的羁绊候选时，`F` 与 `羁绊抽卡` 按钮统一变为 `继续选择` 状态；再次打开时直接回到同一轮候选。
- 玩家不能同时保留多轮未结算的羁绊候选，必须先处理当前这一轮，才能再次消耗木材开启下一轮抽卡。

替换与吞噬规则：
- 当玩家已持有 `7 张未被吞噬的羁绊卡` 时，选择新羁绊卡后必须立刻从当前 `7 张` 中替换 `1 张`。
- 被替换的卡立刻进入 `已吞噬` 状态，失去基础效果，并从所有羁绊进度中移除。
- 若替换某张卡会导致一个已激活羁绊失效，系统必须以红色警示明确提示 `将失去羁绊效果`。
- 若替换某张卡只会降低未成型羁绊进度，系统应显示替换前后的进度变化，例如 `2/3 -> 1/3`。
- 替换推荐只作为辅助，不强制锁定玩家选择；玩家始终可以手动指定要吞噬的卡。
- 系统默认推荐的替换顺序应为：
  优先推荐 `孤张未成套羁绊`，
  其次推荐 `低品质且进度较低的半成品羁绊`，
  最后才推荐 `已成套激活的羁绊卡`。
- 如果当前候选卡能直接补齐一个已拥有进度的羁绊，则该候选卡在卡面上需要额外显示 `可成套` 标识。
- 如果当前候选卡会迫使玩家拆掉一个更高品质的已成型羁绊，卡面上需要额外显示 `高风险替换` 标识。

#### 6.3.6A 羁绊卡配表字段
首版建议把羁绊内容拆为 `羁绊表` 与 `羁绊卡表` 两张主表，分别管理“整套羁绊规则”和“单张羁绊卡内容”。

替换辅助字段：

| 字段 | 含义 |
| --- | --- |
| `replace_priority_hint` | 推荐替换优先级提示 |
| `replace_loss_desc` | 替换后损失描述 |
| `replace_progress_preview` | 替换后的进度预览文案 |
| `breaks_active_bond` | 是否会导致已激活羁绊失效 |

补充约定：
- `bond_id` 是羁绊系统最关键的主键，进度、成套、权重提升、替换推荐都基于它。
- `quality` 在首版必须同时写在 `羁绊表` 和 `羁绊卡表`，便于 UI 和逻辑层各自读取。
- `state_tags` 为运行时字段时，可由逻辑动态生成；若配表阶段只想保留静态字段，也可以只在 UI 层临时计算。

#### 6.3.6B 羁绊分层与升品补充（2026-04）

- `T1 / T2 / T3` 是目标设计层级，不等于当前 runtime 已实现品质。
- 进入进化链的羁绊必须满足“升阶同时升品”：
  - `T1` 目标品质建议为 `rare`
  - `T2` 目标品质建议为 `epic`
  - `T3` 目标品质建议为 `legendary`
- `bond_role` 建议区分：
  - `standalone`
  - `child`
  - `material`
  - `mother`
- `result_type` 建议区分：
  - `terminal_swallow`
  - `evolve_keep`
- 大多数 `T1` 应作为独立终点件或过渡挂件；只有少数 `T1` 进入上位进化链。

- `祝福`
- `贪欲`
- `连射`
- `成长`

- `祝福`：2 张
- `贪欲`：2 张
- `连射`：3 张

### 6.4 商店系统
- 首版不设置 `波间商店阶段`、`随机商人 NPC`、`停表采购房间` 或 `传统回合制买卖商店`。
- 但首版保留一个常驻可访问的 `即时锻造商店` 概念，其本质就是 `成长武器强化入口 + 词缀节点处理入口`。
- 所有局内成长仍主要通过 `攻击技能强化`、`羁绊抽卡`、`成长武器词缀`、`宝物候选` 与 `波次奖励` 完成；商店系统只负责承接 `金币` 这一条资源循环。

#### 6.4.1 设计定位
- 商店系统的职责不是提供另一套随机卡池，而是给 `金币` 一个稳定、随时可用、不会和 `G / F` 混币的常驻出口。
- 商店系统要解决的是“玩家有金币时能立刻转化为稳态战力”，而不是“逼玩家在商店阶段停下来做过长购物决策”。
- 因此首版商店设计必须满足：
  `入口常驻`
  `规则简单`
  `消耗可预估`
  `不会切断主线战斗节奏`

#### 6.4.2 首版商店动作范围
- 首版商店只开放以下动作：
  `成长武器强化 +1`
  `查看当前等级节点词缀`
  `处理待选词缀`
- 首版商店明确不开放以下动作：
  `购买消耗品`
  `刷新独立商品列表`
  `出售装备`
  `重置词缀`
  `临时租借宝物`
  `付费购买技能点 / 木材 / 挑战次数`
- 这样处理的目标是：把 `金币` 的身份稳定为“成长武器强化货币”，不让单局经济在首版就被过多分流。

#### 6.4.3 商店入口与阻塞规则
- 商店入口固定挂在 `成长武器格子 / 背包成长武器页签` 上，不单独占用底部核心按钮位。
- 玩家在 `G`、`F`、`宝物候选`、`烙印选择` 这类中央决策面板未结算时，不能通过 `B` 或点击成长武器格子去覆盖当前决策。
- 若成长武器当前存在 `待选词缀`，则成长武器格子与背包页签都应显示明显的 `待处理` 状态，提醒玩家先完成词缀选择再继续强化。
- 商店入口打开后默认不暂停战斗，玩家需要自己承担“此时分心强化”的战场风险；这也是即时商店与传统波间商店的关键差异。

#### 6.4.4 与其他成长系统的分工
- `金币 -> 成长武器强化`，负责稳定白字成长与词缀节点。
- `技能点 -> G 强化`，负责攻击技能树成长。
- `木材 -> F 抽卡`，负责羁绊与路线拼装。
- `挑战次数 -> 挑战事件`，负责额外资源爆发与高价值候选。
- `宝物奖励 / 烙印奖励` 不进入商店出售池，而是直接以候选面板形式结算。
- 通过这套分工，首版的四种关键局内资源各自对应不同成长入口，降低“所有资源都挤进同一个商店”带来的阅读负担。

### 6.5 事件系统
- 首版正式启用 `局内事件系统`，但事件范围明确收敛为 `限时挑战事件`。
- 首版不做纯文本随机事件、地图分支事件、随机商人事件、答题事件或额外插入式剧情事件，避免把实现复杂度铺得过早。
- 因此，当前版本的 `事件系统` 与 `挑战系统` 在设计上视为同一套东西：玩家主动点击入口，在主战场内触发一段持续 `60 秒` 的限时支线事件。

#### 6.5.1 事件系统定位
- 事件系统的核心目标是为单局主线波次之外，提供 `可主动开启` 的高价值插入式节奏点。
- 主线波次负责稳定推进；事件系统负责让玩家在“我现在要不要多承压，换一笔额外成长资源”之间做主动博弈。
- 事件系统必须做到：
  `不切地图`
  `不暂停主线`
  `奖励够值`
  `失败可承受`

#### 6.5.2 首版事件生命周期
- 每个挑战事件固定经过以下状态：
  `可进入 -> 进行中 -> 已结算`
- 同一种事件在自己尚未结算前不能重复开启第二次，但不同种类的事件允许并行存在。
- 每次事件开启后固定持续 `60 秒`；时间结束前若已清空该事件怪物组，则立即成功结算。
- 时间结束时若仍有该事件怪物残留，则本次事件立即失败，并清理对应挑战怪物组。
- 事件失败不会返还本次消耗的 `挑战次数`，但也不会追加额外惩罚，例如扣金币、扣木材或降低主线奖励。

#### 6.5.3 首版四类挑战事件
- `金币挑战`
  负责提供一笔稳定金币补给，帮助玩家更快推进成长武器等级。
- `木材挑战`
  负责提供一笔稳定木材补给，支撑 `F` 抽卡节奏和羁绊成型速度。
- `经验挑战`
  负责提供一笔集中经验补给，帮助中期追等级、追技能点窗口。
- `宝物挑战`
  负责提供高价值宝物候选，是首版事件系统中价值最高、风险也最高的一类。

首版四类挑战默认全部开放：
- `金币挑战`：承担最早期经济补给入口。
- `木材挑战`：承担木材回补与羁绊节奏支撑。
- `经验挑战`：承担经验补给与追等级窗口。
- `宝物挑战`：承担高风险高收益的宝物入口。

#### 6.5.4 事件与主线并行规则
- 事件怪物直接在主战场侧翼刷新，不额外进入独立房间。
- 事件进行中不会暂停主线普通怪刷新，也不会阻止当前波次 Boss 登场。
- 当 `Boss + 普通怪 + 事件怪` 同场存在时，系统优先保证它们都能被正常结算；若逼近性能软上限，只延后新的普通怪批次，不回收已登场的 Boss 或事件怪。
- 玩家在事件进行中仍可正常使用 `G`、`F`、`宝物候选`、`集火` 等系统，只要当前没有其他未结算中央决策面板阻塞。
- 事件奖励若为普通数值资源，则直接入账；若为 `宝物候选` 等高价值奖励，则写入 `待领取奖励` 队列，等待当前中央面板空闲后再弹出。

#### 6.5.5 事件奖励与风险定位
- 事件成功奖励必须明显高于同时间纯挂机刷怪收益，否则玩家不会主动承担额外压力。
- 但事件奖励不能高到完全碾压主线掉落与波次结算，否则玩法会退化为“攒次数打挑战”。
- `金币 / 木材 / 经验` 三类事件定位为稳定补给，不跟随主线波次倍率滚动暴涨。
- `宝物挑战` 的核心收益来自 `宝物候选` 本身，因此附带金币或木材只应作为甜头，不应反客为主。
- 首版事件系统的核心风险来自：
  `分散火力`
  `额外铺场`
  `Boss 同屏`
  `玩家需要自己判断何时值得开挑战`

#### 6.5.6 事件系统配表要求
- 首版建议事件系统最少拆出一张 `挑战表`，并保留以下关键字段：
  `challenge_id`
  `duration_sec`
  `cost_charge`
  `monster_group_id`
  `spawn_area_id`
  `reward_type`
  `reward_value_fixed`
  `danger_tags`
- 事件的 `显示规则`、`按钮状态`、`追踪区表现` 与 `剩余时间展示` 统一以 [07-UI与信息表现.md](./07-UI与信息表现.md) 中 `11.8 底部常驻玩法入口` 的定义为准。

#### 6.5.7 事件运行时状态机
首版建议把每个挑战事件都实现为独立上下文，而不是只靠“按钮是否亮着”来判断是否在进行中。

单个挑战事件状态建议固定为：
- `ready`
  默认可用、当前没有同名事件在进行中，且玩家可消耗 `1 次挑战次数` 开启。
- `preparing`
  玩家刚点击按钮，正在创建怪物组、追踪条目和计时器。
- `running`
  事件怪物已刷出，`60 秒` 倒计时进行中。
- `success_pending_reward`
  事件已经成功，但奖励仍在等待立即入账或等待中央面板空闲后弹出。
- `failed`
  时间耗尽或特殊失败条件触发，等待收尾清理。
- `closed`
  本轮事件已经完全结算并清理结束。

状态切换建议：
- `ready -> preparing`
  玩家点击按钮且 `挑战次数 >= 1` 时触发。
- `preparing -> running`
  怪物组创建成功、追踪区注册成功后触发。
- `running -> success_pending_reward`
  挑战怪物组被击杀完毕时触发。
- `running -> failed`
  倒计时归零且仍有挑战怪残留时触发。
- `success_pending_reward -> closed`
  奖励已完成入账、入队或弹窗处理后触发。
- `failed -> closed`
  残留挑战怪移除、追踪条目关闭后触发。

补充约定：
- `closed` 表示本轮事件实例结束，不等于按钮重新变为 `ready`；按钮是否重新可开仍取决于 `挑战次数` 与是否存在同名事件实例。
- 同一种挑战同时最多存在 `1 个` 非 `closed` 实例；不同挑战类型可并行各自持有 `1 个` 实例。

#### 6.5.8 事件运行时上下文字段建议
首版建议每一轮挑战事件都生成一份 `challenge_runtime_context`，作为逻辑、UI 与清理流程的统一数据源。

| 字段 | 含义 |
| --- | --- |
| `instance_id` | 本轮事件实例唯一 ID |
| `challenge_id` | 挑战类型 ID，如 `gold_trial` |
| `state` | 当前运行时状态 |
| `start_time` | 开始时间 |
| `end_time` | 预计结束时间 |
| `duration_sec` | 持续时间；首版固定 `60` |
| `spawn_area_id` | 当前使用的刷新区域 |
| `monster_group_id` | 当前怪物组配置 ID |
| `spawned_unit_ids` | 本轮刷出的单位 ID 列表 |
| `alive_count` | 当前存活挑战怪数量 |
| `total_count` | 本轮总挑战怪数量 |
| `success_rule` | 成功条件；首版固定 `kill_all` |
| `fail_reason` | 失败原因，如 `timeout`、`manual_abort` |
| `reward_type` | 奖励类型 |
| `reward_payload` | 奖励内容数据 |
| `ui_track_key` | 右侧追踪区对应条目 ID |

实现建议：
- `spawned_unit_ids + alive_count` 是事件成功与失败结算的核心字段，建议不要仅依赖“击杀数是否达到 total_count”这种单点判断。
- `reward_payload` 建议在挑战开启时就写好基础结构，避免挑战成功时临时拼装奖励导致逻辑分叉过多。
- `ui_track_key` 用于把追踪区条目和事件实例绑定，保证多挑战并行时不会相互覆盖。

#### 6.5.9 挑战怪物组与刷新计划配表建议
仅有 `挑战表` 还不够，首版建议至少再拆两张表：`挑战怪物组表` 与 `挑战刷新计划表`。

`挑战怪物组表`

| 字段 | 含义 |
| --- | --- |
| `monster_group_id` | 怪物组唯一 ID |
| `display_name` | 怪物组显示名称 |
| `elite_unit_id` | 首领单位 ID；无首领可为空 |
| `normal_unit_id` | 普通挑战怪单位 ID |
| `theme_tags` | 主题标签 |
| `soft_alive_limit` | 本怪物组软上限 |
| `despawn_on_fail` | 失败时是否统一移除；首版固定 `true` |

`挑战刷新计划表`

| 字段 | 含义 |
| --- | --- |
| `monster_group_id` | 对应怪物组 ID |
| `wave_index` | 怪物组内批次编号；首版固定递增即可 |
| `spawn_sec` | 本批次在事件开始后多久刷出 |
| `unit_role` | `elite` 或 `normal` |
| `spawn_count` | 本批次数量 |
| `spawn_area_id` | 刷新区域 ID |
| `scatter_min_distance` | 最小出生间距 |
| `follow_owner_instance` | 是否绑定到当前事件实例 |

实现好处：
- `挑战表` 负责入口、解锁和奖励。
- `挑战怪物组表` 负责“这是什么挑战怪”。
- `挑战刷新计划表` 负责“这些怪什么时候、分几批、从哪里出”。
- 这样后续想调单个挑战的密度，只需要改计划表，不需要动入口逻辑。

#### 6.5.10 高价值奖励队列规则
首版挑战事件需要与 `G`、`F`、`宝物候选`、`烙印` 共用同一套中央面板资源，因此建议正式建立 `待领取奖励队列`。

建议字段：

| 字段 | 含义 |
| --- | --- |
| `reward_entry_id` | 奖励条目唯一 ID |
| `source_type` | 来源类型，如 `challenge`、`boss`、`treasure_trial` |
| `source_instance_id` | 来源实例 ID |
| `reward_kind` | 奖励种类：`number`、`treasure_choice`、`mark_choice` |
| `priority` | 队列优先级 |
| `payload` | 奖励详细内容 |
| `state` | `pending`、`shown`、`claimed` |
| `created_time` | 创建时间 |

处理规则：
- 若奖励是 `经验 / 金币 / 木材` 这类纯数值资源，则不进队列，直接入账并播反馈。
- 若奖励是 `宝物候选` 这类必须决策的奖励，则写入队列并按优先级等待中央面板空闲。
- 队列弹出顺序建议按：
  `当前战斗中未完成的既有中央面板`
  `高优先级奖励`
  `同优先级按创建时间先后`
- `宝物挑战` 的奖励若因 `G` 或 `F` 正在待选而无法立刻展示，也必须先写入队列，不能因为用户暂时没打开而丢失。

#### 6.5.11 首版典型时序示例
为了避免实现时每个人理解不同，首版建议至少按以下三个时序做：

`金币挑战成功`
1. 玩家点击 `金币挑战`
2. 扣除 `1 次挑战次数`
3. 创建 `gold_trial` 实例并刷出 `运财小鬼`
4. `60 秒` 内全部击杀
5. 直接入账 `220 金币`
6. 播放资源反馈并关闭事件追踪

`宝物挑战成功且当前已有 G 面板待选`
1. 玩家开启 `宝物挑战`
2. 宝物挑战首领与护卫刷出
3. 玩家期间先触发一次升级，`G` 面板进入待选
4. 玩家击杀宝物挑战全部怪物
5. 系统先把 `宝物候选` 写入 `待领取奖励队列`
6. 玩家完成 `G` 选择后，系统再弹出 `宝物候选`

`木材挑战超时失败`
1. 玩家点击 `木材挑战`
2. 扣除 `1 次挑战次数`
3. 刷出 `伐木巨兽`
