# UI美术需求文档

> 基于《UI 与信息表现》《UI适配规则》《局内成长系统》整理，用于指导首版局内 HUD、中央决策面板、背包、吞噬界面与结算界面的 UI 美术出图、资源拆分与交付规范。

## 1. 文档目标
- 本文面向 `UI设计`、`UI美术`、`程序接入` 三方协作。
- 目标不是重复功能逻辑，而是把“哪些界面要做、做成什么视觉层级、需要哪些资源、如何切图与交付”说清楚。
- 首版优先服务 `标准模式 5 波 MVP`，挑战模式和局外界面只保留扩展接口，不作为本轮美术生产重点。

## 2. 视觉方向

### 2.1 核心关键词
- `单英雄守线`
- `奇幻战场`
- `肉鸽构筑`
- `高压战斗中的清晰信息`

### 2.2 首版视觉基调
- 局内整体建议采用 `深色低饱和底板 + 明亮功能色高亮` 的方向，避免大面积高亮底图抢战场注意力。
- 常驻 HUD 以 `稳` 为主，中央决策面板以 `亮` 为主，危险提示以 `硬` 为主。
- 首版不追求复杂装饰纹样，优先保证：
  `信息一眼可读`
  `功能分区清楚`
  `高价值奖励有明显尊贵感`

### 2.3 风格建议
- 底板材质建议偏 `金属 / 皮革 / 魔法纹路` 的轻装饰奇幻风，不建议做成科幻蓝屏或纯现代扁平卡片。
- 角、边、描边可有适度奇幻感，但必须控制在不压缩正文显示区域的程度。
- 卡牌、按钮、面板建议共用一套基础造型语言，避免每个系统像不同游戏拼起来。

## 3. 色彩与品质体系

### 3.1 品质色
首版固定沿用当前设计定案：

| 品质 | 主色建议 | 用途 |
| --- | --- | --- |
| `普通` | 深灰 / 黑铁色 | 基础卡面、基础标签、低优先级资源 |
| `稀有` | 蓝色 | 中高价值候选、路线成型提示 |
| `史诗` | 紫色 | 高价值奖励、终盘构筑件 |

补充约定：
- 品质色主要用于 `边框`、`角标`、`描边光`、`标题底条`，不要把整张卡面铺成纯高饱和底色。
- 同品质在 `攻击技能卡`、`羁绊卡`、`宝物卡`、`烙印卡` 上要统一语义。

### 3.2 系统功能色

| 系统 / 状态 | 颜色建议 | 说明 |
| --- | --- | --- |
| `金币` | 金黄 | 用于数值跳字、资源图标、奖励提示 |
| `木材` | 偏绿 | 与金币明确区分 |
| `经验` | 青蓝 | 用于经验条与经验飘字 |
| `生命` | 红色 | 血条与危险反馈 |
| `可进入 / 可强化` | 明亮暖色或亮青色 | 轻度脉冲高亮即可 |
| `危险 / 高风险替换 / Boss高危技能` | 橙红 | 需要明显压过普通提示 |
| `已完成 / 成功` | 偏金或亮绿 | 不与资源色冲突即可 |

### 3.3 对比度要求
- 正文文本与背景对比必须高于装饰元素。
- 大号数字、波次、倒计时、资源数值、技能点数值必须优先保证可读性，不允许被纹理吃掉。
- `禁用状态` 不等于“完全不可见”，仍需保留基础识别度。

## 4. 排版与图标原则

### 4.1 文本层级
- 标题层：用于面板名、卡牌名、页签名，建议字重更高。
- 正文层：用于核心效果描述，必须清晰稳定。
- 注释层：用于前置、排斥、替换后果、解锁条件，字号可小一档但不能难读。
- 数字层：金币、木材、经验、技能点、刷新费用、倒计时建议使用更有力量感的数字表现。

### 4.2 图标原则
- 图标优先强调 `轮廓识别`，避免细节过多导致缩小后糊成一团。
- 同类资源图标要统一透视与光向。
- 每个图标在 `32px 到 64px` 观察下仍要能认出是什么。
- 候选卡内图标应以“快速识别类型”为目标，不承担复杂剧情叙事。

## 5. 资源制作总原则

### 5.1 适配原则
- 出图必须兼容 [UI适配规则](c:\Y3TD\Y3GPT\ProjectName002\maps\EntryMap\script\UI适配规则.md)。
- 首版所有 `长条底板`、`顶部栏`、`底部栏`、`弹窗框体` 都建议支持 `九宫` 或 `左右端 + 中段拉伸`。
- 不要把关键视觉信息烘焙进不可拉伸的大底图中央，以免不同宽度下被拉坏。

### 5.2 分层原则
- 面板资源尽量拆成：
  `底板`
  `内框`
  `标题条`
  `按钮`
  `装饰角`
  `状态角标`
  `图标`
- 卡牌资源尽量拆成：
  `卡底`
  `品质边框`
  `标题条`
  `标签底`
  `高亮描边`
  `选中态覆盖`

### 5.3 状态资源原则
以下可交互控件默认需要多状态：
- `默认`
- `悬停 / 聚焦`
- `按下`
- `禁用`
- `高亮推荐`
- `进行中 / 待处理`

## 6. 分界面美术需求

### 6.1 顶部战斗状态栏
需要资源：
- `顶部主条底板`
- `波次徽记`
- `Boss 信息容器`
- `倒计时底条`
- `关卡编号底条`
- `资源格子`
- `玩家统计行底板`

视觉要求：
- 这一区域是高频查看区，装饰要轻，数值要重。
- `第 X / 5 波`、`Boss 来临倒计时`、`Boss 血量信息` 必须是第一阅读优先级。
- 资源格子建议统一形制，不同资源靠 `图标 + 数字色` 区分，不靠不同形状分裂风格。
- 顶部 HUD 建议更贴近 `MOBA 端游用户习惯`：
  `中间扁平状态带`
  `右上紧凑资源组`
  `信息横向阅读`
  不建议做成过强装饰化的漂浮仪表盘。
- 重点信息应遵循：
  `战斗节奏信息最显眼`
  `经济信息次一级`
  `统计信息最弱`
  的层级顺序。

### 6.2 左侧系统按钮与快捷提示
需要资源：
- `退出按钮`
- `设置按钮`
- `快捷键提示底条`
- `折叠 / 展开按钮`

视觉要求：
- 系统按钮建议弱存在感，默认不抢主线操作视觉中心。
- 快捷键提示更像教学提示条，而不是一整块厚重面板。

### 6.3 中央决策面板通用框架
以下面板建议共用一套基础框架：
- `攻击技能强化`
- `羁绊抽卡`
- `宝物候选`
- `烙印选择`
- `成长武器词缀`

需要共用资源：
- `中央弹窗底板`
- `标题栏`
- `副标题栏`
- `关闭 / 返回按钮`
- `说明区底条`
- `卡牌槽位底座`
- `按钮组底板`

视觉要求：
- 中央弹窗是首版最重要的“构筑时刻”，尊贵感必须明显高于常驻 HUD。
- 但不要用过深阴影或过大特效把背景战场完全盖死。
- 建议让中央弹窗有统一的 `中心发光 / 轻度暗角 / 品质描边反馈`。
- 中央决策面板建议更贴近 `MOBA / 端游强化弹窗` 的习惯：
  `顶部标题栏`
  `左侧当前构筑摘要`
  `中间稳定三列候选`
  `底部固定操作栏`
- 不建议做成：
  `移动端整屏卡牌页`
  `漂浮散落式三张大卡`
  `只有卡牌没有当前构筑参考`。
- 玩家做选择时应始终能同时看到：
  `当前 build 摘要`
  `本轮 3 个候选`
  `刷新 / 稍后 / 返回类操作`
  这三类信息。

### 6.4 攻击技能强化卡
需要资源：
- `G卡普通边框`
- `G卡稀有边框`
- `G卡史诗边框`
- `NEW 角标`
- `所属技能标签底`
- `分类标签底`
- `可成型 / 关键强化 / 偏离路线` 标签底
- `选中态描边`

视觉要求：
- `新技能卡` 和 `强化卡` 要能一眼区分，建议靠 `标签结构` 和 `角标` 实现，不建议完全换一套底板。
- 所属技能标签必须稳定、醒目，不能埋进正文里。
- 文本区要留足两行核心描述空间。

### 6.5 羁绊抽卡面板与羁绊卡
需要资源：
- `F面板底板`
- `刷新按钮`
- `免费刷新次数底标`
- `稍后选择按钮`
- `7格羁绊位底座`
- `吞噬替换提示框`
- `普通 / 稀有 / 史诗羁绊卡框`
- `可成套 / 已成型 / 高风险替换 / 将触发替换` 状态标

视觉要求：
- `当前进度 X/Y` 必须有固定展示区，不要和羁绊名称抢位。
- 满 `7 张` 后的替换决策是高压选择，推荐替换目标与风险提示必须明显。
- `已激活` 与 `未激活` 的羁绊状态在卡面和羁绊位里都要有清楚区分。

### 6.6 宝物候选面板与宝物卡
需要资源：
- `宝物面板底板`
- `3个宝物位底座`
- `普通 / 稀有 / 史诗宝物卡框`
- `可联动 / 当前适配 / 将触发替换` 标签底
- `推荐替换目标提示框`
- `已弃置宝物列表底板`

视觉要求：
- 宝物是高价值功能组件，卡面应更“厚重”一点，但仍要让正文足够清楚。
- 当选择会触发替换时，要让玩家在不点开详情的情况下就能意识到风险。
- 宝物按钮、宝物候选面板、背包宝物页签的视觉语言要统一。

### 6.7 烙印选择面板与烙印卡
需要资源：
- `烙印面板底板`
- `普通 / 稀有 / 史诗烙印卡框`
- `节点标题条`
- `来源等级节点角标`
- `已获得烙印列表条目底板`

视觉要求：
- 烙印是少次数的高价值节点，面板需要比宝物更“仪式化”一点。
- 但它不提供刷新、不允许稍后选择，所以交互按钮可以更简洁，重点放在 3 张候选卡本身。
- 烙印卡需要更强的“刻印 / 铭文 / 法印”气质，区别于宝物的实物感。

### 6.8 底部角色操作区
需要资源：
- `底部主条底板`
- `英雄头像框`
- `等级框`
- `经验条`
- `成长武器格`
- `4个攻击技能位底座`
- `技能(G)主按钮`
- `羁绊(F)主按钮`
- `宝物 / 集火 / 已吞噬` 次级按钮
- `技能点徽标`
- `木材消耗角标`
- `继续选择 / 待选词缀 / 将触发替换` 按钮状态标

视觉要求：
- `技能(G)` 与 `羁绊(F)` 必须明显大于功能按钮。
- `技能位` 和 `成长武器格` 需要在同一条操作区里保持统一框体语言。
- `已吞噬` 按钮虽然是次级入口，但当有新烙印时需要支持明显高亮。
- 底部操作区建议明确对齐 `MOBA 端游用户习惯`：
  `左侧英雄核心`
  `中间技能热区`
  `右侧功能簇`
  `底边细经验轨`
- 不建议把所有按钮做成同一尺寸的横向平铺；应当明确区分：
  `战斗热区`
  `构筑主按钮`
  `次级功能按钮`
- `技能位` 必须保持整个底栏的视觉中心，不能因为右侧功能按钮变多而整体右偏。
- `技能(G)`、`羁绊(F)` 虽然属于构筑操作，但布局上应更像 `主操作键`，而不是普通菜单入口。

### 6.9 背包与吞噬界面
需要资源：
- `背包主面板`
- `页签底板`
- `成长武器页`
- `宝物页`
- `奖励记录页`
- `吞噬界面主面板`
- `烙印 / 生效中羁绊 / 已吞噬记录` 页签
- `记录条目底板`

视觉要求：
- 背包与吞噬界面属于查看型界面，信息密度高于战斗操作区，但视觉优先级低于中央构筑面板。
- 页签层级必须清楚，避免像一整页大图。
- `已吞噬记录` 建议偏档案式视觉，但不要过度做旧导致字难读。

### 6.10 右侧任务追踪区
需要资源：
- `追踪面板底板`
- `主线任务条目`
- `挑战任务条目`
- `新手引导条目`
- `自动任务开关`
- `剩余时间 / 进度条` 小组件

视觉要求：
- 任务追踪区是信息辅助区，装饰要轻。
- 主线、挑战、新手引导三类条目要能快速区分，但不能做成三套完全不同风格。

### 6.11 挑战入口按钮
需要资源：
- `金币挑战按钮`
- `木材挑战按钮`
- `经验挑战按钮`
- `宝物挑战按钮`
- `挑战次数徽记`
- `进行中倒计时覆盖层`
- `未解锁 / 次数不足 / 进行中 / 可进入` 状态底条

视觉要求：
- 4个挑战按钮要统一底板结构，差异主要来自图标和状态色。
- 即使不可用也不灰掉到看不见，只通过状态字、角标和描边变化表达。
- `宝物挑战` 需要比其他 3 个挑战有更高的尊贵感。

### 6.12 结算界面
需要资源：
- `胜利结算面板`
- `失败结算面板`
- `挑战模式结算面板`
- `数据条目底板`
- `奖励条目底板`
- `进入挑战模式 / 返回` 按钮

视觉要求：
- 结算界面要有明确的大结果氛围，但不能复杂到影响数据阅读。
- `胜利 / 失败 / 挑战额外完成` 需要有清楚的主色差异。

## 7. 动效与特效需求

### 7.1 必需动效
- `按钮高亮呼吸`
- `资源入账数字跳动`
- `经验条增长`
- `卡牌翻入 / 淡入`
- `高品质候选描边闪光`
- `挑战按钮进行中倒计时脉冲`
- `已获得新烙印后的按钮提醒`

### 7.2 高优先级战斗提示
- `Boss 高危技能前摇标签`
- `高价值奖励获得提示`
- `待处理中央面板入口高亮`
- `待选词缀 / 继续选择` 提醒

### 7.3 动效原则
- 战斗中动效以 `短、准、清楚` 为主，避免长时间漂浮遮挡。
- 高品质反馈建议靠 `描边光、闪烁、粒子短爆发` 实现，不建议大面积全屏光污染。

## 8. 交付格式与命名规范

### 8.1 建议交付内容
- `源文件`
  推荐 `PSD / Figma / 其他分层源文件`
- `程序可接资源`
  透明背景导出图
- `切图说明`
  标明是否九宫、是否拉伸、哪些区域不能裁
- `状态对照表`
  标明默认 / 高亮 / 禁用 / 选中对应资源名

### 8.2 命名规范
建议统一前缀：
- `ui_hud_*`
- `ui_panel_*`
- `ui_card_*`
- `ui_btn_*`
- `ui_icon_*`
- `ui_tag_*`
- `ui_settle_*`

示例：
- `ui_hud_top_bar_bg`
- `ui_btn_g_upgrade_normal`
- `ui_btn_g_upgrade_disabled`
- `ui_card_treasure_epic_frame`
- `ui_tag_can_complete`
- `ui_panel_swallow_tab_mark`

### 8.3 切图规范
- 长条背景优先支持 `九宫`。
- 卡框若有复杂边角，需单独确认拉伸安全区。
- 图标尽量与边框分离，避免程序后续无法换底。
- 同类按钮尽量复用底图，只替换图标、标题和角标。

### 8.4 补标要求
- 当前阶段默认走 `AI 快速补标模式`，不走传统大而全交付单。
- 每个 `P0` 页面只需要准备：
  `页面说明`
  `结构草图`
  `资源清单`
- 目标不是做得正式，而是让：
  `AI 能快速出第一版`
  `程序能直接搭骨架`
  `后续少返工`

#### 8.4.1 结构标注图必须包含
- `页面名`
- `主要模块`
- `模块大概位置`
- `模块大概大小关系`
- `哪些区域是动态文本`
- `哪些底板需要拉伸`

#### 8.4.2 程序对齐方式标注规范
- 若模块贴边，只要明确写：
  `顶部 / 底部 / 左侧 / 右侧`
- 若模块居中，只要明确写：
  `center`
  或 `percent(x, y)`
- 若模块在父容器内部排版，只要明确写：
  `局部排版`
  或 `局部坐标`
- 当前阶段不强制每个控件都做像素级坐标标注。

#### 8.4.3 切图 / 九宫标注图必须包含
- `资源逻辑名`
- `资源类型`
- `属于哪个模块`
- `是否可拉伸`
- `是否有状态变化`
- 若需要九宫，再补：
  `left/right/top/bottom`

#### 8.4.4 文本与数字补标要求
- 标题、正文、数字区只要补：
  `示例文案`
  `最大行数`
  `对齐方式`
- 若文案明显会变长，再额外补一个 `最长示例` 即可。
- 当前阶段不强制每个文本区都写完整字号和像素尺寸。

#### 8.4.5 推荐补标交付格式
- `1 份页面说明`
  一句话说清目标和风格词。
- `1 份结构草图`
  可以是框图、ASCII 图或 Markdown 结构清单。
- `1 份资源清单`
  用最小字段列清底板、图标、标签、文本。
- 若走高 AI 自动化流程，也可以直接合并为：
  `1 份单文件 YAML 规格`

### 8.5 标注与程序 API 对应关系

| 美术标注项 | 程序落地方式 |
| --- | --- |
| `顶部 / 底部 / 左侧 / 右侧吸附` | `set_relative_parent_pos()` |
| `percent(x, y)` | `GameAPI.set_ui_comp_pos_percent()` |
| `容器宽高` | `set_ui_size()` |
| `锚点` | `set_anchor()` |
| `父容器内局部位置` | `set_pos()` |
| `九宫边距` | `set_ui_9_enable(true)` + `set_ui_9()` |
| `文本框尺寸与对齐` | `set_ui_size()` + `set_text_alignment()` |
| `调试容器框` | `create_child('图片') + set_image(999)` |

## 11. 程序与美术协作需求

### 11.1 目标
- 首版 UI 制作流程明确要求：`尽量不靠手调 UI`。
- 程序与美术的协作目标不是“美术出整张成品图，程序再一个一个拖位置”，而是：
  `美术提供可拼装资源`
  `程序按统一布局规则自动生成`
  `后续换图时不改布局逻辑`
- 所有新 UI 需求都应优先沉淀为：
  `资源规范`
  `布局规格`
  `状态规格`
  而不是“某个分辨率下看着差不多就行”。

### 11.2 职责分工
- `UI设计 / UI美术`
  负责定义视觉方案、资源拆分、状态图、九宫信息、尺寸规格和导出命名。
- `程序`
  负责基于 [UI适配规则](c:\Y3TD\Y3GPT\ProjectName002\maps\EntryMap\script\UI适配规则.md) 中的 API 建立统一 UI builder，不直接把美术图当成整屏静态贴图摆放。
- `策划`
  负责给出界面结构、信息优先级、状态数量与交互规则，不直接指定具体像素坐标。

### 11.3 首版禁止事项
- 不允许把整张 `1920x1080` 成品图直接作为可交互 UI 的主要制作方式。
- 不允许把标题、正文、数值、按钮文案长期烘焙进背景图中。
- 不允许程序通过反复手调 `set_pos + 屏幕宽度计算` 完成全屏适配。
- 不允许同一套界面在不同资源版本下频繁改 Lua 坐标来“迁就美术图”。
- 不允许在未拆清 `底板 / 图标 / 状态 / 九宫信息` 前就进入程序接入阶段。

### 11.4 资源交付要求
- 美术交付必须优先提供 `可拼装资源`，而不是单张成品大图。
- 每个界面资源包都必须附带可供程序直接落地的 `补标图`，程序不负责从成品图反推尺寸、边距和九宫。
- 以下资源默认要拆分：
  `底板`
  `内框`
  `标题条`
  `图标`
  `按钮`
  `按钮状态`
  `标签底`
  `品质边框`
  `选中态 / 高亮态`
- 所有可拉伸底板都必须标注：
  `是否九宫`
  `九宫边距`
  `哪些区域允许拉伸`
  `哪些装饰区域不能压缩`
- 所有图标都应单独导出透明底版本，不与按钮底板焊死。
- 所有卡牌类资源都应把：
  `卡底`
  `品质框`
  `标签`
  `状态角标`
  分开交付，避免后续同一逻辑卡牌只能绑定一张死图。

### 11.5 程序接入 API 使用规范
以下规范直接基于 [UI适配规则](c:\Y3TD\Y3GPT\ProjectName002\maps\EntryMap\script\UI适配规则.md) 中已有 API 制定。

#### 11.5.1 控件创建
- 所有 UI 控件统一通过 `parent:create_child(comp_type)` 创建。
- 当需要纯容器节点时，首版优先使用：
  `create_child('图片') + set_image(999)`
  而不是大量使用不可见空节点。
- 这样做的目的，是方便程序和美术共同调试容器尺寸与占位范围。

#### 11.5.2 最外层布局
- 全屏根节点、顶部栏、底部栏、右侧追踪区、中央弹窗根容器，统一优先使用：
  `ui:set_relative_parent_pos('顶部', offset)`
  `ui:set_relative_parent_pos('底部', offset)`
  `ui:set_relative_parent_pos('左侧', offset)`
  `ui:set_relative_parent_pos('右侧', offset)`
- 这类 API 只推荐用于最外层 `1 到 2 层容器`，不建议深入到每一个子控件都依赖边缘吸附。
- 原因是最外层适配目标明确，内部细排版再继续使用边缘吸附会让结构变得难维护。

#### 11.5.3 居中与比例布局
- 所有需要随分辨率稳定居中的控件，统一使用：
  `GameAPI.set_ui_comp_pos_percent(player.handle, ui.handle, x, y)`
- 适用范围：
  `中央弹窗`
  `弹窗内三列候选卡`
  `顶部居中信息`
  `需要稳定居中的标题与主按钮`
- 禁止使用：
  `get_screen_width()` 配合 `set_pos()` 自行计算居中。
- 这是首版“避免手调 UI”的核心要求之一。

#### 11.5.4 固定尺寸与局部排版
- 所有控件的实际尺寸统一通过 `ui:set_ui_size(width, height)` 设置。
- 在一个已经确定尺寸的父容器内部，局部排版允许使用：
  `ui:set_anchor(x, y)`
  `ui:set_pos(x, y)`
- 但 `set_pos()` 只允许用于：
  `固定尺寸父容器内部`
  `局部图标、角标、数字、标签微调`
- 不允许把 `set_pos()` 作为全屏主布局手段。

#### 11.5.5 九宫与可拉伸底板
- 所有面板底板、长条背景、按钮底板、页签底板，只要存在横向或纵向拉伸需求，必须启用：
  `ui:set_ui_9_enable(true)`
  `ui:set_ui_9(left, right, top, bottom)`
- 美术必须同步提供九宫边距，程序不得凭感觉猜测可拉伸区域。
- 未提供九宫信息的资源，不应进入正式程序接入。

#### 11.5.6 文本规范
- 文本控件统一单独创建，不允许长期把正文烘进背景图。
- 文本宽高必须显式设置 `set_ui_size()`，避免出现“默认宽度不够导致显示不全”。
- 文本对齐统一使用：
  `ui:set_text_alignment(h, v)`
- 标题、数值、正文、注释四类文本建议在程序层使用统一字号和样式枚举，不要每个界面各自随意写死。

### 11.6 推荐工程结构
- 首版 UI 工程结构建议至少拆为：
  `ui_assets.lua`
  `ui_builder.lua`
  `ui_spec/*.lua`
  `ui_screen/*.lua`
- `ui_assets.lua`
  只负责资源 ID、九宫参数、字号、颜色等静态配置。
- `ui_builder.lua`
  只负责封装 `create_child / percent / fill / local_box / enable_9` 这类通用搭建逻辑。
- `ui_spec/*.lua`
  只负责声明某个界面有哪些区域、哪些控件、哪些状态，不直接写业务逻辑。
- `ui_screen/*.lua`
  负责把运行时数据绑定到界面上。

### 11.7 推荐制作流程
首版统一按以下流程协作：

1. 策划先写 `页面一句话目标 + 模块清单 + 风格词`，推荐直接写成 `单文件 YAML`。
2. AI 或程序先根据这份轻规格生成 `结构草图` 和 `占位骨架`。
3. AI / 美术再补第一版可拼装资源，不等整套正式图做完才开工。
4. 程序只替换资源、九宫和样式，不重新手调主布局。
5. 若发现问题，优先回改：
   `页面说明`
   `结构清单`
   `资源清单`
   而不是直接堆坐标修补。

### 11.8 程序实现需求
- 程序层优先支持“吃简单规格”，不要要求先有完整版施工图。
- 最小通用 builder 只要优先支持：
  `top / bottom / left / right / center`
  `row / column / local_box`
  `enable_9`
  `text_style`
  `placeholder`
- 程序层必须能支持“替换资源不改结构”，否则不满足首版协作要求。
- 任一重要界面都应能先用占位资源跑起来，再逐步替换为正式美术资源。
- 当资源信息还不完整时，程序允许先用：
  `999 透明图`
  `纯色底板`
  `文本占位`
  推进开发，不阻塞主流程。

### 11.9 界面变更需求约束
- 若只是更换视觉资源，不应要求程序重写布局逻辑。
- 若只是扩展按钮状态，不应要求重做整张按钮底图。
- 若只是同模板下新增一个面板，应优先复用既有框架与卡牌槽位模板。
- 若某个界面改动已经影响到容器层级、九宫策略或适配方式，则必须回到本需求文档重新确认，而不是由程序临时手调修补。
