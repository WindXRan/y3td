# 2026-04-09 特殊效果直挂英雄调试器设计

## 目标

为当前战斗内的“特殊效果”提供一套界面化调试工具，让开发时可以跳过羁绊抽卡、前置节点和链路拼装，直接把指定特殊效果临时挂到英雄身上，立即验证是否触发、生效和结算正确。

这套工具是调试设施，不改正式玩法获取路径，不改变正式羁绊设计，只为开发验证服务。

## 用户确认后的设计结论

- 以界面操作为主，不依赖指令输入
- 不再按羁绊节点逐个调试
- 只需要能把“特殊效果列表”直接挂到英雄身上
- 目标是降低调试成本，提升单效果验收效率

## 本轮范围

- 在现有 GM 调试面板中新增“特效调试”入口
- 新增一个“特殊效果直挂英雄”调试子面板
- 面板内可浏览全部特殊效果
- 支持把单个特殊效果直接挂到英雄
- 支持卸下单个效果与清空全部调试效果
- 支持查看当前调试挂载状态
- 支持查看效果当前冷却、计数器、最近触发结果
- 支持从界面触发一次调试执行
- 支持打印最近调试日志

## 不在本轮范围

- 不重做正式羁绊面板
- 不把调试挂载写回存档或正式构筑状态
- 不新增完整的自动化回归测试 UI
- 不解决所有效果“自动造场景”的问题
- 不替代现有 `.ebond`、快捷键和开发命令，只是不把它们作为主入口

## 设计方案选择

本轮采用“GM 面板扩展页 + 独立调试运行层”的方案。

原因：

- 现有项目已经有 GM 面板、调试按钮和状态刷新循环，接入成本最低
- 用户明确希望以界面为主，不希望通过命令行逐项调试
- 当前真正痛点不是“入口缺失”，而是“特殊效果必须先拼羁绊才可验证”
- 独立调试运行层可以与正式羁绊状态并存，不污染正式系统

## 交互设计

### 入口

在现有 GM 面板中新增一个按钮：

- `特效调试`

点击后打开一个二级调试面板。面板默认只在 debug 模式下出现。

### 面板布局

调试面板分为四块：

1. 左侧效果列表
- 列出全部可调试的特殊效果
- 每项显示：
  - 效果名
  - `effect_id`
  - 触发类型
  - 当前是否已挂载

2. 右侧详情区
- 显示当前选中效果的核心信息：
  - 效果名
  - 来源类型与来源 id
  - 触发类型
  - 冷却配置
  - 计数需求
  - 关键数值字段
  - 当前运行时状态

3. 底部操作区
- `挂载到英雄`
- `卸下当前效果`
- `清空全部`
- `触发一次`
- `开始观测`
- `打印日志`

4. 日志区
- 显示最近若干条调试日志
- 每条日志包含：
  - 时间
  - 效果名
  - 动作类型
  - 结果摘要

### 第一版交互原则

- 列表优先，一眼能看到当前哪些效果已经挂载
- 不要求用户输入参数
- 不做复杂分页和筛选，先保证可挂载、可触发、可观察
- 若后续效果变多，再补筛选和分类

## 数据与运行时设计

### 核心思路

新增一层“调试挂载效果状态”，独立于正式羁绊和正式来源激活状态。

正式效果生效判断当前是：

- 来源已激活时生效

本轮改为：

- 来源已激活，或
- 被调试器手动挂载

这样特殊效果可以在不满足正式来源条件时，仍然由调试面板强制生效。

### 新增运行时状态

在 `STATE` 上新增一组调试数据：

- `effect_debug_runtime`

建议结构：

```lua
{
  mounted_effect_ids = {},
  selected_effect_id = nil,
  logs = {},
  last_trigger_result = {},
  observing_effect_id = nil,
  observe_remaining = 0,
}
```

用途：

- `mounted_effect_ids`
  - 记录当前哪些特殊效果被调试器直接挂载到英雄
- `selected_effect_id`
  - 面板当前选中的效果
- `logs`
  - 最近调试日志
- `last_trigger_result`
  - 最近一次触发执行结果，按效果 id 记录
- `observing_effect_id` / `observe_remaining`
  - 用于短时持续观测某个效果

### 模块划分

新增模块：

- `maps/EntryMap/script/runtime/effect_debug.lua`

职责：

- 管理调试挂载状态
- 提供效果列表与详情数据
- 提供挂载、卸下、清空、手动触发、日志写入
- 提供给 UI 的只读模型

现有模块的改动职责：

- `runtime/auto_active_effects.lua`
  - 支持查询调试挂载状态
  - 暴露冷却、计数器、触发结果的读取接口
  - 暴露“强制触发单个效果”的调试接口

- `runtime/debug_tools.lua`
  - 在 GM 面板里新增“特效调试”入口
  - 创建并刷新调试子面板

- `runtime/debug_actions.lua`
  - 承接 UI 按钮触发的动作调用

- `runtime/boot.lua`
  - 创建并注入 `effect_debug_system`
  - 把调试状态接入现有刷新循环

## auto_active_effects 的改造要求

### 现状

`runtime/auto_active_effects.lua` 当前通过 `is_source_active(def)` 判断某个效果是否可生效，主要取决于：

- 羁绊来源
- 宝物来源
- 烙印来源

### 改造后

增加一层判断：

```lua
if debug_mount_active(effect_id) then
  return true
end
```

优先级要求：

- 调试挂载只影响“是否可触发”
- 不改效果本身的冷却、触发、伤害和结算逻辑
- 不改正式来源数据
- 不把调试挂载伪装成正式拥有

### 需要补充的调试接口

建议在 `auto_active_effects.lua` 暴露以下接口：

- `get_effect_defs()`
- `is_effect_debug_mounted(effect_id)`
- `get_effect_runtime_snapshot(effect_id)`
- `force_trigger_effect(effect_id, context)`
- `clear_effect_runtime(effect_id)`

其中 `get_effect_runtime_snapshot(effect_id)` 返回：

- 当前是否可生效
- 当前冷却剩余
- 当前计数器
- 最近一次触发是否成功
- 最近一次失败原因

## UI 行为设计

### 列表项显示字段

每个效果列表项显示以下文本：

- 效果名
- `trigger_type`
- `mounted` / `unmounted`

例如：

- `雷链脉冲 | periodic | mounted`

### 详情区显示字段

建议详情区显示：

- 名称
- `effect_id`
- 来源：`source_type/source_id`
- 触发：`trigger_type`
- 冷却：`cooldown`
- 计数需求：`counter_required`
- 关键字段：
  - `damage_ratio`
  - `radius`
  - `range`
  - `chain_bounces`
  - `chain_ratio`
  - 其他当前效果定义里存在的关键字段
- 当前状态：
  - 已挂载
  - 冷却剩余
  - 当前计数
  - 最近一次触发结果

### 操作按钮行为

`挂载到英雄`

- 将选中效果加入 `mounted_effect_ids`
- 写入日志

`卸下当前效果`

- 从 `mounted_effect_ids` 移除
- 可选清理该效果的临时运行时数据

`清空全部`

- 清空所有调试挂载效果
- 清理全部调试日志之外的运行时挂载状态

`触发一次`

- 调用 `auto_active_effects.force_trigger_effect`
- 不绕过效果本身结算逻辑
- 若触发失败，记录失败原因

`开始观测`

- 针对当前效果开启短时观察，比如 10 秒
- 持续把冷却、计数器和触发结果写入日志

`打印日志`

- 在消息区输出最近若干条日志
- 便于截图和快速确认

## 日志设计

日志只服务调试，不追求复杂结构。

建议统一格式：

```text
[effect_debug] 雷链脉冲 | mount | success
[effect_debug] 雷链脉冲 | trigger | success | cooldown=6.0
[effect_debug] 雷链脉冲 | observe | cd=4.8 counter=0
[effect_debug] 雷链脉冲 | trigger | failed | reason=no_target
```

UI 日志区展示简化后的可读文本，消息输出保留完整调试信息。

## 与正式玩法的边界

必须严格保持以下边界：

- 调试挂载不写入正式羁绊状态
- 调试挂载不占用正式羁绊槽位
- 调试挂载不改变正式来源拥有关系
- 退出战斗或重置会话时清空调试挂载
- UI 上必须明确这是“调试效果”，避免误解为正式构筑内容

## 重置与清理

以下时机需要清理调试挂载状态：

- `reset_battle_state`
- `reset_session_state`
- 回到局外
- 英雄重建

至少保证：

- 不把上一局的调试挂载带进下一局
- 不把旧日志和旧冷却误带到新会话

## 实施顺序

1. 新增 `effect_debug.lua`，搭好调试状态与数据接口
2. 扩展 `auto_active_effects.lua`，接入 debug 挂载判定与状态查询
3. 扩展 `debug_actions.lua`，提供挂载、卸下、强制触发等动作
4. 扩展 `debug_tools.lua`，把 UI 面板挂到现有 GM 面板
5. 在 `boot.lua` 接线并接入重置逻辑
6. 手动验证：
   - 挂载单个效果
   - 触发一次
   - 查看冷却变化
   - 卸下效果
   - 重开会话确认状态清空

## 验收标准

满足以下条件即可认为本轮完成：

- 能从界面直接看到全部特殊效果列表
- 能从界面把任意一个效果挂到英雄身上
- 挂载后该效果可在不满足正式来源条件下被触发
- 能从界面查看该效果当前冷却和计数器
- 能从界面手动触发一次并看到结果
- 能从界面清空全部调试挂载
- 战斗重置后调试挂载不会残留

## 风险与控制

主要风险：

- 调试挂载与正式来源判定混淆，导致正式玩法也被绕过
- 强制触发接口侵入过深，影响正常战斗逻辑
- 新增调试 UI 后刷新过重，导致面板状态不同步

控制方式：

- 所有调试逻辑集中在 `effect_debug.lua`
- `auto_active_effects.lua` 只补调试分支，不重写主体逻辑
- 调试接口统一以 `debug_` 或 `effect_debug` 命名，避免误用
- 第一版不做复杂筛选和自动造场景，先保证挂载与可观测性

## 预期结果

完成后，开发者在游戏内无需再通过拼羁绊链路进入某个特殊效果，只需要：

1. 打开 GM 面板
2. 进入特效调试页
3. 选择一个特殊效果
4. 直接挂到英雄
5. 观察和触发验证

这样能把“验证单个特殊效果是否生效”的时间，从一整套羁绊前置流程，缩短为一次界面点击流程。
