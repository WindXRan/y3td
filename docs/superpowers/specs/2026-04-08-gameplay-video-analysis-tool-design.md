# 游戏试玩视频分析工具设计

## 目标

开发一个可在本地运行的工具，输入最长约 30 分钟的游戏试玩视频，自动完成以下工作：

1. 捕获并筛选关键屏幕内容。
2. 识别视频中的主要玩法、界面、提示、战斗、奖励、成长与反馈信息。
3. 生成一份“时间轴纪要 + 设计分析”混合文档。
4. 同时输出结构化中间结果，便于后续复用、调试和升级分析能力。

工具需要同时支持：

1. 命令行使用。
2. 本地网页上传和结果预览。
3. 两种分析模式：
   - 离线模式：本地抽帧、OCR、规则总结。
   - 默认模式：本地处理后调用 OpenAI 生成更接近人工撰写的总结文档。

## 非目标

第一阶段不做以下内容：

1. 不做云端部署。
2. 不做多人协作、账号系统或数据库服务。
3. 不做逐帧精细物体检测训练。
4. 不保证对所有游戏类型都达到专家级识别准确率。
5. 不做视频剪辑、自动配音或字幕导出。

## 设计原则

1. 核心分析流程与交互层解耦。
2. 所有结果尽量落地为文件，方便追查和二次处理。
3. 先保证 30 分钟视频可稳定跑通，再优化识别精度和体验。
4. 默认使用模型总结，但必须保留离线兜底路径。
5. 中间数据结构尽量明确，避免“只能看最终文档，无法解释过程”。
6. 工具代码应放在项目中的独立文件夹内，避免与现有游戏运行脚本混杂。

## 推荐方案

采用双层架构：

1. 底层为独立视频分析引擎。
2. 上层提供 CLI 与本地网页两种入口。

推荐原因：

1. CLI 可以最先落地，最快验证核心能力。
2. 网页只作为壳层，不复制分析逻辑。
3. 后续若要接入批处理、队列、远程服务，可以直接复用底层引擎。

## 整体架构

### 1. 分层

系统拆为两层：

1. `analysis engine`
   - 负责视频读取、抽帧、OCR、时间轴合并、关键帧选择、结构化摘要、模型总结、报告输出。
2. `delivery layer`
   - CLI：输入文件路径并生成结果目录。
   - Web：上传视频、展示进度、预览结果、下载文档。

### 2. 数据流

输入视频后按以下流水线处理：

1. 视频读取与基础探测。
2. 抽帧与镜头变化检测。
3. 帧级内容提取。
4. 事件段合并。
5. 关键帧筛选。
6. 规则摘要生成。
7. 可选模型总结。
8. Markdown/JSON/截图产物输出。

## 核心流程设计

### 1. 视频预处理

职责：

1. 读取本地视频文件。
2. 获取时长、分辨率、帧率等元数据。
3. 按固定时间间隔抽帧。
4. 使用画面差异补充关键转场帧。

建议策略：

1. 基础抽帧间隔默认 2 秒。
2. 若检测到明显镜头变化、UI 大幅变化、文本区域变化，则额外保留转场帧。
3. 为控制成本，30 分钟视频第一阶段目标保留 600 到 1500 张候选帧，再进一步压缩。

输出：

1. `video_meta.json`
2. `frames/index.json`
3. 原始候选帧图片

### 2. 帧级内容提取

职责：

1. OCR 识别文本。
2. 生成帧摘要。
3. 标记帧类型和候选事件标签。

第一阶段建议识别的信息：

1. 界面文本
2. 数字与资源变化
3. 战斗/菜单/奖励/结算/教程/抽卡/构筑/养成等粗粒度场景标签
4. 是否存在明显弹窗、按钮组、卡牌选择、商店、结算页

实现建议：

1. OCR 采用本地引擎，优先选支持中文的方案。
2. 标签识别第一阶段先用规则与轻量图像特征，不依赖训练专用模型。
3. 每帧生成统一结构，避免不同模式下字段漂移。

输出示例字段：

1. 时间戳
2. 帧路径
3. OCR 文本块
4. 文本摘要
5. 候选场景标签
6. 视觉复杂度分数
7. 是否推荐作为关键帧

### 3. 时间轴事件合并

职责：

将相邻、相似、连续的帧合并为更高层级的“事件段”。

目标：

把“成百上千张帧”整理成“几十个可读事件段”，例如：

1. `00:40 - 01:25` 教程引导
2. `03:12 - 04:05` 进行羁绊三选一
3. `08:20 - 09:10` Boss 战

合并依据：

1. 相邻时间接近
2. 主要 OCR 文本相似
3. 标签相似
4. 界面结构变化较小

事件段需要产出的字段：

1. 开始时间
2. 结束时间
3. 事件标题
4. 事件类型
5. OCR 汇总
6. 关键文本
7. 代表截图集合
8. 简短描述

### 4. 关键帧筛选

职责：

从每个事件段中挑出少量最有代表性的截图。

筛选目标：

1. 能体现界面结构
2. 能体现反馈表现
3. 能体现关键玩法决策
4. 能在文档中作为证据引用

建议策略：

1. 每段默认保留 1 到 3 张关键帧。
2. 优先保留文字清晰、结构明显、变化节点、奖励/选择/结算等帧。
3. 若事件段过长，可按阶段分层保留。

### 5. 规则摘要

职责：

在不依赖模型的前提下，先产出一份稳定、可解释的中间总结。

内容包括：

1. 时间轴事件列表
2. 高频文本与关键词
3. 被识别到的核心系统
4. 可能的资源/成长/奖励/战斗循环

价值：

1. 离线模式直接使用。
2. 默认模式作为模型输入前的压缩层。
3. 降低模型输入体积与成本。

### 6. 模型总结

默认模式下，使用 OpenAI 对规则摘要与关键帧摘要进行提升式总结。

输入给模型的内容不应是整段视频或所有原始帧，而应是：

1. 视频基本信息
2. 时间轴事件摘要
3. 每个事件段的代表截图说明
4. OCR 提取到的关键文字
5. 识别出的核心系统标签

模型输出目标：

1. 人类可读的 Markdown 文档
2. 同时具备“纪要”和“设计分析”两层结构

模型总结要覆盖：

1. 试玩流程概览
2. 时间轴内容纪要
3. 核心系统分析
4. 核心循环推断
5. 反馈表现总结
6. 亮点
7. 问题
8. 改进建议

### 7. 文档生成

最终生成的主文档为 Markdown。

建议结构如下：

1. 视频概览
   - 文件名
   - 时长
   - 分析模式
   - 抽帧数
   - 事件段数量
2. 时间轴纪要
   - 按时间段描述发生了什么
3. 核心系统识别
   - 战斗
   - 成长
   - 资源
   - 抽卡/构筑
   - 关卡/目标
   - UI 导航
   - 反馈表现
4. 设计分析
   - 核心循环
   - 节奏
   - 决策点
   - 玩家目标引导
5. 优点与风险
6. 改进建议
7. 附录
   - 关键帧索引
   - 引用截图路径

## 输出产物设计

每次分析应输出到独立目录，建议结构：

```text
outputs/<task_id>/
  report.md
  timeline.json
  analysis.json
  video_meta.json
  artifacts/
    frames/
    keyframes/
```

说明：

1. `report.md`：最终阅读文档。
2. `timeline.json`：事件段时间轴。
3. `analysis.json`：中间分析结果全集。
4. `video_meta.json`：视频元信息。
5. `artifacts/frames/`：候选帧。
6. `artifacts/keyframes/`：最终引用帧。

## 模块划分

建议按以下模块组织：

工具整体建议单独放入一个独立目录，例如：

```text
tools/gameplay_video_analyzer/
```

该目录下再按职责拆分分析引擎、CLI、网页与模板资源。这样做的原因是：

1. 与当前游戏脚本和 Y3 运行时代码隔离。
2. 方便单独安装依赖与维护说明。
3. 方便以后把工具迁移到其他项目或单独开仓。

### 1. `video_ingest`

职责：

1. 视频读取
2. 抽帧
3. 镜头变化检测
4. 元数据输出

### 2. `vision_extract`

职责：

1. OCR
2. 文本清洗
3. 画面粗分类
4. 帧级摘要生成

### 3. `timeline_builder`

职责：

1. 帧分组
2. 事件段合并
3. 时间轴标题生成

### 4. `keyframe_selector`

职责：

1. 从候选帧中选代表帧
2. 输出文档引用图

### 5. `summarizer`

职责：

1. 规则摘要
2. OpenAI 模式总结
3. 离线模式总结

### 6. `report_writer`

职责：

1. Markdown 输出
2. JSON 输出
3. 资源目录整理

### 7. `cli`

职责：

1. 解析命令行参数
2. 启动分析任务
3. 输出结果路径与日志

### 8. `web_app`

职责：

1. 视频上传
2. 任务进度展示
3. Markdown 预览
4. 下载文档与关键帧

## CLI 设计

第一阶段 CLI 应支持：

1. 输入视频路径
2. 指定输出目录
3. 指定模式
4. 指定抽帧间隔
5. 指定是否启用模型总结

建议命令形态：

```text
analyze-video --input <video_path> --output <dir> --mode hybrid
```

必需体验：

1. 有清晰日志输出
2. 出错时能定位在哪个阶段失败
3. 完成后打印主文档路径

## Web 设计

第一阶段网页建议保持极简，目标是“可用而不是复杂”。

页面只需包含：

1. 视频上传区
2. 参数选择区
   - 分析模式
   - 抽帧间隔
   - 是否使用模型总结
3. 进度区
   - 当前阶段
   - 已完成事件数/帧数
4. 结果区
   - Markdown 预览
   - 关键帧画廊
   - 下载按钮

不需要在第一阶段加入：

1. 用户系统
2. 历史任务库
3. 多标签页复杂管理

## 技术选型建议

推荐使用 Python 作为核心实现语言。

原因：

1. 视频处理和 OCR 生态成熟。
2. 文件流与批处理简单。
3. 模型调用、Markdown 输出和 Web 快速原型都方便。

技术建议：

1. 核心分析：Python
2. CLI：`typer` 或 `argparse`
3. 本地网页：优先 `Gradio`
4. 模型调用：OpenAI API
5. 报告输出：Markdown + JSON

选择 `Gradio` 的原因：

1. 上传文件与长任务流程简单。
2. 本地运行成本低。
3. 适合作为 CLI 之上的薄壳。

## 错误处理

必须显式处理以下问题：

1. 视频文件不存在
2. 视频格式无法读取
3. 抽帧阶段中断
4. OCR 引擎不可用
5. 模型 API 配置缺失
6. 模型调用超时或失败
7. 输出目录不可写

处理原则：

1. 每一阶段写日志。
2. 默认尽量保留已完成的中间产物。
3. 模型阶段失败时，可回退为离线总结，不应直接整任务失败。

## 性能与规模约束

第一阶段的成功标准不是极致快，而是稳定完成任务。

目标约束：

1. 30 分钟视频可完整跑通。
2. 中间帧数可控，不因为逐帧分析而爆炸。
3. 模型输入经过压缩，避免直接发送大量图片。

第一阶段不做：

1. GPU 优化
2. 分布式处理
3. 实时视频流分析

## 可扩展性

后续可扩展方向：

1. 批量分析多个视频
2. 支持指定关注主题
   - 仅分析 UI
   - 仅分析战斗反馈
   - 仅分析新手引导
3. 支持多模板输出
   - 竞品分析版
   - 关卡体验版
   - 商业化/成长系统版
4. 支持更强的图像理解与区域检测
5. 支持把结果导出为富文本或 PDF

## 验收标准

第一阶段视为成功，需要满足：

1. 用户可通过 CLI 输入视频路径并生成完整输出目录。
2. 工具可通过本地网页上传视频并展示分析结果。
3. 30 分钟试玩视频可稳定生成 `report.md`。
4. 文档包含时间轴纪要与设计分析两个部分。
5. 中间产物可追溯，包括关键帧和时间轴 JSON。
6. 模型模式失败时可回退到离线总结模式。

## 分阶段落地顺序

推荐按以下顺序开发：

1. 先完成 CLI 核心管线。
2. 再完成 Markdown/JSON 输出。
3. 再接入 OpenAI 总结。
4. 最后挂本地网页。

原因：

1. 先证明“能分析”。
2. 再提升“能写得好”。
3. 最后补“更好用的界面”。

## 风险与对策

### 风险 1：OCR 噪声很大

对策：

1. 加文本清洗。
2. 以事件段聚合结果为准，不依赖单帧文本。

### 风险 2：场景标签不准

对策：

1. 第一阶段只做粗粒度标签。
2. 文档中把不确定结论写成推断而非断言。

### 风险 3：模型输入过大

对策：

1. 模型只接收结构化摘要与关键帧说明。
2. 不直接发送全部抽帧。

### 风险 4：网页壳抢占太多开发时间

对策：

1. 强制先做 CLI。
2. 网页只作为调用层，不承载业务核心。

## 决策总结

最终确认如下：

1. 输出形式为混合版文档：时间轴纪要 + 设计分析。
2. 同时支持离线模式和 OpenAI 总结模式，默认使用后者。
3. 同时支持 CLI 和本地网页，先做 CLI，再加网页。
4. 架构采用“独立分析引擎 + 薄交互层”。
