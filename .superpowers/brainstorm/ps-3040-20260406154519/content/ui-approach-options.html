方案 A：全部写在 Lua

延续当前 `entry_runtime_hud.lua` / `entry_runtime_outgame.lua` 风格，直接 `create_child`。

优点

  • 接入快
  • 状态驱动清晰
  • 不依赖编辑器先出稿

缺点

  • 美术很难对位
  • 结构不可视
  • 换图和调布局成本高

方案 B：先做完整静态 UI 资产

主要在 `maps/EntryMap/ui` 做 prefab / panel，Lua 只负责拿节点和填数据。

优点

  • 美术和策划容易协作
  • 结构清晰
  • 后续扩展正式 UI 最顺

缺点

  • 前期编辑器工作量大
  • JSON 资产 diff 难看
  • 第一次接线更慢

方案 C：静态骨架 + 运行时数据/换图

把面板和卡槽结构固定在 UI 资产里，Lua 只负责打开、刷新、绑定图片资源 ID 和点击回调。

优点

  • 兼容当前 Lua 主链路
  • 也给未来美术留足槽位
  • G / F / 宝物可共用一套面板

缺点

  • 需要先定义字段规范
  • 要补节点命名规范
  • 第一版要多写一点文档
我建议

选 C：在 `maps/EntryMap/ui` 放一个通用三选一面板/卡槽 prefab；在 `maps/EntryMap/script` 放 presenter + data adapter；图片统一走 `ui_res.lua` 和“业务对象 icon 字段”。
